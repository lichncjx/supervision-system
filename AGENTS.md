# 公司督办管理系统 - AI 开发规范

本文件是 Codex、Claude Code 等 AI Agent 参与本项目开发时的主入口。开始任何任务前必须先阅读本文件；涉及业务规则时还必须阅读对应的 `docs/*` 文档。

## 项目定位

本系统用于管理公司重点工作、主要工作、待办事项，核心闭环是事项、节点、审批、权限、统计、提醒和发布验收。

## 技术栈

- Next.js App Router
- React
- TypeScript
- Tailwind CSS
- shadcn/ui
- PostgreSQL
- Prisma
- pnpm

## 第三方文档查询

当用户询问库、框架、SDK、API、CLI 工具或云服务的用法、配置、迁移、调试时，必须优先使用 `ctx7` CLI 获取当前文档：

1. 解析库 ID：`npx ctx7@latest library <name> "<用户完整问题>"`
2. 选择最匹配的 `/org/project` 或 `/org/project/version`
3. 获取文档：`npx ctx7@latest docs <libraryId> "<用户完整问题>"`
4. 如结果不足，再使用同一命令追加 `--research`

Context7 请求需要在 Codex 默认沙箱外执行；如果遇到 DNS、ENOTFOUND、fetch failed 等网络错误，应按工具权限流程请求外部执行。不要在查询中包含密钥、密码、Token 等敏感信息。

## 核心业务原则

1. 权限优先于页面展示。
2. 页面隐藏按钮不等于权限控制，所有关键操作必须有服务端权限校验。
3. 状态流转必须通过 workflow 层统一处理，页面不得直接修改事项状态。
4. 审批记录不得被删除。
5. 统计口径必须和列表查询口径一致。
6. 首页卡片数量必须能和点击后的列表结果对应。
7. 待我处理 = 待审批 + 待办理。
8. 待审批只统计当前用户作为当前审批人的待审批事项。
9. 待办理指不需要审批但需要当前用户处理的事项，例如待分解、退回后修改、待完成确认等。
10. 退回后修改的处理人判定使用 `firstSubmitterId`（首次提交审批人），不是 `creatorId`；同部门其他主管/领导不应因部门关联而获得处理权限。
11. 涉及权限、状态、审批、统计的修改必须说明影响范围。

## 开发边界

1. 不要擅自重构数据库结构。
2. 不要擅自引入新的流程引擎。
3. 不要擅自改变角色枚举。
4. 不要擅自改变状态枚举。
5. 不要把测试账号密码硬编码到前端页面。
6. 不要绕过服务端权限校验。
7. 不要删除已有数据。
8. 不要清空数据库。
9. 不要修改 Docker 和部署配置，除非 Issue 明确要求。
10. 不要为了修复一个页面问题而大范围重构全局架构。

## 开工流程

AI Agent 开始任务时必须先检查当前上下文，不要无条件切分支或覆盖用户改动。

1. 阅读 `AGENTS.md` 和任务相关文档。
2. 执行 `git branch --show-current` 确认当前分支。
3. 执行 `git status --short` 检查是否存在未提交改动。
4. 如果存在用户未提交改动，必须保护这些改动；只有在确认不会冲突时继续。
5. 需要新建任务分支时，从最新 `main` 创建对应分支；如本地状态不适合切换，先向用户说明。
6. 修改前先定位相关代码和文档，尤其是权限、审批、状态、统计相关文件。
7. 修改后运行必要检查，并在无法运行时说明原因。

## AI 自动执行协议

每个 Issue 必须指定执行模式：

- `Autopilot`：低风险任务，AI 可以自动完成分析、修改、自测、PR 创建。
- `Supervised-Batch`：已完成审计、用户已确认关键风险点、可以在明确边界内批量执行的中低风险任务。
- `Supervised`：中风险任务，AI 可以自动推进低风险小步骤，但大阶段结束后需要报告，关键节点需要 Codex Review。
- `Locked`：高风险任务，AI 只能分析和出方案，不能直接编码。

### Autopilot 执行规则

适用于文档、配置、小 UI、低风险 bug、lint/type/build 修复。

AI 可以：
1. 自动分析；
2. 自动修改；
3. 自动运行检查；
4. 自动修复低风险问题；
5. 自动创建 PR。

AI 只在触发暂停条件时停止。

### Supervised-Batch 执行规则

适用于已完成审计、用户已确认关键风险点、可以在明确边界内批量执行的中低风险任务。

典型场景：
- 文档目录重构；
- 历史过程文档归档；
- 无用测试文件清理；
- 工程配置整理；
- 低风险重复文件整理；
- 已确认无引用文件删除；
- 非业务代码的仓库结构治理；
- 文档内容去重和过时口径校准。

AI 可以：
1. 在用户确认的边界内一次性完成多个相关操作；
2. 不需要每移动、归档、删除、编辑一个文件都等待确认；
3. 先做引用检查和影响范围分析，再批量执行；
4. 只在触发暂停条件时停止；
5. 完成后输出最终收口报告。

### Supervised 执行规则

适用于统计口径、权限展示、列表筛选、导出逻辑、页面和 API 联动调整。

AI 可以：
1. 自动完成同一大阶段内的小步骤；
2. 大阶段结束后输出阶段报告；
3. 在核心逻辑阶段后触发 Codex Review；
4. Codex 无高风险问题时继续下一阶段；
5. Codex 有高风险问题时暂停。

### Locked 执行规则

适用于数据库 schema、审批流、状态机、权限模型、部署架构、数据迁移。

AI 不得直接编码。必须先输出：
1. 影响范围；
2. 当前代码调用链；
3. 实施方案；
4. 风险点；
5. 需要用户确认的问题。

### Codex Review 规则

Codex 只在关键节点介入：

1. Supervised 任务完成核心逻辑阶段后；
2. PR 从 Draft 转 Ready for Review 前；
3. CI/Vercel/类型检查失败并由 Claude 修复后；
4. 用户明确要求 review 时。

Codex Review 结果处理：

1. 轻微问题：Claude 可自动修复。
2. 高风险问题：Claude 必须暂停，交给用户/ChatGPT判断。

### 暂停条件

任何模式下，遇到以下情况必须暂停：

1. 修改 Prisma schema 或 migrations；
2. 修改审批流；
3. 修改状态机；
4. 修改角色权限模型；
5. 改变统计口径业务定义；
6. 修改 Docker / Vercel / Supabase / 群晖部署架构；
7. 删除、重置、迁移历史数据；
8. 引入大型依赖；
9. 认证、登录、JWT、安全逻辑变化；
10. CI/typecheck/build 连续两次修复失败；
11. Codex Review 指出高风险问题；
12. 需求与现有业务规则冲突；
13. 修改范围超过 Issue 要求。

## 分支规则

- `main` 是测试环境稳定分支，不允许直接提交。
- 不创建 `develop` 分支。
- Bug 修复使用 `fix/xxx`。
- 功能需求使用 `feature/xxx`。
- 权限、审批流、状态机、统计口径等业务规则调整使用 `logic/xxx`。
- 文档更新使用 `docs/xxx`。
- 依赖升级、配置维护、忽略文件、工具配置、工程化维护等不涉及业务逻辑的修改使用 `chore/xxx`。
- 所有修改通过 PR 合并到 `main`。
- PR 必须等待 CI 通过后才能合并。

## 修改说明要求

提交 PR 时使用 `.github/pull_request_template.md`。说明内容至少覆盖：

1. 修改目的和关联 Issue。
2. 修改文件。
3. 修改前逻辑和修改后逻辑。
4. 影响角色、页面、接口。
5. 是否影响权限、审批流、状态机、统计口径。
6. 是否涉及数据库结构或迁移。
7. 自测步骤和测试角色。
8. 风险点和回退方式。

## 本地检查

每次提交前至少确保：

```bash
pnpm lint
pnpm typecheck
pnpm build
```

项目也提供聚合检查：

```bash
pnpm check
```

涉及权限时必须说明测试角色；涉及统计时必须说明统计口径；涉及审批时必须说明状态流转前后变化。

## 部署规则

- 普通发布执行 `sh deploy.sh`。
- 涉及数据库结构、Prisma schema 或 migrations 变化时，先执行 `sh migrate.sh`，再执行 `sh deploy.sh`。
- 群晖部署命令统一使用 `docker-compose`，不使用 `docker compose`，不使用 `-f` 参数。
- 详细部署说明以 `docs/deploy/部署说明-群晖.md` 为准。
