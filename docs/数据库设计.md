# 数据库设计

## 当前状态口径

当前系统运行态统一为 9 状态：`DRAFT`、`PENDING_DECOMPOSE`、`PROPOSING`、`IN_PROGRESS`、`ADJUSTING`、`CANCELLING`、`COMPLETING`、`COMPLETED`、`CANCELLED`。`REJECTED` 已取消，退回待修改由 `DRAFT` + 退回痕迹派生；`APPROVED` 已取消，立项通过后进入 `IN_PROGRESS`；完成审批统一为 `COMPLETING`；取消审批统一为 `CANCELLING`。

## 一、当前状态枚举

`WorkItemStatus` 当前已收敛为：

| 状态 | 说明 |
| --- | --- |
| `DRAFT` | 草稿/待修改 |
| `PENDING_DECOMPOSE` | 待分解 |
| `IN_PROGRESS` | 进行中 |
| `COMPLETED` | 已完成 |
| `CANCELLED` | 已取消 |
| `PROPOSING` | 立项/分解方案审批中 |
| `ADJUSTING` | 调整审批中 |
| `CANCELLING` | 取消审批中 |
| `COMPLETING` | 完成审批中 |

## 二、新增审批类型枚举

当前已新增：

```prisma
enum ApprovalType {
  PROPOSE
  ADJUST
  CANCEL
  COMPLETE
}
```

## 三、WorkItem 目标字段

当前已新增：

```prisma
beforeApprovalStatus WorkItemStatus?
approvalType         ApprovalType?
```

字段规则：

1. 进入审批态时写入 `beforeApprovalStatus` 和 `approvalType`。
2. 审批退回时恢复 `beforeApprovalStatus`。
3. 审批结束时清空审批辅助字段和当前审批人字段。

## 五、系统用户字段

`User` 用于登录、审批、办理、上传和权限判断。

| 字段 | 说明 |
| --- | --- |
| `creatorId` | 系统创建人 / 系统填报人 / 经办账号 |
| `firstSubmitterId` | 首次提交人，用于立项审批退回后的处理参考 |
| `currentApproverId` | 当前审批节点用户 |
| `currentApproverRole` | 当前审批节点角色 |
| `proposedLeaderId` | 待办提出领导或相关公司领导 |
| `approvalLeaderId` | 审批领导，后续应明确与 `proposedLeaderId` 的职责边界 |

## 六、业务责任人字段

业务责任人不一定登录系统，只用于展示、导入导出和历史留痕。

### 重点工作 / 主要工作

目标字段：

| 字段 | 说明 |
| --- | --- |
| `responsibleLeader` | 事项业务责任领导姓名文本 |
| `responsiblePerson` | 事项实际责任人姓名文本 |

说明：

1. `responsibleLeader` 不等于系统部门领导账号。
2. `responsiblePerson` 不等于 `DEPT_MANAGER`。
3. 二者不是 `User` 外键。
4. 二者不是 `Member` 外键。
5. 二者不参与审批权限、办理权限、数据可见权限判断。
6. `deptLeaderId`、`deptManagerId` 已在 Phase 7 中删除，不再作为当前字段。

### 待办事项

目标继续保留：

| 字段 | 说明 |
| --- | --- |
| `responsiblePersons` | 主责责任人姓名文本或 JSON 数组 |
| `cooperatePersons` | 配合责任人姓名文本或 JSON 数组 |

本轮不新增 `Member`、`DepartmentMember` 和 `WorkParticipant`。

## 七、部门字段

| 字段 | 目标语义 |
| --- | --- |
| `responsibleDepartmentIds` | 主责部门 ID 列表（Phase 7 已从 departmentIds 重命名） |
| `departmentId` | 单主责部门 / 历史兼容字段，短期不彻底删除 |
| `cooperateDepartmentIds` | 配合部门 ID 列表 |

Phase 7 已完成 `departmentIds` → `responsibleDepartmentIds` 的迁移，无需兼容旧字段。`getResponsibleDepartmentIds(work)` 读取 `responsibleDepartmentIds`，为空时回退 `departmentId`。

## 八、Member / DepartmentMember 说明

本轮不强制新增 `Member` 或 `DepartmentMember` 表。

如果后续引入，`Member` / `DepartmentMember` 应表示“部门人员名录 / 业务人员名录”，不等于普通员工登录账号。成员可以是部门领导、部门副职、专业负责人、普通业务人员，也可以不登录系统。责任领导和责任人都可以来自成员名录，但成员名录不天然产生审批、办理或可见权限。

## 九、清库与迁移

第六阶段 PR 6.1 采用清库重建方案 A：删除旧测试数据，执行 9 状态 schema migration，并重新 seed 目标 9 状态测试数据。

即使采用清库路径，也必须保留：

1. 旧状态到新状态映射。
2. 目标字段设计。
3. 测试数据重建方式。
4. 未来正式数据迁移参考说明。

如果后续不能清库，则需要一次性迁移脚本处理状态、审批辅助字段、部门字段和责任人字段；旧状态映射以上表为准。
