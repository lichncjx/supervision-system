# 状态机设计

## 零、当前 9 状态

`src/lib/work-status.ts` 当前只维护目标 9 个 `WorkItemStatus` 的展示文案、颜色分组和基础判断。旧状态不再作为当前运行口径。

统一元数据字段包括：

1. `value`：当前代码使用的小写状态值。
2. `prismaValue`：Prisma enum 名称。
3. `label` / `description`：页面、导出和流程记录展示文案。
4. `visualGroup` / `badgeClass`：状态视觉分组和 Badge 样式。
5. `isTerminal`：是否终态，目前仅 `COMPLETED`、`CANCELLED`。
6. `isApproving`：是否处于审批过程。
7. `isHandling`：是否处于业务办理/待处理过程。
8. `isInProgress`：是否沿用当前“进行中”统计/筛选口径。
9. `countsForDeadline`：是否计入临期/超期，`COMPLETED`、`CANCELLED` 不计入。
10. `isLegacy`：当前固定为 `false`，旧展示兼容状态已不再纳入运行口径。

当前 9 状态元数据口径：

| 状态 | 文案 | 视觉分组 | 终态 | 审批中 | 办理中 | 进行中 | 计入临期/超期 |
| --- | --- | --- | --- | --- | --- | --- | --- |
| `DRAFT` | 草稿 | `handling` | 否 | 否 | 是 | 否 | 是 |
| `PENDING_DECOMPOSE` | 待分解 | `handling` | 否 | 否 | 是 | 否 | 是 |
| `PROPOSING` | 立项审批中 | `approving` | 否 | 是 | 否 | 否 | 是 |
| `IN_PROGRESS` | 进行中 | `inProgress` | 否 | 否 | 是 | 是 | 是 |
| `ADJUSTING` | 调整审批中 | `approving` | 否 | 是 | 否 | 否 | 是 |
| `CANCELLING` | 取消审批中 | `approving` | 否 | 是 | 否 | 否 | 是 |
| `COMPLETING` | 完成审批中 | `approving` | 否 | 是 | 否 | 否 | 是 |
| `COMPLETED` | 已完成 | `completed` | 是 | 否 | 否 | 否 | 否 |
| `CANCELLED` | 已取消 | `cancelled` | 是 | 否 | 否 | 否 | 否 |

兼容状态 `pending_approval`、`active`、`decomposed` 已不再作为当前状态元数据维护。

## 一、目标状态集合

### 稳定业务态

| 状态 | 说明 |
| --- | --- |
| `DRAFT` | 草稿/待修改 |
| `PENDING_DECOMPOSE` | 待分解，仅服务于公司领导发起的待办事项 |
| `IN_PROGRESS` | 进行中 |
| `COMPLETED` | 已完成 |
| `CANCELLED` | 已取消 |

### 审批过程态

| 状态 | 说明 |
| --- | --- |
| `PROPOSING` | 立项/分解方案审批中 |
| `ADJUSTING` | 调整审批中 |
| `CANCELLING` | 取消审批中 |
| `COMPLETING` | 完成审批中 |

## 三、状态设计原则

1. `APPROVED` 已取消，审批通过后统一进入 `IN_PROGRESS`。
2. `REJECTED` 已取消，立项审批退回后回到 `DRAFT`，退回痕迹通过退回字段和 workflow record 展示。
3. `PENDING_EVIDENCE_DEPT`、`PENDING_EVIDENCE_COMPANY`、`PENDING_COMPLETE` 已统一为 `COMPLETING`。
4. `PENDING_MAIN_LEADER_CANCEL` 不再单独作为状态，而是在 `CANCELLING` 状态下通过 `currentApproverId`、`currentApproverRole`、`approvalType` 判断当前审批节点。
5. `PENDING_DECOMPOSE` 保留。它不是审批态，而是公司领导发起待办后的业务准备/待分解状态。
6. 任何审批态都必须能通过 `beforeApprovalStatus` 找到退回后的业务状态。

## 四、审批辅助字段

PR 6.1 已在 `WorkItem` 上新增：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `beforeApprovalStatus` | `WorkItemStatus?` | 进入审批态之前的业务状态，用于退回恢复 |
| `approvalType` | `ApprovalType?` | 当前审批类型 |

`ApprovalType` 当前包括：

| 枚举值 | 说明 |
| --- | --- |
| `PROPOSE` | 立项或分解方案审批 |
| `ADJUST` | 调整审批 |
| `CANCEL` | 取消审批 |
| `COMPLETE` | 完成审批 |

## 五、进入审批态规则

进入 `PROPOSING`、`ADJUSTING`、`CANCELLING`、`COMPLETING` 时统一写入：

1. `beforeApprovalStatus = 当前业务状态`。
2. `approvalType = 当前审批类型`。
3. `status = 对应审批态`。
4. `currentApproverId` 或 `currentApproverRole` 指向第一审批节点。

示例：

| 场景 | 进入审批前 | 审批态 | `beforeApprovalStatus` | `approvalType` |
| --- | --- | --- | --- | --- |
| 普通事项提交立项 | `DRAFT` | `PROPOSING` | `DRAFT` | `PROPOSE` |
| 公司领导发起待办后部门分解提交 | `PENDING_DECOMPOSE` | `PROPOSING` | `PENDING_DECOMPOSE` | `PROPOSE` |
| 进行中事项申请调整 | `IN_PROGRESS` | `ADJUSTING` | `IN_PROGRESS` | `ADJUST` |
| 进行中事项申请取消 | `IN_PROGRESS` | `CANCELLING` | `IN_PROGRESS` | `CANCEL` |
| 进行中事项提交完成材料 | `IN_PROGRESS` | `COMPLETING` | `IN_PROGRESS` | `COMPLETE` |

## 六、审批通过规则

1. 如果还有下一审批节点，`status` 保持当前审批态，仅切换 `currentApproverId` / `currentApproverRole`。
2. 如果审批结束，则进入目标业务状态，并清空：
   - `beforeApprovalStatus`
   - `approvalType`
   - `currentApproverId`
   - `currentApproverRole`
3. `PROPOSE` 审批结束后进入 `IN_PROGRESS`。
4. `ADJUST` 审批结束后进入 `IN_PROGRESS`。
5. `CANCEL` 审批结束后进入 `CANCELLED`。
6. `COMPLETE` 审批结束后进入 `COMPLETED`。

## 七、审批退回规则

审批退回时统一执行：

1. `status = beforeApprovalStatus`。
2. `beforeApprovalStatus = null`。
3. `approvalType = null`。
4. `currentApproverId = null`。
5. `currentApproverRole = null`。
6. 写入退回原因和 workflow record。

特别规则：

1. 普通事项 `DRAFT -> PROPOSING`，退回后回 `DRAFT`。
2. 公司领导发起待办 `PENDING_DECOMPOSE -> PROPOSING`，退回后回 `PENDING_DECOMPOSE`。
3. 不使用 `needDecompose` 之类布尔字段解决退回路径，退回路径只以 `beforeApprovalStatus` 为准。

## 八、取消审批中的主要领导节点

重点工作取消不再使用 `PENDING_MAIN_LEADER_CANCEL`。在目标状态机中：

1. 事项保持 `CANCELLING`。
2. `approvalType = CANCEL`。
3. 当前审批节点由 `currentApproverId` / `currentApproverRole` 表示。
4. 当需要公司主要领导审批时，设置 `currentApproverRole = PRESIDENT` 或精确设置 `currentApproverId`。

## 九、兼容和迁移说明

1. PR 6.1 采用清库重建方案 A，target-contract seed 已直接使用目标 9 状态。
2. 旧 `APPROVED` 数据如需保留，应迁移为 `IN_PROGRESS`。
3. 旧 `REJECTED` 数据如需保留，应迁移为 `DRAFT`，退回原因保留在历史字段和 workflow record 中。
4. 旧部门/公司审批中的事项应统一迁移为 `PROPOSING`，审批节点由当前审批人字段表达。
5. 旧完成审批中的事项应统一迁移为 `COMPLETING`。
6. 旧重点工作主要领导取消审批应迁移为 `CANCELLING`，并保留主要领导审批节点。
