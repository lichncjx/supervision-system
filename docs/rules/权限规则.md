# 权限规则

> 当前系统运行态为 9 状态，详见 [状态机设计](状态机设计.md)。

## 零、统一权限模块

第二阶段新增/完善 `src/lib/server-permissions.ts`，作为服务端权限和数据范围的集中入口。第六阶段 PR 6.2 已将 workflow POST 的审批和办理判断同步到该模块。

当前统一 helper 包括：

1. `getResponsibleDepartmentIds(work)`：返回 `work.departmentId ? [work.departmentId] : []`（Phase 8B 统一为单一主责部门）。
2. `getCooperatorDepartmentIds(work)`：从 `work.cooperators` 解析 `departmentId` 并返回配合部门 ID 数组。`getCooperateDepartmentIds` 已标记为 deprecated，指向 `getCooperatorDepartmentIds`。
3. `buildWorkVisibilityWhere(user)`：构造事项可见范围查询条件。
4. `canViewWorkItem(user, work)`：事项查看权限。
5. `canApproveWorkItem(user, work)`：当前审批节点权限。
6. `canHandleWorkItem(user, work)`：部门办理权限。
7. `canEditWorkItem(user, work)`：草稿/退回编辑权限。
8. `canViewAttachment(user, work)`、`canUploadAttachment(user, work)`、`canDeleteAttachment(user, work, attachment)`：附件权限。

已接入范围：

1. `GET /api/works`
2. `GET /api/works/[id]`
3. `PUT /api/works/[id]`
4. `GET /api/works/[id]/workflow`
5. `GET /api/dashboard/summary`
6. `GET /api/dashboard/completion-rate`
7. `GET /api/excel/completion-rate`
8. `GET /api/excel/export`
9. `POST /api/excel/import/[type]`
10. `src/lib/attachment-permissions.ts` 及上传/下载/删除附件接口
11. `POST /api/works/[id]/workflow`

PR 6.2 后 workflow 相关口径：

1. 审批态为 `PROPOSING`、`ADJUSTING`、`CANCELLING`、`COMPLETING`。
2. `canApproveWorkItem` 优先按 `currentApproverId` 精确匹配当前用户。
3. `currentApproverRole` 作为审批节点辅助，部门级审批还必须匹配主责部门范围。
4. 公司级审批尽量写入精确 `currentApproverId`，避免按 `VICE_PRESIDENT` 角色默认放开。
5. `canHandleWorkItem` 支持部门角色办理 `PENDING_DECOMPOSE`；`DRAFT` 仅 `creatorId` 对应创建人或 `firstSubmitterId ?? creatorId` 退回修改人可办理；`IN_PROGRESS` 仅在审批退回（`rejectReason` / `rejectedFromStatus` 非空）时由 `firstSubmitterId ?? creatorId` 可办理。

## 一、核心原则

1. 查询统计按组织权限。
2. 办理操作按部门管理权限。
3. 审批操作按 `currentApproverId` / `currentApproverRole`。
4. 核心内容修改必须走 `ADJUSTING`。
5. `User` 是系统登录账号，责任领导、责任人、主责责任人、配合责任人是业务责任人姓名记录。
6. `DEPT_MANAGER` 是部门事项管理岗，不是事项实际责任人。
7. `DEPT_LEADER` 和 `DEPT_MANAGER` 查看本部门相关事项时，不应按 `creatorId` / `firstSubmitterId` 限制。
8. 公司领导之间不能默认互相修改对方负责事项，`VICE_PRESIDENT` 视角应围绕其提出、分管、当前审批的事项。
9. `SUPERVISOR` 和 `ADMIN` 可以按全局口径查看。

## 二、User 与业务责任人边界

### User

`User` 用于：

1. 登录认证。
2. 审批权限判断。
3. 办理权限判断。
4. 上传和删除附件权限判断。
5. 数据可见范围判断。
6. workflow record 审计。

### 业务责任人

业务责任人用于：

1. 事项展示。
2. 责任留痕。
3. 导入导出。
4. 历史档案。

业务责任人不一定登录系统，也不参与系统审批、办理、数据可见权限判断。

## 三、关键系统字段

| 字段 | 权限语义 |
| --- | --- |
| `creatorId` | 系统创建人 / 系统填报人 / 事项在系统中的经办账号 |
| `firstSubmitterId` | 首次提交人，用于立项审批退回后的处理参考 |
| `currentApproverId` | 当前审批节点的精确用户 |
| `currentApproverRole` | 当前审批节点的角色 |
| `departmentId` | 唯一主责部门（Phase 8B 统一） |
| `cooperators` | 配合方列表 JSON（Phase 8B 新增） |

Phase 8B 已删除 `responsibleDepartmentIds`、`cooperateDepartmentIds`、`responsiblePersons`、`cooperatePersons`。
主责部门通过 `getResponsibleDepartmentIds(work)` 统一读取，返回 `[work.departmentId]`（单一主责部门）。
配合部门通过 `getCooperatorDepartmentIds(work)` 从 `work.cooperators` 解析。

## 四、查看权限

1. `ADMIN`、`SUPERVISOR` 按全局口径查看。
2. `DEPT_LEADER`、`DEPT_MANAGER` 可查看本部门相关事项：
   - `departmentId` 为本部门；
   - `cooperators[].departmentId` 包含本部门。
3. 部门角色查看本部门相关事项时，不按 `creatorId` 或 `firstSubmitterId` 限制。
4. `VICE_PRESIDENT` 默认查看其提出、分管、当前审批或明确授权范围内的事项，不默认拥有其他公司领导负责事项的修改权。
5. `PRESIDENT` 可查看其当前审批、本人提出、需要主要领导决策的事项；是否拥有全局查看应由业务授权明确配置。

## 五、办理权限

办理指不需要当前用户审批，但需要当前用户推进事项的操作，例如待分解、提交材料、退回后修改、部门办理。

1. 办理权限按部门管理权限判断。
2. `DEPT_MANAGER` 是部门事项管理岗，可以代表部门维护和提交本部门相关事项。
3. `DEPT_LEADER` 可办理其部门职责范围内需要部门处理的事项。
4. 责任领导、责任人、主责责任人、配合责任人姓名不产生办理权限。
5. 配合部门是否可办理应按具体动作定义；默认配合部门只参与可见和协同，不自动获得主责部门办理权。
6. 退回后修改优先参考 `firstSubmitterId`，但部门角色对本部门事项的查看不应因此受限。

## 六、审批权限

1. 审批权限只按当前审批节点判断。
2. `currentApproverId` 等于当前用户 ID 时可审批。
3. `currentApproverRole` 等于当前用户角色时可审批；部门级审批必须同时匹配部门范围。
4. 公司级审批不应通过“公司领导角色泛化”默认放开，应尽量使用精确 `currentApproverId`。
5. `approvalType` 用于区分当前审批是立项、调整、取消还是完成，不替代审批人判断。
6. 审批通过时如果还有下一节点，只切换当前审批人，状态保持当前审批态。

## 七、编辑与调整权限

1. 草稿和待修改内容可以由有办理权限的用户维护。
2. 事项进入 `IN_PROGRESS` 后，核心内容修改必须走 `ADJUSTING`。
3. 普通编辑接口不得直接修改状态、当前审批人或审批辅助字段。
4. 公司领导之间不能默认互相修改对方负责事项。
5. 业务责任人姓名字段不产生编辑权限。

## 八、附件权限

1. 附件查看应与事项查看权限保持一致。
2. 附件上传应与事项办理权限或当前动作权限保持一致。
3. 上传人可以删除自己上传的附件；`ADMIN` / `SUPERVISOR` 可按管理口径删除。
4. 业务责任人姓名不产生附件上传或删除权限。

## 九、导入导出权限

1. 普通导出按组织查看权限执行，数据范围与 `GET /api/works` 可见范围一致。
2. 普通导出使用 `buildWorkVisibilityWhere(user)` 和 `canViewWorkItem(user, work)`，不单独维护部门可见范围。
3. 普通导入按主责部门范围执行：部门用户只能导入主责部门包含本部门的事项。
4. 普通导入允许配合部门填写其他部门，但配合部门不能替代主责部门授权。
5. `ADMIN` / `SUPERVISOR` 可按全局范围导入导出。
6. `VICE_PRESIDENT` / `PRESIDENT` 不默认获得全局导入导出范围；导出跟随可见范围，导入仅允许本人提出或本人审批的待办事项。
7. 导入的责任领导、责任人、主责责任人、配合责任人是姓名记录，不做登录账号匹配要求。
8. 导入不能绕过状态机直接生成审批中或已完成状态，除非是明确的历史数据迁移任务。
# PR 6.3 当前补充

1. 当前系统运行态已经切换为 9 状态，审批态只包括 `PROPOSING`、`ADJUSTING`、`CANCELLING`、`COMPLETING`。
2. `APPROVED` 和 `REJECTED` 不再作为当前权限判断状态。
3. “退回待修改”通过 `DRAFT` + 退回痕迹派生；办理和编辑优先按 `firstSubmitterId` 判断，历史数据可回退到 `creatorId`。
4. `canHandleWorkItem` 对退回草稿不再按同部门泛化放开，避免同部门其他主管/领导误获得退回修改权限。
5. 普通编辑接口不得写 `beforeApprovalStatus`、`approvalType` 或直接修改事项状态；状态流转必须经过 workflow。
