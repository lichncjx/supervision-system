# 重构计划

> 第六阶段收尾状态：PR 6.1 至 PR 6.4 已完成或基本完成。当前系统运行态已经统一为 9 状态，旧状态映射仅作为历史迁移说明保留。本文后续章节包含早期规划和历史迁移背景；阅读当前业务规则时应以 `docs/业务规则.md`、`docs/状态机设计.md`、`docs/API说明.md`、`docs/首页统计口径.md` 为准。

本文档基于 `docs/现状审计报告.md` 和目标架构讨论结论制定。计划分为两层：第一层先收敛当前实现并建立保护网，第二层再迁移到 9 状态目标状态机和新的人员、权限、Dashboard 架构。

当前系统尚未正式上线，因此目标架构迁移阶段可以优先考虑清库重建测试数据；但仍必须保留旧状态到新状态的迁移映射，方便后续参考、回滚和正式数据迁移。

## 一、总体目标

1. 将当前 15 个 `WorkItemStatus` 收敛为 9 个目标状态。
2. 使用 `beforeApprovalStatus` 和 `approvalType` 解决审批态退回路径，不新增 `needDecompose` 之类布尔补丁字段。
3. 明确系统登录用户 `User` 与业务责任人的边界。
4. 重点/主要工作的责任领导、责任人回归业务姓名记录，不参与系统权限。
5. 待办事项责任人、配合责任人继续保留姓名数组，不引入人员 ID 级关联。
6. 首页改为统一 `GET /api/dashboard`，不再拉全量事项到前端计算首页列表。
7. 权限、统计、导入导出和页面展示统一使用同一套服务端口径。

## 二、实施原则

1. 权限和状态流转以服务端为准。
2. 状态变更只能通过 workflow 层完成。
3. 查询统计按组织权限，办理操作按部门管理权限，审批操作按 `currentApproverId` / `currentApproverRole`。
4. 核心内容修改必须走 `ADJUSTING`，普通编辑接口不直接修改已进入流程的核心内容。
5. 业务责任人姓名字段只用于展示、导入导出和历史留痕，不参与审批、办理、可见权限判断。
6. 每个阶段都要可测试、可回退、可说明影响范围。

## 三、第一层：现状收敛与保护

第一层不改变最终业务模型，目标是把当前分散逻辑收拢，降低迁移到目标架构时的风险。

## 四、第二层：目标架构迁移

第二层进入目标模型迁移，重点是 9 状态状态机、审批辅助字段、人员字段语义、统一 Dashboard API 和数据迁移。

### 阶段 6：新 9 状态状态机迁移

目标：将当前 15 状态迁移为 9 状态。

目标状态：

1. `DRAFT`：草稿/待修改。
2. `PENDING_DECOMPOSE`：待分解，仅服务于公司领导发起的待办事项。
3. `IN_PROGRESS`：进行中。
4. `COMPLETED`：已完成。
5. `CANCELLED`：已取消。
6. `PROPOSING`：立项/分解方案审批中。
7. `ADJUSTING`：调整审批中。
8. `CANCELLING`：取消审批中。
9. `COMPLETING`：完成审批中。

迁移规则：

1. `APPROVED` 取消，审批通过后统一进入 `IN_PROGRESS`。
2. `REJECTED` 取消，立项审批退回后回到 `DRAFT`。
3. `PENDING_EVIDENCE_DEPT`、`PENDING_EVIDENCE_COMPANY`、`PENDING_COMPLETE` 统一为 `COMPLETING`。
4. `PENDING_MAIN_LEADER_CANCEL` 不再单独作为状态，在 `CANCELLING` 下通过 `currentApproverId`、`currentApproverRole`、`approvalType` 判断当前审批节点。
5. `PENDING_DECOMPOSE` 保留，它不是审批态，而是公司领导发起待办后的业务准备/待分解状态。

验收：

1. 新状态机能覆盖当前所有业务路径。
2. 旧状态到新状态的迁移映射完整记录在 `docs/状态机设计.md` 和 `docs/数据库设计.md`。

### 阶段 7：新增 `beforeApprovalStatus` / `approvalType`

目标：用审批辅助字段统一处理审批态退回和多节点审批。

目标字段：

1. `beforeApprovalStatus WorkItemStatus?`
2. `approvalType ApprovalType?`

`ApprovalType` 建议值：

1. `PROPOSE`
2. `ADJUST`
3. `CANCEL`
4. `COMPLETE`

规则：

1. 进入审批态时，`beforeApprovalStatus = 当前业务状态`，`approvalType = 当前审批类型`。
2. 审批退回时，`status = beforeApprovalStatus`，并清空审批辅助字段和当前审批人字段。
3. 审批通过且仍有下一节点时，`status` 保持当前审批态，仅切换 `currentApproverId` / `currentApproverRole`。
4. 审批结束时，`status` 进入目标业务状态，并清空 `beforeApprovalStatus`、`approvalType`、`currentApproverId`、`currentApproverRole`。
5. 普通事项 `DRAFT -> PROPOSING`，退回后回 `DRAFT`。
6. 公司领导发起待办 `PENDING_DECOMPOSE -> PROPOSING`，退回后回 `PENDING_DECOMPOSE`。
7. 不使用 `needDecompose` 之类布尔字段解决退回路径。

验收：

1. 任意审批态退回都能从 `beforeApprovalStatus` 恢复。
2. 多节点审批不依赖拆分更多状态。

### 阶段 8：重点/主要事项责任领导、责任人语义和命名调整

目标：把业务责任人从系统用户权限体系中拆出来。

任务：

1. 重点工作、主要工作填写 `responsibleLeader` 和 `responsiblePerson`。
2. `responsibleLeader` 表示事项业务责任领导。
3. `responsiblePerson` 表示事项实际责任人。
4. 二者只用于展示、导入导出和历史留痕。
5. 二者不参与审批权限、办理权限、数据可见权限判断。
6. 审批权限仍依赖 `currentApproverId` / `currentApproverRole`。
7. 系统填报和退回处理仍依赖 `creatorId` / `firstSubmitterId`。
8. 部门办理权限仍依赖当前登录用户角色和部门。

说明：

如后续引入 `Member` / `DepartmentMember`，它表示“部门人员名录 / 业务人员名录”，不等于普通员工账号，也不要求能登录系统。责任领导和责任人都可以来自该名录。本轮不强制新增 `Member` 表。

### 阶段 9：待办事项责任人字段保留姓名数组逻辑

目标：保留当前轻量姓名记录，不引入多对多人员权限模型。

任务：

1. 保留 `responsiblePersons`，用于记录待办事项主责责任人姓名文本或 JSON 数组。
2. 保留 `cooperatePersons`，用于记录配合责任人姓名文本或 JSON 数组。
3. 暂不新增 `Member` 表。
4. 暂不新增 `DepartmentMember` 表。
5. 暂不新增 `WorkParticipant` 表。
6. `responsiblePersons` / `cooperatePersons` 只用于展示、导入导出、历史留痕。
7. `responsiblePersons` / `cooperatePersons` 不参与审批权限、办理权限、数据可见权限判断。
8. 将 `departmentIds` 目标命名为 `responsibleDepartmentIds`，短期保留 `departmentId` 作为单主责部门/历史兼容字段。
9. 配合部门继续使用 `cooperateDepartmentIds`。
10. 如果本轮做数据库迁移，则将 `departmentIds` 迁移/重命名为 `responsibleDepartmentIds`。
11. 如果本轮暂不做数据库迁移，则封装 `getResponsibleDepartmentIds(work)`，按 `responsibleDepartmentIds` -> `departmentIds` -> `departmentId` 顺序读取。
12. 权限、统计、导入导出中不要各自手写部门字段兼容逻辑，应通过统一 helper 或统一权限模块读取。

已接受风险：

1. 可能存在人员重名。
2. 人员调岗后历史姓名不会自动归档到新组织。
3. 无法按人员 ID 做精确统计、通知和权限控制。
4. 如后续出现人员维度统计、通知、权限控制需求，再单独评估 `DepartmentMember` / `WorkParticipant`。

### 阶段 10：统一 Dashboard API

目标：首页改为一次调用轻量 Dashboard API。

目标接口：

```http
GET /api/dashboard
```

响应结构：

```ts
{
  summary: DashboardSummary;
  lists: {
    expiringAndOverdue: DashboardWorkListItem[];
    myActionRequired: DashboardWorkListItem[];
  };
}
```

任务：

1. 首页不再调用 `/api/works` 拉全量事项后在前端计算列表。
2. `summary` 返回卡片统计。
3. `lists.expiringAndOverdue` 返回临超期轻量列表。
4. `lists.myActionRequired` 返回待处理轻量列表。
5. 轻量列表只返回首页需要字段，不返回完整 `WorkItem`。

验收：

1. 首页一次请求即可拿到卡片和轻量列表。
2. Dashboard 统计、临超期列表、待处理列表使用同一服务端权限口径。

## 七、回退策略

1. 第一层每个阶段只做收敛，不改变业务目标，按文件和测试可逐步回退。
2. 第二层状态机迁移前先保留完整旧状态映射。
3. 若采用清库路径，需先确认无正式数据，并保留种子数据和测试数据重建脚本。
4. 若采用迁移脚本路径，必须提供迁移前备份、迁移后核对和回滚 SQL。
5. Dashboard 新接口上线时可短期并存旧 `/api/dashboard/summary`，页面切换后再清理旧接口。
