# 事项责任方模型设计

> **实施状态（2026-05-10）：Phase 8B 已按本设计落地。** 当前模型：`departmentId`（唯一）+ `responsibleLeader` + `responsiblePerson` + `cooperators: [{ departmentId, departmentName?, leader?, person? }]`。旧字段 `responsibleDepartmentIds` / `cooperateDepartmentIds` / `responsiblePersons` / `cooperatePersons` 已删除。Phase 8C 已完成页面、Excel、Dashboard 完整适配。本文档保留作为设计记录。

## 一、背景

当前系统已完成 9 状态状态机、Workflow 流转、Dashboard、权限范围、Excel 导入导出和字段命名规范化等重构工作。随着责任字段逐步规范，发现当前事项责任模型仍存在一定不协调：

1. 待办事项曾支持多个主责部门，但从业务语义看，“主责部门”应当唯一，否则会出现“谁是主责中的主责”的问题。
2. 当前存在 `responsibleDepartmentIds`、`responsiblePersons`、`cooperateDepartmentIds`、`cooperatePersons` 等字段，部门和人员之间缺少一一对应关系。
3. 配合部门和配合责任人分开存储时，无法准确表达“哪个配合部门对应哪个责任人”。
4. 未来如果需要记录配合部门责任领导，现有字段结构不易扩展。
5. 重点工作、主要工作、待办事项的责任字段不完全统一，增加页面、导入导出和统计逻辑复杂度。

因此，拟对事项责任方模型进行进一步抽象和统一。

---

## 二、目标原则

本次模型设计遵循以下原则：

1. **主责方唯一**
   每个事项只允许一个主责部门。主责部门对应一个责任领导和一个责任人。

2. **配合方可多个**
   一个事项可以有多个配合部门，每个配合部门对应自己的配合责任人，未来可扩展配合责任领导。

3. **统一事项类型字段**
   重点工作、主要工作、待办事项统一使用同一套责任字段，不再为待办事项单独维护多主责字段。

4. **责任人仍为姓名文本**
   本阶段不引入 `Member / DepartmentMember / WorkParticipant`，不做人 ID 化，不做人员主数据关联。

5. **责任字段不参与权限判断**
   `responsibleLeader`、`responsiblePerson`、`cooperators[].leader`、`cooperators[].person` 仅用于展示、导入导出和历史留痕，不参与审批、办理、可见范围判断。

6. **权限仍基于组织和流程**
   可见范围、办理权限、审批权限仍基于 `departmentId`、`cooperators[].departmentId`、`currentApproverId`、`currentApproverRole`、用户角色等判断。

7. **不引入复杂多对多表**
   当前系统尚未正式上线，且责任方主要用于展示和业务留痕，暂不引入独立关联表，优先使用 JSON 结构表达配合方。

## 三、核心概念

### 1. 主责方

主责方表示事项的主要责任承担单位及其业务责任人。

每个事项有且只有一个主责方。

主责方由三个字段表达：

```ts
departmentId: number | null
responsibleLeader: string | null
responsiblePerson: string | null
```

含义如下：

| 字段                  | 含义         | 是否参与权限 |
| ------------------- | ---------- | ------ |
| `departmentId`      | 主责部门 ID    | 是      |
| `responsibleLeader` | 主责责任领导姓名文本 | 否      |
| `responsiblePerson` | 主责责任人姓名文本  | 否      |

说明：

* `departmentId` 是组织权限和完成率统计的核心字段。
* `responsibleLeader` 和 `responsiblePerson` 是业务留痕字段，不代表系统用户。
* `responsibleLeader` 不等于 `DEPARTMENT_LEADER` 系统角色。
* `responsiblePerson` 不等于 `DEPARTMENT_MANAGER` 系统角色。

---

### 2. 配合方

配合方表示参与协同办理的部门及其责任人员。

一个事项可以有多个配合方。

建议使用 JSON 字段：

```ts
cooperators: WorkCooperator[]
```

其中：

```ts
type WorkCooperator = {
  departmentId: number
  departmentName?: string
  leader?: string
  person?: string
}
```

字段含义：

| 字段               | 含义         | 是否必填 | 是否参与权限 |
| ---------------- | ---------- | ---: | -----: |
| `departmentId`   | 配合部门 ID    |    是 |      是 |
| `departmentName` | 配合部门名称快照   |   可选 |      否 |
| `leader`         | 配合责任领导姓名文本 |   可选 |      否 |
| `person`         | 配合责任人姓名文本  |   可选 |      否 |

说明：

* `departmentId` 用于判断配合部门可见范围。
* `departmentName` 是历史快照字段，可用于导出和展示。
* `leader` 为将来记录配合部门责任领导预留。
* `person` 表示该配合部门对应的配合责任人。
* 配合方中的人员字段不参与权限判断。

## 四、目标数据模型

### 1. WorkItem 目标字段

建议目标模型如下：

```prisma
model WorkItem {
  id                 Int      @id @default(autoincrement())

  departmentId        Int?
  responsibleLeader   String?
  responsiblePerson   String?

  cooperators         Json?

  // 其他已有字段保持不变
}
```

### 2. 字段保留与删除建议

目标保留：

```text
departmentId
responsibleLeader
responsiblePerson
cooperators
```

建议删除：

```text
responsibleDepartmentIds
cooperateDepartmentIds
responsiblePersons
cooperatePersons
```

说明：

* `responsibleDepartmentIds` 用于支持多个主责部门。若确认主责部门唯一，则该字段不再需要。
* `responsiblePersons` 用于待办事项多个主责责任人。若主责方唯一，则统一使用 `responsiblePerson`。
* `cooperateDepartmentIds` 与 `cooperatePersons` 分离存储，无法表达部门和人员对应关系，应由 `cooperators` 替代。
* `cooperatePersons` 删除后，配合责任人改为 `cooperators[].person`。
* 当前无正式历史数据，可以清库重建，不做旧数据兼容和 backfill。

## 五、不同事项类型统一规则

### 1. 重点工作

重点工作使用：

```text
departmentId
responsibleLeader
responsiblePerson
cooperators
```

典型示例：

```json
{
  "departmentId": 2,
  "responsibleLeader": "张主任",
  "responsiblePerson": "李工",
  "cooperators": [
    {
      "departmentId": 3,
      "departmentName": "工艺技术处",
      "leader": "王主任",
      "person": "赵工"
    }
  ]
}
```

---

### 2. 主要工作

主要工作使用同一结构：

```text
departmentId
responsibleLeader
responsiblePerson
cooperators
```

说明：

* 主要工作与重点工作不再使用不同责任字段。
* 责任领导和责任人均为姓名文本。
* 配合方按部门逐条添加。

---

### 3. 待办事项

待办事项也使用同一结构：

```text
departmentId
responsibleLeader
responsiblePerson
cooperators
```

说明：

* 待办事项不再支持多个主责部门。
* 待办事项主责部门为 `departmentId`。
* 待办事项主责责任领导为 `responsibleLeader`。
* 待办事项主责责任人为 `responsiblePerson`。
* 待办事项配合部门通过 `cooperators` 表达。
* 每个配合部门对应一个配合责任人，未来可选填配合责任领导。

## 六、页面交互设计建议

### 1. 主责方填写区

页面中统一展示为：

```text
主责部门
责任领导
责任人
```

字段对应：

| 页面字段 | 数据字段                |
| ---- | ------------------- |
| 主责部门 | `departmentId`      |
| 责任领导 | `responsibleLeader` |
| 责任人  | `responsiblePerson` |

输入方式：

* 主责部门：部门下拉选择；
* 责任领导：文本输入，后续可改为人员名录选择；
* 责任人：文本输入，后续可改为人员名录选择。

---

### 2. 配合方填写区

页面中不再使用：

```text
先选多个配合部门，再填多个配合责任人
```

而改为逐条添加：

```text
配合部门 + 配合责任领导 + 配合责任人
```

示例交互：

```text
[配合部门下拉] [配合责任领导输入] [配合责任人输入] [删除]
[添加配合方]
```

字段对应：

| 页面字段   | 数据字段                         |
| ------ | ---------------------------- |
| 配合部门   | `cooperators[].departmentId` |
| 配合责任领导 | `cooperators[].leader`       |
| 配合责任人  | `cooperators[].person`       |

其中：

* `leader` 可选；
* `person` 可选或必填，可根据业务规则决定；
* 同一事项中配合部门是否允许重复，建议不允许。

## 七、权限规则

### 1. 可见范围

事项对用户可见，满足以下任一条件：

1. 用户为 `ADMIN`；
2. 用户为 `SUPERVISOR`；
3. 用户是当前审批人；
4. 用户部门等于事项 `departmentId`；
5. 用户部门出现在 `cooperators[].departmentId`；
6. 公司领导按既定公司领导范围规则可见。

其中：

```text
responsibleLeader
responsiblePerson
cooperators[].leader
cooperators[].person
```

不参与可见范围判断。

---

### 2. 办理权限

办理权限仍基于：

```text
事项状态
用户角色
用户部门
主责部门
配合部门
```

建议规则：

* 主责部门的部门领导 / 部门事项管理岗可以办理主责相关事项；
* 配合部门的部门领导 / 部门事项管理岗可以查看或配合办理，具体办理动作可按现有权限规则控制；
* 责任人姓名文本不产生系统办理权限。

---

### 3. 审批权限

审批权限仍基于：

```text
currentApproverId
currentApproverRole
approvalType
status
```

责任字段不参与审批权限判断。

---

## 八、统计规则

### 1. 完成率统计

完成率只按主责部门统计：

```text
departmentId
```

不按配合部门统计完成率。

原因：

* 主责部门是事项最终责任承担部门；
* 配合部门只是协同参与；
* 若把配合部门纳入完成率，会造成同一事项重复计入多个部门完成率。

---

### 2. 首页统计

首页统计按用户可见范围统计：

```text
departmentId
cooperators[].departmentId
currentApproverId
角色范围
```

其中：

* 主责部门事项计入本部门可见；
* 配合部门事项也可进入本部门可见；
* 完成率仍只按 `departmentId` 归属。

---

### 3. 临期 / 超期统计

临期和超期统计不受责任方模型变化影响，仍按：

```text
计划完成时间
事项状态
用户可见范围
```

---

## 九、导入导出规则

### 1. 导入字段建议

Excel 导入字段建议统一为：

```text
主责部门
责任领导
责任人
配合方
```

其中配合方可以采用一种结构化文本格式，例如：

```text
工艺技术处|王主任|赵工；质量管理处|刘主任|钱工
```

解析为：

```json
[
  {
    "departmentId": 3,
    "departmentName": "工艺技术处",
    "leader": "王主任",
    "person": "赵工"
  },
  {
    "departmentId": 4,
    "departmentName": "质量管理处",
    "leader": "刘主任",
    "person": "钱工"
  }
]
```

也可以拆成多列：

```text
配合部门1
配合责任领导1
配合责任人1
配合部门2
配合责任领导2
配合责任人2
```

但这种方式扩展性较差，不建议长期使用。

---

### 2. 导出字段建议

导出字段建议为：

```text
主责部门
责任领导
责任人
配合方
```

其中配合方导出为：

```text
工艺技术处（责任领导：王主任，责任人：赵工）；质量管理处（责任领导：刘主任，责任人：钱工）
```

## 十一、与 Member 的关系

本阶段不引入 Member。

当前字段仍然是姓名文本：

```text
responsibleLeader
responsiblePerson
cooperators[].leader
cooperators[].person
```

如果未来引入人员名录，应优先扩展为：

```text
responsibleLeaderMemberId
responsiblePersonMemberId
cooperators[].leaderMemberId
cooperators[].personMemberId
```

不建议使用：

```text
deptLeaderMemberId
deptManagerMemberId
```

原因：

* `deptLeader` 容易被理解为系统部门领导角色；
* `deptManager` 容易被理解为部门事项管理岗；
* 当前表达的是事项业务责任人，不是系统登录角色。

## 十五、最终目标模型总结

最终事项责任方模型为：

```text
WorkItem
├── departmentId              主责部门，唯一
├── responsibleLeader         主责责任领导姓名文本
├── responsiblePerson         主责责任人姓名文本
└── cooperators               配合方列表
    ├── departmentId          配合部门
    ├── departmentName        配合部门名称快照，可选
    ├── leader                配合责任领导姓名文本，可选
    └── person                配合责任人姓名文本，可选
```

该模型的核心收益：

1. 主责关系唯一清晰；
2. 重点工作、主要工作、待办事项字段统一；
3. 配合部门和配合人员一一对应；
4. 为未来配合责任领导预留空间；
5. 暂不引入人员主数据，复杂度可控；
6. 权限仍基于部门和流程，不被姓名字段污染；
7. 完成率统计回归唯一主责部门；
8. 后续 preview seed、导入导出和页面表单会更简洁。
