## 一、当前状态枚举及使用位置

### 1. 状态枚举

`WorkItemStatus` 定义在 `prisma/schema.prisma`，数据库值通过 `@map` 映射为小写字符串。

| 枚举值 | 数据库值 | 当前语义 |
| --- | --- | --- |
| `DRAFT` | `draft` | 草稿，创建后未进入审批 |
| `PENDING_DEPT` | `pending_dept` | 等待部门负责人审批 |
| `PENDING_COMPANY` | `pending_company` | 等待公司领导审批 |
| `APPROVED` | `approved` | 重点工作、主要工作审批通过，进入执行或验收前状态 |
| `IN_PROGRESS` | `in_progress` | 待办事项执行中 |
| `PENDING_DECOMPOSE` | `pending_decompose` | 待办事项待分解 |
| `PENDING_COMPLETE` | `pending_complete` | 待办事项待完成确认 |
| `PENDING_EVIDENCE_DEPT` | `pending_evidence_dept` | 重点/主要工作完成材料待部门审批 |
| `PENDING_EVIDENCE_COMPANY` | `pending_evidence_company` | 重点/主要工作完成材料待公司审批 |
| `PENDING_MAIN_LEADER_CANCEL` | `pending_main_leader_cancel` | 重点工作取消待主要领导审批 |
| `COMPLETED` | `completed` | 已完成 |
| `REJECTED` | `rejected` | 创建审批被退回，等待首次提交人修改 |
| `ADJUSTING` | `adjusting` | 调整申请审批中 |
| `CANCELLING` | `cancelling` | 取消申请审批中 |
| `CANCELLED` | `cancelled` | 已取消 |

## 二、当前状态流转路径

状态流转主要集中在 `src/lib/workflow.ts`。

### 1. 创建与提交审批

| 事项类型 | 提交人角色 | 初始状态 | 提交后状态 | 当前审批人来源 |
| --- | --- | --- | --- | --- |
| 重点/主要工作 | 部门主管 | `DRAFT`/`REJECTED` | `PENDING_DEPT` | `DEPARTMENT_LEADER` |
| 重点/主要工作 | 部门负责人 | `DRAFT`/`REJECTED` | `PENDING_COMPANY` | `VICE_PRESIDENT` |
| 待办事项 | 公司领导 | `DRAFT`/`REJECTED` | `PENDING_DECOMPOSE` | 无 |
| 待办事项 | 部门主管 | `DRAFT`/`REJECTED` | `PENDING_DEPT` | `DEPARTMENT_LEADER` |
| 待办事项 | 部门负责人 | `DRAFT`/`REJECTED` | `PENDING_COMPANY` | `proposedLeaderId` |

`firstSubmitterId` 在首次提交到 `PENDING_DEPT` 或 `PENDING_COMPANY` 时写入；公司领导直接提交待办到 `PENDING_DECOMPOSE` 时当前不会写入该字段。

### 2. 正常审批通过

| 当前状态 | 事项类型 | 审批通过后状态 | 后续审批人 |
| --- | --- | --- | --- |
| `PENDING_DEPT` | 待办事项 | `PENDING_COMPANY` | `proposedLeaderId` |
| `PENDING_DEPT` | 重点/主要工作 | `PENDING_COMPANY` | `VICE_PRESIDENT` |
| `PENDING_COMPANY` | 待办事项 | `IN_PROGRESS` | 无 |
| `PENDING_COMPANY` | 重点/主要工作 | `APPROVED` | 无 |
| `PENDING_EVIDENCE_DEPT` | 重点/主要工作 | `PENDING_EVIDENCE_COMPANY` | `VICE_PRESIDENT` |
| `PENDING_EVIDENCE_COMPANY` | 重点/主要工作 | `COMPLETED` | 无 |
| `PENDING_COMPLETE` | 待办事项 | `COMPLETED` | 无 |

### 3. 审批退回

| 当前状态 | 退回后状态 | 说明 |
| --- | --- | --- |
| `PENDING_DEPT` / `PENDING_COMPANY` | `REJECTED` | 创建审批退回，后续由 `firstSubmitterId ?? creatorId` 修改和重新提交 |
| `PENDING_EVIDENCE_DEPT` / `PENDING_EVIDENCE_COMPANY` | `APPROVED` | 完成材料退回，回到已审批待补材料状态 |
| `ADJUSTING` | 待办回 `IN_PROGRESS`，重点/主要回 `APPROVED` | 调整申请退回 |
| `CANCELLING` | 待办回 `IN_PROGRESS`，重点/主要回 `APPROVED` | 取消申请退回 |
| `PENDING_MAIN_LEADER_CANCEL` | `APPROVED` | 重点工作主要领导取消审批退回 |

### 4. 完成材料与完成确认

| 事项类型 | 提交材料前状态 | 提交材料后状态 | 审批路径 |
| --- | --- | --- | --- |
| 待办事项 | `IN_PROGRESS` | `PENDING_COMPLETE` | `proposedLeaderId` 确认后 `COMPLETED` |
| 重点/主要工作，部门主管提交 | `APPROVED` | `PENDING_EVIDENCE_DEPT` | 部门负责人审批后到公司审批 |
| 重点/主要工作，部门负责人提交 | `APPROVED` | `PENDING_EVIDENCE_COMPANY` | 公司领导审批后 `COMPLETED` |

### 5. 调整申请

| 事项类型 | 发起角色 | 调整中状态 | 审批路径 | 通过后状态 |
| --- | --- | --- | --- | --- |
| 待办事项 | 部门主管 | `ADJUSTING` | 部门负责人 -> `proposedLeaderId` | `IN_PROGRESS` |
| 待办事项 | 部门负责人 | `ADJUSTING` | `proposedLeaderId` | `IN_PROGRESS` |
| 重点/主要工作 | 部门主管 | `ADJUSTING` | 部门负责人 -> `VICE_PRESIDENT` | `APPROVED` |
| 重点/主要工作 | 部门负责人 | `ADJUSTING` | `VICE_PRESIDENT` | `APPROVED` |

### 6. 取消申请

| 事项类型 | 发起角色 | 取消中状态 | 审批路径 | 通过后状态 |
| --- | --- | --- | --- | --- |
| 待办事项 | 部门主管 | `CANCELLING` | 部门负责人 -> `proposedLeaderId` | `CANCELLED` |
| 待办事项 | 部门负责人 | `CANCELLING` | `proposedLeaderId` | `CANCELLED` |
| 待办事项 | 公司领导 | `CANCELLING` | 部门负责人 -> `proposedLeaderId` | `CANCELLED` |
| 主要工作 | 部门主管/负责人 | `CANCELLING` | 部门负责人或 `VICE_PRESIDENT` | `CANCELLED` |
| 重点工作 | 部门主管/负责人 | `CANCELLING` | 部门负责人或 `VICE_PRESIDENT` -> `PENDING_MAIN_LEADER_CANCEL` -> `PRESIDENT` | `CANCELLED` |

### 7. 待办分解

| 当前状态 | 操作角色 | 操作后状态 | 说明 |
| --- | --- | --- | --- |
| `PENDING_DECOMPOSE` | 部门主管 | `PENDING_DEPT` | 进入部门负责人审批 |
| `PENDING_DECOMPOSE` | 部门负责人 | `PENDING_COMPANY` | 进入 `proposedLeaderId` 审批 |

## 三、当前首页调用的 API 和前端计算字段

首页文件为 `src/app/(app)/page.tsx`。

### 1. 首页数据来源

| 来源 | 调用位置 | 用途 |
| --- | --- | --- |
| `getVisibleWorks(user)` | 首页服务端组件直接调用 | 获取当前用户可见事项，用于前端侧逾期、即将到期和待处理列表计算 |
| `GET /api/dashboard/summary` | 首页服务端组件通过 `fetchDashboardSummary()` 调用 | 获取首页卡片统计 |
| `GET /api/excel/completion-rate` | 首页服务端组件通过 `fetchCompletionRate()` 调用 | 获取各部门完成率 |

### 2. 首页卡片字段来源

| 首页字段 | 当前来源 | 说明 |
| --- | --- | --- |
| 重点工作总数 | `/api/dashboard/summary.priorityTotal` | API 按 `type=PRIORITY` 统计 |
| 主要工作总数 | `/api/dashboard/summary.mainTotal` | API 按 `type=MAIN` 统计 |
| 待办事项总数 | `/api/dashboard/summary.todoTotal` | API 按 `type=TODO` 统计 |
| 待审批 | `/api/dashboard/summary.approving` | API 读取相关事项后用本地 `canApproveWorkOnServer()` 计算 |
| 待办理 | `/api/dashboard/summary.handling` | API 读取相关事项后用本地 `canHandleWorkOnServer()` 计算 |
| 进行中 | `/api/dashboard/summary.inProgress` | API 按状态分组统计 |
| 已完成 | `/api/dashboard/summary.completed` | API 按 `COMPLETED` 统计 |
| 已取消 | `/api/dashboard/summary.cancelled` | API 按 `CANCELLED` 统计 |
| 逾期 | 首页前端用 `visibleWorks.filter(isOverdueWork)` 计算 | 未直接使用 summary API 的 `overdue` 字段 |
| 即将到期 | 首页前端用 `visibleWorks.filter(isExpiringWork)` 计算 | 未直接使用 summary API 的 `thisMonthDue` 字段 |
| 类型完成数 | `/api/dashboard/summary.priorityCompleted/mainCompleted/todoCompleted` | API 按类型和 `COMPLETED` 统计 |

### 3. 当前统计口径差异

`/api/dashboard/summary` 中普通总数、完成数、进行中、逾期等字段对部门用户主要使用 `departmentId` 标量字段过滤；但待审批和待办理会先构造 `allRelevantWorks`，同时考虑 `departmentId`、`departmentIds`、`cooperateDepartmentIds`、`currentApproverId`。

前端 `getVisibleWorks()` 也认为待办事项可能通过 `departmentIds` 和 `cooperateDepartmentIds` 与用户相关。因此首页不同卡片之间存在潜在口径差异。

| 低 | `src/lib/permissions.ts`、`src/components/common/permission-helpers.tsx` | 通用权限 helper 与业务工作流权限边界需要澄清 |

## 七、潜在风险点

1. 状态流转虽然集中在 `workflow.ts`，但 `PUT /api/works/[id]` 和前端 `updateWork()` 仍保留写入 `status` 的路径，存在绕过 workflow 层的风险。
2. 首页统计、列表查询、详情页访问、导出统计对部门用户的数据范围口径不一致；部分地方只看 `departmentId`，部分地方同时看 `departmentIds` 和 `cooperateDepartmentIds`。
3. 待审批和待办理判断分散在 `workflow.ts`、`work-store.ts`、Dashboard API 和页面中，`ADJUSTING` 等状态在不同状态分组里的归类不完全一致。
4. 首页 `overdue` 和 `expiring` 由前端基于 `getVisibleWorks()` 计算，而 summary API 也返回 `overdue` 和 `thisMonthDue`，两套字段容易漂移。
5. 完成率和逾期统计中对 `REJECTED` 是否排除的口径不一致；前端逾期排除退回，部分统计 API 只排除已完成和已取消。
6. `approvalLeaderId`、`proposedLeaderId`、`currentApproverId` 在待办审批、调整、取消中职责边界不清，页面可选择拟审批领导，但 workflow 多数路径仍按 `proposedLeaderId` 或当前审批人推进。
7. 附件权限没有完整覆盖 TODO 的 `departmentIds` 和 `cooperateDepartmentIds`，可能导致相关部门人员可见事项但不可查看或上传附件。
8. 导入 TODO 时，部门用户的多部门权限校验不如重点/主要工作明确，存在跨部门导入风险。
9. 公司层角色在 `/api/works` 可取到较宽数据，但前端 `getVisibleWorks()` 会隐藏部分公司前置审批状态，首页统计和列表展示可能不一致。
10. 状态标签和颜色分散在多个文件，且 `badges.tsx` 中仍有历史状态值，后续新增或调整状态时容易漏改。
11. 详情页和操作面板的按钮显示逻辑与服务端真实可执行动作不完全一致，用户可能看到按钮但提交后被服务端拒绝。
12. `server-permissions.ts` 已包含较完整的数据访问判断，但主 API 未统一使用，导致统一修复成本升高。

## 八、建议分阶段实施顺序

### 第 0 阶段：冻结现状口径

补齐状态、权限、统计的测试矩阵，先把当前可接受行为固定下来。重点覆盖创建提交、审批退回、完成材料、调整取消、待我处理、首页卡片和导入导出。

### 第 1 阶段：统一状态元数据

建立唯一状态元数据模块，集中维护状态枚举、中文文案、颜色、状态分组、是否终态、是否审批中、是否办理中等信息。替换 `work-store.ts`、`badges.tsx`、`status-colors.ts`、导出 API 中的重复状态映射。

### 第 2 阶段：统一服务端权限与数据范围

以服务端为准，整理 `canViewWorkItem`、`canSubmitWorkItem`、`canApproveWorkItem`、`canHandleWorkItem`、`canEditWorkItem`、`canUploadAttachment` 等能力判断，并让列表、详情、Dashboard、导出、附件接口复用同一套数据范围构造。

### 第 3 阶段：收敛状态流转入口

移除普通编辑接口直接写入 `status` 的路径，所有状态变更只允许通过 workflow 动作完成。将状态迁移、审批人分配、`firstSubmitterId` 写入、退回恢复状态整理为可测试的状态机。

### 第 4 阶段：统一首页、列表和统计口径

让首页卡片、状态页、待我处理页、完成率统计、导出统计复用同一批查询和权限函数，明确 TODO 多部门、协办部门、退回、逾期、取消事项的统计规则。

### 第 5 阶段：梳理人员字段职责

明确 ID 字段和名称快照字段的边界，统一 `creatorId`、`firstSubmitterId`、`deptLeaderId`、`deptManagerId`、`proposedLeaderId`、`approvalLeaderId`、`currentApproverId` 的使用场景。对历史文本字段仅保留展示或兼容职责。

### 第 6 阶段：迁移导入导出与页面展示

在服务端权限、状态元数据和统计口径统一后，再改造导入导出、详情页按钮、操作面板、状态页和待处理页，避免页面层先行改动造成新的口径分裂。

## 九、目标架构修订补充

本报告前八节描述当前实现现状。根据后续目标架构讨论，后续重构不止是收敛现状，还应迁移到 9 状态状态机，并重新明确系统用户、业务责任人、权限和 Dashboard API 设计。

### 1. 15 状态到 9 状态

当前 15 个 `WorkItemStatus` 后续计划收敛为 9 个状态：

| 旧状态 | 新状态 |
| --- | --- |
| `DRAFT` | `DRAFT` |
| `PENDING_DEPT` | `PROPOSING` |
| `PENDING_COMPANY` | `PROPOSING` |
| `APPROVED` | `IN_PROGRESS` |
| `IN_PROGRESS` | `IN_PROGRESS` |
| `PENDING_DECOMPOSE` | `PENDING_DECOMPOSE` |
| `PENDING_COMPLETE` | `COMPLETING` |
| `PENDING_EVIDENCE_DEPT` | `COMPLETING` |
| `PENDING_EVIDENCE_COMPANY` | `COMPLETING` |
| `PENDING_MAIN_LEADER_CANCEL` | `CANCELLING` |
| `COMPLETED` | `COMPLETED` |
| `REJECTED` | `DRAFT` |
| `ADJUSTING` | `ADJUSTING` |
| `CANCELLING` | `CANCELLING` |
| `CANCELLED` | `CANCELLED` |

`APPROVED` 和 `REJECTED` 不再作为目标状态；完成审批统一使用 `COMPLETING`；重点工作主要领导取消审批不再使用单独状态，而是在 `CANCELLING` 下通过当前审批节点表达。

### 2. 审批辅助字段

目标架构需要新增 `beforeApprovalStatus WorkItemStatus?` 和 `approvalType ApprovalType?`。进入审批态时记录审批前业务状态和审批类型；退回时恢复 `beforeApprovalStatus`；审批通过如仍有下一节点则保持审批态，只切换当前审批人；审批结束后进入目标状态并清空审批辅助字段。

### 3. User 与业务责任人边界

后续应明确：

1. `User` 只用于登录、审批、办理、上传、权限判断。
2. 责任领导、责任人、主责责任人、配合责任人是业务责任人姓名记录，只用于展示、导入导出和历史留痕。
3. `DEPARTMENT_MANAGER` 是部门事项管理岗，不是事项实际责任人。
4. 重点/主要工作的目标字段为 `responsibleLeader` 和 `responsiblePerson`，二者不参与审批、办理、可见权限。
5. 待办事项继续保留 `responsiblePersons`、`cooperatePersons` 姓名数组逻辑，本轮不新增 `Member`、`DepartmentMember` 或 `WorkParticipant`。

### 4. 权限目标原则

目标权限应按以下原则重建：

1. 查询统计按组织权限。
2. 办理操作按部门管理权限。
3. 审批操作按 `currentApproverId` / `currentApproverRole`。
4. 核心内容修改必须走 `ADJUSTING`。
5. 部门领导和部门事项管理岗查看本部门相关事项时，不按 `creatorId` / `firstSubmitterId` 限制。
6. 公司领导之间不能默认互相修改对方负责事项。
7. `SUPERVISOR` 和 `ADMIN` 可以按全局口径查看。

### 5. Dashboard 目标接口

后续首页应统一使用 `GET /api/dashboard`，一次返回：

1. `summary`：卡片统计；
2. `lists.expiringAndOverdue`：临超期轻量列表；
3. `lists.myActionRequired`：待处理轻量列表。

首页不再调用 `/api/works` 拉全量事项后在前端计算列表。TODO 多主责部门后续优先使用 `responsibleDepartmentIds`，为空时兼容回退 `departmentId`；配合部门继续使用 `cooperateDepartmentIds`。
