## 二、按 PR 拆分的实施 checklist（评审修正版）

### PR 6.1：9 状态 schema + 编译最小闭环

> **评审修正**：V1 中 PR 6.1 只改 schema 不改业务代码，会导致 `tsc --noEmit` 大量失败、项目可能无法启动，这样的 PR 无法单独合并。
> V2 调整为"编译最小闭环"——schema 改完后，同步修改所有依赖旧 enum 的引用以保证类型检查通过。workflow 深度逻辑留到 PR 6.2。

**要做什么：**

- [ ] 修改 `prisma/schema.prisma`：
  - `WorkItemStatus` 从 15 状态收敛为 9 状态（`DRAFT`, `PENDING_DECOMPOSE`, `PROPOSING`, `IN_PROGRESS`, `ADJUSTING`, `CANCELLING`, `COMPLETING`, `COMPLETED`, `CANCELLED`）
  - 新增 `ApprovalType` 枚举（`PROPOSE`, `ADJUST`, `CANCEL`, `COMPLETE`）
  - `WorkItem` 新增 `beforeApprovalStatus WorkItemStatus?`
  - `WorkItem` 新增 `approvalType ApprovalType?`
- [ ] 采用方案 A 清库重建，生成新 migration
- [ ] **（关键修正）同步更新所有直接引用旧 enum 的 helper 以保证 tsc 通过**：
  - `src/lib/work-status.ts`：完整重写，只维护 9 状态元数据
  - `src/lib/server-permissions.ts`：`APPROVAL_STATUSES` 改为新审批态列表；常量引用层面适配（深层权限逻辑可留到 PR 6.2/6.3）
  - `src/lib/dashboard-data.ts`：状态分组常量改为 9 状态（统计逻辑细节可留到 PR 6.3）
  - `src/lib/work-store.ts`：状态值常量、类型定义层面适配（筛选和流程逻辑可留到 PR 6.3）
  - `src/lib/excel-utils.ts`：导入默认状态改为 `'draft'`
  - `src/lib/status-colors.ts`：移除 `rejected` 颜色组
- [ ] 更新 `tests/target-contract/target-contract-data.cjs`：`targetStateGroups` 每状态只映射自身，移除 `targetStatusByLegacy`
- [ ] 更新 `tests/target-contract/seed-target-data.cjs`：全部改为 9 状态 seed
- [ ] 更新 `tests/target-contract/verify-target-contract.cjs`：断言不再兼容旧状态
- [ ] 更新 `tests/target-contract/target-contract-safety.cjs`：`assertResetApproved` → `assertResetConfirmed`
- [ ] 同步更新文档：`docs/数据库设计.md`、`docs/状态机设计.md`（schema 部分）

**禁止做什么：**

- [ ] 不修改 workflow 流转逻辑（`src/lib/workflow.ts`）
- [ ] 不修改前端页面和组件
- [ ] 不修改 API route
- [ ] 不重写 `work-store.ts` 的流程步骤和筛选逻辑
- [ ] 不引入 Member / DepartmentMember / WorkParticipant
- [ ] 不重新设计状态机
- [ ] 不改变现有重构路线

**验收标准：**

- [ ] `prisma generate` 通过
- [ ] `tsc --noEmit` 通过（**不允许类型错误**）
- [ ] `pnpm lint` 通过
- [ ] 项目能启动（`pnpm dev` 不报错）
- [ ] schema 中 `WorkItemStatus` 只有 9 个值、`ApprovalType` 有 4 个值
- [ ] `WorkItem` model 包含 `beforeApprovalStatus` 和 `approvalType` 字段
- [ ] 状态元数据只维护 9 状态，无旧状态引用
- [ ] target-contract seed 成功
- [ ] target-contract verify 允许 workflow 相关 expected failure（workflow 尚未改造），但不允许 schema/seed 层面的 failure

**可能出现的 expected failure：**

- [ ] `tests/regression-baseline.cjs` 旧状态断言失败（workflow 逻辑未改，旧流转路径走到旧状态会失败）

- [ ] 如 Excel 导入依赖 workflow submit 函数，可能因 `workflow.ts` 未改而行为异常
- [ ] 以上 expected failure 均属 PR 6.2 范围，PR 6.1 合并时需在 PR 描述中标记

### PR 6.2：workflow 新状态流转 + 审批权限同步

> **评审修正**：V1 中 PR 6.2 禁止修改 `server-permissions.ts`，这不合理——workflow 的 approve/reject 依赖 `canApproveWorkItem`，后者必须认识新审批态和 `approvalType`。
> V2 改为：workflow 重写的同时，**同步完成 `server-permissions.ts` 中审批权限的最小适配**。Dashboard 和页面筛选仍留到 PR 6.3。

**要做什么：**

- [ ] 重写 `src/lib/workflow.ts` 全部流转函数：

## 四、高风险点清单

### 1. `beforeApprovalStatus` 回退路径

**来源**：实施方案 三 + 四.reject

9 状态中审批退回统一通过 `beforeApprovalStatus` 回到业务状态，共 5 种路径：

| 审批态 | 退回目标 | `beforeApprovalStatus` 依赖 |
|--------|----------|---------------------------|
| `PROPOSING`（来自 `DRAFT`） | `DRAFT` | 正确为 `DRAFT` |
| `PROPOSING`（来自 `PENDING_DECOMPOSE`） | `PENDING_DECOMPOSE` | 正确为 `PENDING_DECOMPOSE` |
| `ADJUSTING` | `IN_PROGRESS` | 正确为 `IN_PROGRESS` |
| `CANCELLING` | `IN_PROGRESS` | 正确为 `IN_PROGRESS` |
| `COMPLETING` | `IN_PROGRESS` | 正确为 `IN_PROGRESS` |

**风险**：如果 `beforeApprovalStatus` 写入错误或清空逻辑有遗漏，退回进入错误状态。必须在 target-contract 中覆盖全部 5 种路径。

### 2. `PENDING_DECOMPOSE → PROPOSING` 被退回后回 `PENDING_DECOMPOSE`

**来源**：实施方案 三 + 四.decompose + 六.3

公司领导待办 → 部门分解 → `PROPOSING` → 退回 → 必须回 `PENDING_DECOMPOSE` 而非 `DRAFT`。这是区别于普通事项的关键路径。

**验证**：target-contract 须覆盖：公司领导待办分解 → 提交审批 → 退回 → 确认回到 `PENDING_DECOMPOSE` → 再分解 → 再提交。

### 3. 普通 `DRAFT → PROPOSING` 被退回后回 `DRAFT`

**来源**：实施方案 三 + 四.submit

部门主管/领导创建重点/主要工作，提交审批后退回必须回 `DRAFT`。不能因 `beforeApprovalStatus` 被误写而回到其他状态。

### 4. `REJECTED → DRAFT` 后如何展示"退回待修改"

**来源**：实施方案 二 + 八.3 | **已确认**：需要显示"退回待修改"标签

`REJECTED` 取消后，退回事项变为 `DRAFT`。页面需区分：
- 普通草稿（从未提交过审批）
- 退回待修改（有 `rejectReason` / `rejectedFromStatus` / workflow record 最新记录为 reject）

实现建议：
```
status === DRAFT && (rejectReason || rejectedFromStatus || 最新 workflow record action === 'reject')
  → 显示"退回待修改"标签
否则 → 显示"草稿"
```

依赖字段：`firstSubmitterId`、`rejectReason`、`rejectedFromStatus`、workflow record。

### 5. `APPROVED → IN_PROGRESS` 后首页和状态页口径

**来源**：实施方案 三.说明1 + 五.页面筛选风险1

`APPROVED` 取消后，`/status/inProgress` 不再包含旧 `APPROVED`。首页 `inProgressCount` 只看 `IN_PROGRESS`。旧 `APPROVED` 数据在方案 A 下直接清库重建，不存在迁移问题。

### 6. `PENDING_COMPLETE / PENDING_EVIDENCE_* → COMPLETING`

**来源**：实施方案 三.说明3 + 五.页面筛选风险5

三种旧完成审批状态统一为 `COMPLETING`。待办完成申请、重点/主要工作材料审批都显示 `COMPLETING`。具体节点通过 `currentApproverId` / `currentApproverRole` / `approvalType = COMPLETE` 区分。

### 7. `PENDING_MAIN_LEADER_CANCEL → CANCELLING`

**来源**：实施方案 三.说明4

重点工作取消中主要领导审批不再使用单独状态。通过当前审批人是否为 `PRESIDENT` 判断是否处于主要领导审批节点。

### 8. `CANCELLING` 中主要领导审批节点不再用单独状态

**来源**：实施方案 三.说明4 + 四.cancel

重点工作取消审批链路中，主要领导审批节点在 `CANCELLING` 下仅切换 `currentApproverId` / `currentApproverRole`，不创建新状态。

### 9. `workflow_records` 旧状态迁移（方案 B 场景）

**来源**：实施方案 二.方案B.SQL

`workflow_records.statusBefore` / `statusAfter` 在方案 B 下需按映射表迁移。**方案 A 清库重建不涉及此问题**，新 record 直接使用 9 状态值。

### 10. target-contract 从旧状态 fixture 切换到 9 状态 fixture

**来源**：实施方案 六.1

`targetStateGroups` 从兼容映射（如 `IN_PROGRESS: ['APPROVED', 'IN_PROGRESS']`）切换为单状态映射（`IN_PROGRESS: ['IN_PROGRESS']`）。所有 seed 数据从旧状态值改为 9 状态值。verify 断言不再兼容旧状态。
