## 二、数据库迁移方案

### 方案 A：清库重建

适用场景：

1. 系统尚未正式上线。
2. 当前数据主要是开发、演示、target-contract 或可重建测试数据。
3. 可以接受删除旧测试数据并重新 seed。

实施步骤：

1. 备份当前本地或测试库，保留必要截图或导出文件用于对比。
2. 清空旧测试数据。
3. 修改 `prisma/schema.prisma`：
   - `WorkItemStatus` 改为 9 状态。
   - 新增 `ApprovalType`。
   - `WorkItem` 新增 `beforeApprovalStatus` 和 `approvalType`。
4. 重新生成并执行 migration。
5. 更新 `tests/target-contract/seed-target-data.cjs`，直接创建目标 9 状态数据。
6. 重新运行 target-contract，目标为全部 `PASS`。
7. 如测试环境可清库，重新部署后重新 seed 测试数据。

优点：

1. 实施简单，不需要处理 PostgreSQL enum 删除旧值的复杂回填。
2. 测试数据天然符合目标状态机。
3. 避免旧审批态无法可靠还原 `beforeApprovalStatus` 的问题。

缺点：

1. 不适用于已有生产数据。
2. 无法验证真实旧数据迁移质量。

### 方案 B：保留数据迁移

适用场景：

1. 已有真实数据需要保留。
2. 审批记录、退回原因、取消原因、附件等业务痕迹需要继续可查。

#### SQL / backfill 思路

PostgreSQL enum 删除旧值通常不能直接 `DROP VALUE`。建议使用“新 enum 类型替换旧 enum 类型”的迁移方式：

1. 新增 `ApprovalType` enum。
2. 临时增加 `beforeApprovalStatus`、`approvalType` 字段。
3. 用旧状态先回填新字段。
4. 创建新的 `WorkItemStatus` enum 类型，只包含 9 个目标状态。
5. 将 `work_items.status` 通过 `CASE` 映射为新 enum。
6. 同步迁移 `workflow_records.statusBefore`、`workflow_records.statusAfter`。
7. 删除旧 enum 类型，重命名新 enum 类型为原类型名。
8. 重新生成 Prisma Client。

伪 SQL：

```sql
-- 1. 新增审批类型，具体类型名以 Prisma 生成的数据库 enum 名称为准。
CREATE TYPE "ApprovalType" AS ENUM ('propose', 'adjust', 'cancel', 'complete');

ALTER TABLE "work_items"
  ADD COLUMN "beforeApprovalStatus" "WorkItemStatus",
  ADD COLUMN "approvalType" "ApprovalType";

-- 2. 回填 approvalType。
UPDATE "work_items"
SET "approvalType" = CASE
  WHEN "status" IN ('pending_dept', 'pending_company') THEN 'propose'::"ApprovalType"
  WHEN "status" IN ('pending_complete', 'pending_evidence_dept', 'pending_evidence_company') THEN 'complete'::"ApprovalType"
  WHEN "status" = 'adjusting' THEN 'adjust'::"ApprovalType"
  WHEN "status" IN ('cancelling', 'pending_main_leader_cancel') THEN 'cancel'::"ApprovalType"
  ELSE NULL
END;

-- 3. 回填 beforeApprovalStatus。以下是可推断值；无法可靠推断的见后文。
UPDATE "work_items"
SET "beforeApprovalStatus" = CASE
  WHEN "status" IN ('pending_dept', 'pending_company') THEN 'draft'::"WorkItemStatus"
  WHEN "status" IN ('pending_complete', 'pending_evidence_dept', 'pending_evidence_company') THEN 'in_progress'::"WorkItemStatus"
  WHEN "status" = 'adjusting' THEN 'in_progress'::"WorkItemStatus"
  WHEN "status" IN ('cancelling', 'pending_main_leader_cancel') THEN 'in_progress'::"WorkItemStatus"
  ELSE NULL
END;
```

状态 enum 替换伪 SQL：

```sql
CREATE TYPE "WorkItemStatus_new" AS ENUM (
  'draft',
  'pending_decompose',
  'proposing',
  'in_progress',
  'adjusting',
  'cancelling',
  'completing',
  'completed',
  'cancelled'
);

ALTER TABLE "work_items"
  ALTER COLUMN "status" TYPE "WorkItemStatus_new"
  USING (
    CASE "status"::text
      WHEN 'draft' THEN 'draft'
      WHEN 'pending_dept' THEN 'proposing'
      WHEN 'pending_company' THEN 'proposing'
      WHEN 'approved' THEN 'in_progress'
      WHEN 'in_progress' THEN 'in_progress'
      WHEN 'pending_decompose' THEN 'pending_decompose'
      WHEN 'pending_complete' THEN 'completing'
      WHEN 'pending_evidence_dept' THEN 'completing'
      WHEN 'pending_evidence_company' THEN 'completing'
      WHEN 'pending_main_leader_cancel' THEN 'cancelling'
      WHEN 'completed' THEN 'completed'
      WHEN 'rejected' THEN 'draft'
      WHEN 'adjusting' THEN 'adjusting'
      WHEN 'cancelling' THEN 'cancelling'
      WHEN 'cancelled' THEN 'cancelled'
    END
  )::"WorkItemStatus_new";
```

`workflow_records.statusBefore` / `statusAfter` 也要按同一映射迁移。审批记录本身保留，旧状态名称可从迁移前备份或迁移说明中解释。

#### REJECTED -> DRAFT

目标架构取消 `REJECTED`。旧 `REJECTED` 数据迁移为 `DRAFT`。

保留方式：

1. `rejectReason` 字段继续保留退回原因。
2. `rejectedFromStatus` 继续保留历史退回前状态，如当前字段有值。
3. `rejectedAt`、`rejectedBy` 如当前 schema 没有独立字段，则继续依赖 workflow record 留痕。
4. `firstSubmitterId` 保留，用于退回后谁有权处理。

注意：迁移后页面不能再用 `status = REJECTED` 判断“待修改”，应改为 `status = DRAFT` 且存在退回痕迹或 workflow 最新记录为 reject。

#### PROPOSING backfill

`PENDING_DEPT` / `PENDING_COMPANY` 迁移为 `PROPOSING`，`approvalType = PROPOSE`。

`beforeApprovalStatus` 回填建议：

1. 如果能根据 workflow record 明确该事项是从 `PENDING_DECOMPOSE` 进入审批，则填 `PENDING_DECOMPOSE`。
2. 否则默认填 `DRAFT`。

无法可靠 backfill 的情况：

1. 旧数据如果缺少完整 workflow record，无法区分 `DRAFT -> PENDING_DEPT` 和 `PENDING_DECOMPOSE -> PENDING_DEPT`。
2. 待办事项从公司领导发起、分解后进入审批链路的历史数据，若没有分解记录，只能靠 `type = TODO`、`creatorRole`、`createdAt` 等弱推断，不建议自动强推。

#### COMPLETING backfill

`PENDING_EVIDENCE_DEPT` / `PENDING_EVIDENCE_COMPANY` / `PENDING_COMPLETE` 迁移为 `COMPLETING`，`approvalType = COMPLETE`。

`beforeApprovalStatus` 建议填 `IN_PROGRESS`。旧重点/主要工作从 `APPROVED` 提交见证材料，迁移后 `APPROVED` 已归并为 `IN_PROGRESS`，因此退回也应回 `IN_PROGRESS`。

#### CANCELLING backfill

`CANCELLING` / `PENDING_MAIN_LEADER_CANCEL` 迁移为 `CANCELLING`，`approvalType = CANCEL`。

1. 当前 `currentApproverId` / `currentApproverRole` 保留。
2. `PENDING_MAIN_LEADER_CANCEL` 的当前审批人通常应是 `PRESIDENT` 或主要领导用户；如已有 `currentApproverId`，优先保留。
3. `beforeApprovalStatus` 建议填 `IN_PROGRESS`。旧重点/主要工作可能处于 `APPROVED` 后申请取消，迁移后也归并到 `IN_PROGRESS`。

#### ADJUSTING backfill

`ADJUSTING` 保持 `ADJUSTING`，`approvalType = ADJUST`。

`beforeApprovalStatus` 建议填：

1. 待办事项：`IN_PROGRESS`。
2. 重点/主要工作：旧系统可能从 `APPROVED` 或 `IN_PROGRESS` 进入，迁移后统一填 `IN_PROGRESS`。

#### 无法可靠 backfill 的字段

以下内容无法完全依赖当前状态自动还原：

1. `PROPOSING.beforeApprovalStatus` 到底是 `DRAFT` 还是 `PENDING_DECOMPOSE`。
2. 多节点审批中已经经过哪些节点，只能保留当前 `currentApproverId` / `currentApproverRole`，不能从状态本身还原完整审批路径。
3. `REJECTED` 迁移为 `DRAFT` 后，页面如何区分普通草稿和退回待修改，需要依赖 `rejectReason`、`rejectedFromStatus`、workflow record 或后续新增派生查询。
4. 旧 `APPROVED` 是“已立项待提交材料”还是被某些页面当作进行中展示，迁移后统一为 `IN_PROGRESS`。
5. 旧 `PENDING_EVIDENCE_DEPT` / `PENDING_EVIDENCE_COMPANY` 的前置节点虽然可推断为完成审批，但无法从单一状态还原所有审批链路历史。

## 三、状态流转目标方案

目标 `WorkItemStatus`：

1. `DRAFT`
2. `PENDING_DECOMPOSE`
3. `PROPOSING`
4. `IN_PROGRESS`
5. `ADJUSTING`
6. `CANCELLING`
7. `COMPLETING`
8. `COMPLETED`
9. `CANCELLED`

目标 `ApprovalType`：

1. `PROPOSE`
2. `ADJUST`
3. `CANCEL`
4. `COMPLETE`

目标流转：

```text
DRAFT -> PROPOSING -> IN_PROGRESS
PENDING_DECOMPOSE -> PROPOSING -> IN_PROGRESS
PROPOSING reject -> beforeApprovalStatus
IN_PROGRESS -> ADJUSTING -> IN_PROGRESS
IN_PROGRESS -> CANCELLING -> CANCELLED
IN_PROGRESS -> COMPLETING -> COMPLETED
COMPLETING reject -> IN_PROGRESS
DRAFT -> CANCELLED
```

说明：

1. `APPROVED` 取消后，进行中只看 `IN_PROGRESS`。
2. `REJECTED` 取消后，退回事项恢复 `DRAFT`。
3. 完成材料审批统一为 `COMPLETING`。
4. 取消主要领导审批不再用 `PENDING_MAIN_LEADER_CANCEL`，在 `CANCELLING` 下通过 `currentApproverId` / `currentApproverRole` / `approvalType` 表达当前节点。
5. 所有审批态都必须有 `approvalType`。
6. 所有可退回审批态都必须有 `beforeApprovalStatus`。

## 四、workflow 改造方案

### 统一规则

进入审批态时：

1. `beforeApprovalStatus = 当前业务状态`。
2. `approvalType = 当前审批类型`。
3. 设置 `currentApproverId` 或 `currentApproverRole`。
4. 写 workflow record。

审批通过但还有下一节点时：

1. `status` 保持当前审批态。
2. `beforeApprovalStatus` 保持不变。
3. `approvalType` 保持不变。
4. 只切换 `currentApproverId` / `currentApproverRole`。
5. 写 workflow record。

审批结束时：

1. `status` 进入目标状态：
   - `PROPOSE` -> `IN_PROGRESS`
   - `ADJUST` -> `IN_PROGRESS`
   - `CANCEL` -> `CANCELLED`
   - `COMPLETE` -> `COMPLETED`
2. 清空 `beforeApprovalStatus`。
3. 清空 `approvalType`。
4. 清空 `currentApproverId` / `currentApproverRole`。
5. 写 workflow record。

审批退回时：

1. `status = beforeApprovalStatus`。
2. 清空 `beforeApprovalStatus`。
3. 清空 `approvalType`。
4. 清空 `currentApproverId` / `currentApproverRole`。
5. 保留 `rejectReason` / `rejectedFromStatus` / workflow record。

### submit

适用：

1. `DRAFT` 提交立项审批。
2. 被退回后恢复为 `DRAFT` 的事项重新提交。

目标逻辑：

1. 校验当前用户有提交权限。
2. 根据事项类型和创建人角色决定首个审批节点：
   - 部门主管创建 -> 部门领导。
   - 部门领导创建 -> 公司领导。
   - 公司领导创建待办 -> 可直接进入 `PENDING_DECOMPOSE` 或进入 `PROPOSING`，按现有业务场景区分。
3. 如进入审批链路：
   - `status = PROPOSING`
   - `beforeApprovalStatus = DRAFT`
   - `approvalType = PROPOSE`
   - 设置当前审批人。
4. 首次进入正式审批链路时写入 `firstSubmitterId`。

### decompose

适用：公司领导发起待办，状态为 `PENDING_DECOMPOSE`。

目标逻辑：

1. 部门用户完成分解后，进入立项审批。
2. `status = PROPOSING`
3. `beforeApprovalStatus = PENDING_DECOMPOSE`
4. `approvalType = PROPOSE`
5. 设置当前审批人。
6. 如果 `PROPOSING` 被退回，应回到 `PENDING_DECOMPOSE`，而不是普通 `DRAFT`。

### approve

适用：所有审批态。

目标逻辑：

1. 先用 `canApproveWorkItem` 判断当前用户是否为当前节点审批人。
2. 根据 `approvalType` 和当前节点计算是否还有下一节点。
3. 还有下一节点：
   - `status` 不变。
   - 切换 `currentApproverId` / `currentApproverRole`。
4. 无下一节点：
   - 按 `approvalType` 进入最终目标状态。
   - 清空审批辅助字段和当前审批人。

### reject

适用：所有审批态。

目标逻辑：

1. 校验当前用户可审批。
2. `status = beforeApprovalStatus`。
3. 清空审批辅助字段和当前审批人。
4. 写入退回原因。
5. `PROPOSING` 从 `DRAFT` 来则回 `DRAFT`；从 `PENDING_DECOMPOSE` 来则回 `PENDING_DECOMPOSE`。
6. `COMPLETING` 退回回 `IN_PROGRESS`。
7. `ADJUSTING` / `CANCELLING` 退回回 `IN_PROGRESS`。

### adjust

适用：`IN_PROGRESS`。

目标逻辑：

1. 校验当前用户可办理或可申请调整。
2. 记录调整原因和调整内容。
3. `status = ADJUSTING`
4. `beforeApprovalStatus = IN_PROGRESS`
5. `approvalType = ADJUST`
6. 设置当前审批人。

### cancel

适用：

1. `DRAFT` 可直接取消。
2. `IN_PROGRESS` 需要进入取消审批。

目标逻辑：

1. `DRAFT -> CANCELLED`：不进入审批态，不写 `approvalType`。
2. `IN_PROGRESS -> CANCELLING`：
   - `beforeApprovalStatus = IN_PROGRESS`
   - `approvalType = CANCEL`
   - 设置当前审批人。
3. 重点工作如需主要领导审批，仍保持 `CANCELLING`，只切换当前审批人到主要领导。

### complete

适用：`IN_PROGRESS`。

目标逻辑：

1. 提交完成材料或完成申请。
2. `status = COMPLETING`
3. `beforeApprovalStatus = IN_PROGRESS`
4. `approvalType = COMPLETE`
5. 设置当前审批人。
6. 审批通过后进入 `COMPLETED`。
7. 退回后回 `IN_PROGRESS`。

## 五、兼容风险

### helper 风险

需要同步更新：

1. `src/lib/work-status.ts`
2. `src/lib/work-store.ts`
3. `src/lib/server-permissions.ts`
4. `src/lib/dashboard-data.ts`
5. 所有状态 Badge 和筛选 helper。

特别注意：

1. `APPROVED` 取消后，进行中只看 `IN_PROGRESS`。
2. `REJECTED` 取消后，退回事项回 `DRAFT`。
3. 完成材料审批统一为 `COMPLETING`。
4. 取消主要领导审批不再用 `PENDING_MAIN_LEADER_CANCEL`，而是在 `CANCELLING` 下根据当前审批人判断。

### target-contract 风险

1. 当前 target-contract 中的旧状态兼容映射需要移除或改成 migration 专用断言。
2. seed 数据必须直接使用 9 状态。
3. dashboard、Excel、权限、完成率断言都要改为目标状态。
4. workflow 路径断言将成为第六阶段主要风险点。

### 页面筛选风险

1. `/status/inProgress` 不再包含 `APPROVED`。
2. `/status/approving` 应覆盖 `PROPOSING`、`ADJUSTING`、`CANCELLING`、`COMPLETING`。
3. `/status/handling` 不再依赖 `REJECTED`。
4. 退回待修改应通过 `DRAFT` + 退回痕迹展示，而不是 `REJECTED` 状态。
5. 完成审批列表统一展示 `COMPLETING`。

### 数据风险

1. 如果采用方案 B，`beforeApprovalStatus` 部分场景无法可靠回填。
2. 旧 workflow record 中的状态值迁移后会失去旧状态字面值；如需审计旧状态原文，应先备份或新增迁移备注。
3. `REJECTED -> DRAFT` 后，普通草稿和退回草稿区分依赖退回字段或 workflow record。
