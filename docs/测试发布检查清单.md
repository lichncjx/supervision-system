# 测试发布检查清单

- [ ] CI 是否通过
- [ ] Docker 镜像是否构建成功
- [ ] 是否涉及数据库迁移

## 部署检查

- [ ] 是否执行 migrate.sh
- [ ] 是否执行 deploy.sh
- [ ] docker-compose ps 状态是否正常
- [ ] supervision-app 是否 running
- [ ] supervision-db 是否 running

## 功能验证

- [ ] 页面是否能打开
- [ ] /api/health 是否正常
- [ ] 核心账号是否能登录
- [ ] 本次修改涉及页面是否验证
- [ ] 不同角色权限是否验证
- [ ] 数据库迁移后是否执行数据回填 SQL

## 审批退回流程回归测试

涉及 `firstSubmitterId` 变更时，使用不同角色账号验证以下场景：

- [ ] 部门主管首次提交事项 → 部门领导退回 → 仅该部门主管能在待我处理中看到并处理
- [ ] 部门领导首次提交事项 → 公司领导退回 → 仅该部门领导能处理
- [ ] 公司领导创建待办 → 提交到待分解（不设置 firstSubmitterId）→ 部门主管分解并送审 → firstSubmitterId = 部门主管
- [ ] 公司领导创建的待办被退回后，公司领导的待我处理列表中不出现该退回事项
- [ ] 同部门其他部门主管/部门领导不能看到该退回事项的退回处理面板
- [ ] 退回后重新提交：生成 workflow record，重新进入正式审批链路
- [ ] Dashboard 待办理统计与待我处理列表数量一致
- [ ] 历史退回事项（firstSubmitterId 为空）：仍能处理，不卡死
