# 公司督办管理系统 - 项目文档

## 项目概览

公司督办管理系统是一个用于跟踪和管理公司重点工作、主要工作、待办事项的企业级应用。

### 版本技术栈

- **Framework**: Next.js 15 (App Router)
- **Core**: React 19
- **Language**: TypeScript 5
- **UI 组件**: shadcn/ui (基于 Radix UI)
- **Styling**: Tailwind CSS 4
- **数据库**: PostgreSQL (使用 Prisma ORM)

## 功能模块

### 1. 登录与认证
- 用户名 + 密码登录
- 会话管理（24小时有效）
- 登录信息存储在Cookie中

### 2. 首页 (Dashboard)
- 统计卡片：总事项、待审批、进行中、已完成、超期
- 分类统计：重点工作、主要工作、待办事项数量
- 最近事项列表
- 待审批提示

### 3. 事项管理
- **重点工作**：红色标识，公司级重要事项
- **主要工作**：蓝色标识，部门主要工作
- **待办事项**：绿色标识，日常任务
- 列表页：搜索、状态筛选、部门筛选、月份筛选
- 详情页：基本信息、节点进度、见证材料、审批记录
- 支持 Excel 导入导出

### 4. 审批中心
- 待审批列表
- 审批操作：同意/退回
- 审批历史

### 5. 系统管理
- 部门管理（固定16个部门）
- 用户管理
- 角色权限说明

## 权限控制

### 用户角色

| 角色 | 标识 | 说明 |
|------|------|------|
| 督办管理员 | ADMIN | 查看全部、录入/导入 |
| 督办人员 | SUPERVISOR | 督办跟踪 |
| 部门主管 | DEPARTMENT_MANAGER | 查看本部门、发起的申请 |
| 部门领导 | DEPARTMENT_LEADER | 查看本部门、审批（部门级） |
| 公司主管领导 | VICE_PRESIDENT | 查看全部、所有审批权限 |
| 公司主要领导 | PRESIDENT | 查看全部、仅重点工作取消审批 |

### 权限规则

1. **数据访问权限**
   - 督办管理员、公司领导：可查看全部数据
   - 部门主管、部门领导：仅可查看本部门数据

2. **审批权限**
   - 部门领导：可审批本部门提交的申请
   - 公司主管领导（副总）：可审批所有申请
   - 公司主要领导（一把手）：仅审批重点工作取消

3. **操作权限**
   - 发起申请：部门主管、部门领导、公司领导
   - 完成/调整/取消：事项责任人或部门领导

## 工作流操作

| action | 说明 |
|--------|------|
| submit | 提交审批 |
| evidence | 提交见证材料 |
| approve | 审批通过 |
| reject | 审批退回 |
| adjust | 申请调整 |
| cancel | 申请取消 |
| decompose | 分解待办 |

### 工作流程说明

**重点工作/主要工作流程**:
1. 部门提交 → 待部门领导审批
2. 部门领导审批 → 待公司领导审批
3. 公司领导审批通过 → 已立项/进行中
4. 提交完成材料 → 待见证审批
5. 见证审批通过 → 已完成

**待办事项流程**:
1. 部门发起并分解 → 待分解
2. 部门领导审批分解 → 待提出领导审批
3. 提出领导审批 → 进行中
4. 提交完成材料 → 待完成审批
5. 完成审批通过 → 已完成

**调整/取消流程**:
- 进行中状态可申请调整或取消
- 调整/取消需经过相应的审批流程
- 重点工作取消需公司主要领导审批

## 群晖部署指南

### 部署架构

```
GitHub Actions (CI/CD)
       ↓
   Docker Registry
       ↓
   群晖服务器
       ↓
docker-compose.yml + .env
       ↓
  应用容器 + 数据库容器
```

### 前提条件

1. 群晖已安装 Docker 套件
2. 已创建共享文件夹用于存储数据（如 `/volume1/docker`）
3. GitHub 仓库已配置好 GitHub Actions 和 Container Registry

### 部署步骤

#### 1. 在群晖上创建部署目录

在群晖的 SSH 终端中执行：

```bash
# 创建项目目录
mkdir -p /volume1/docker/supervision-system
cd /volume1/docker/supervision-system

# 创建数据存储目录
mkdir -p ../postgres_data
mkdir -p ../uploads
```

#### 2. 上传配置文件

将以下文件上传到 `/volume1/docker/supervision-system/` 目录：

- `docker-compose-synology.yml` → 重命名为 `docker-compose.yml`
- `.env.synology.example` → 重命名为 `.env`，并修改其中的配置项

```bash
# 上传后重命名
cd /volume1/docker/supervision-system
mv docker-compose-synology.yml docker-compose.yml
cp .env.synology.example .env
```

#### 3. 修改 .env 配置

编辑 `.env` 文件，修改以下配置项：

```bash
# 使用 vi 或其他编辑器
vi .env

# 修改以下配置：
POSTGRES_PASSWORD=你的强密码
JWT_SECRET=你的随机密钥
NEXT_PUBLIC_APP_URL=http://你的群晖IP:18080
```

#### 4. 创建 deploy.sh 部署脚本

在 `/volume1/docker/supervision-system/` 目录下创建 `deploy.sh` 文件：

```bash
#!/bin/sh
set -e

cd /volume1/docker/supervision-system

echo "Pulling latest image..."
docker compose pull

echo "Restarting containers..."
docker compose up -d

echo "Cleaning unused images..."
docker image prune -f

echo "Deploy completed."
```

#### 5. 设置脚本执行权限

首次创建后执行：

```bash
chmod +x /volume1/docker/supervision-system/deploy.sh
```

### 手动部署

每次 GitHub Actions 构建成功后，在群晖终端执行：

```bash
sh /volume1/docker/supervision-system/deploy.sh
```

### 群晖任务计划配置（可选）

可以通过群晖的"控制面板 → 任务计划"来创建手动运行任务：

1. **控制面板 → 任务计划 → 新增 → 用户定义的脚本**
2. **任务名称**：supervision-system-deploy
3. **用户**：root
4. **事件**：手动
5. **脚本内容**：

```bash
sh /volume1/docker/supervision-system/deploy.sh
```

### 部署时机建议

> **重要提示**：当前阶段不建议按时间定时自动部署，建议采用以下流程：

1. 开发人员提交代码到 GitHub
2. GitHub Actions 自动构建 Docker 镜像
3. 等待构建成功后，在群晖上手动运行部署脚本
4. 验证部署结果

这样可以确保每次部署都是可控的，避免在构建失败时影响生产环境。
