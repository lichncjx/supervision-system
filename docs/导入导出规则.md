# 导入导出规则

本文档定义导入导出的目标业务口径。导入导出应服务于展示、责任留痕和批量维护，不应绕过系统权限和 workflow。

## 一、基本原则

1. 导入导出使用统一组织权限。
2. 导入不能绕过 workflow 直接创建审批中、已完成或已取消事项，除非是明确的历史迁移任务。
3. 导出数据范围必须与列表查询范围一致。
4. 业务责任人姓名字段只用于展示、导入导出和历史留痕。
5. 业务责任人姓名字段不参与审批权限、办理权限、数据可见权限判断。

第五阶段当前实现：

1. 普通导出接口为 `GET /api/excel/export`，数据范围与 `GET /api/works` 可见范围一致。
2. 普通导入接口为 `POST /api/excel/import/[type]`，默认创建 `DRAFT`，状态流转仍必须通过 workflow。
3. completionRate 导出已在前面阶段按主责部门口径处理，本阶段重点是普通事项导入导出。
4. `src/lib/excel-utils.ts` 仍被模板下载和部分前端工具函数引用，暂作为 legacy 工具保留；普通导出以服务端接口为准，后续可单独清理未使用的客户端导入导出函数。

## 二、重点工作和主要工作字段

重点工作、主要工作中需要填写：

| 字段 | 含义 | 权限作用 |
| --- | --- | --- |
| `responsibleLeader` | 事项业务责任领导姓名文本 | 无 |
| `responsiblePerson` | 事项实际责任人姓名文本 | 无 |

说明：

1. `responsibleLeader` 不等同于系统中的部门领导账号。
2. `responsiblePerson` 不等同于系统中的部门事项管理岗。
3. 二者不是 `User` 外键。
4. 二者不是 `Member` 外键。
5. 二者可以是不登录系统的业务人员。
6. 导入时不强制匹配 `User` 或 `Member`。
7. 导出时按事项记录中的姓名输出。

## 三、待办事项字段

待办事项继续保留姓名数组逻辑：

| 字段 | 含义 | 权限作用 |
| --- | --- | --- |
| `responsibleDepartmentIds` | 目标字段，主责部门 ID 数组 | 用于组织可见、办理、统计 |
| `departmentId` | 单主责部门 / 历史兼容字段 | `responsibleDepartmentIds` 为空时回退 |
| `responsiblePersons` | 主责责任人姓名文本或 JSON 数组 | 无 |
| `cooperateDepartmentIds` | 配合部门 ID 数组 | 用于可见，不默认用于办理和完成率 |
| `cooperatePersons` | 配合责任人姓名文本或 JSON 数组 | 无 |

本轮不新增：

1. `Member` 表。
2. `DepartmentMember` 表。
3. `WorkParticipant` 表。

主责部门兼容规则：

1. 如果本轮做数据库迁移，则将当前 `departmentIds` 迁移/重命名为 `responsibleDepartmentIds`。
2. 如果本轮暂不做数据库迁移，则通过统一 helper，例如 `getResponsibleDepartmentIds(work)`，按 `responsibleDepartmentIds` -> `departmentIds` -> `departmentId` 顺序读取。
3. 权限、统计、导入导出中不要各自手写部门字段兼容逻辑，应通过统一 helper 或统一权限模块读取。

## 四、状态导入导出

### 普通导入

1. 普通业务导入默认创建 `DRAFT`。
2. 导入后由系统用户通过 workflow 提交。
3. 导入不直接写入 `currentApproverId` / `currentApproverRole`。
4. 导入不直接写入 `beforeApprovalStatus` / `approvalType`。

### 历史迁移导入

历史迁移可使用旧状态到新状态映射：

| 旧状态 | 新状态 |
| --- | --- |
| `DRAFT` | `DRAFT` |
| `PENDING_DEPT` | `PROPOSING` |
| `PENDING_COMPANY` | `PROPOSING` |
| `APPROVED` | `IN_PROGRESS` |
| `IN_PROGRESS` | `IN_PROGRESS` |
| `PENDING_DECOMPOSE` | `PENDING_DECOMPOSE` |
| `PENDING_COMPLETE` | `COMPLETING` |
| `PENDING_EVIDENCE_DEPT` | `COMPLETING` |
| `PENDING_EVIDENCE_COMPANY` | `COMPLETING` |
| `PENDING_MAIN_LEADER_CANCEL` | `CANCELLING` |
| `COMPLETED` | `COMPLETED` |
| `REJECTED` | `DRAFT` |
| `ADJUSTING` | `ADJUSTING` |
| `CANCELLING` | `CANCELLING` |
| `CANCELLED` | `CANCELLED` |

## 五、导出展示

1. 状态文案使用统一状态元数据 `getWorkStatusLabel`。
2. 重点/主要工作导出 `responsibleLeader` 和 `responsiblePerson` 目标口径；当前 schema 尚未落地独立 `responsiblePerson` 时，兼容使用历史 `supervisor` / `deptManagerName` 作为展示 fallback。
3. 待办事项导出主责部门、主责责任人、配合部门、配合责任人。
4. 主责部门通过 `getResponsibleDepartmentIds(work)` 读取，兼容 `departmentIds` 和 `departmentId`。
5. 配合部门通过 `getCooperateDepartmentIds(work)` 读取。
6. 导出不暴露密码、token、内部审计字段、审批记录、附件、证明材料、节点 JSON 等详情页字段。
7. 如需导出审批记录，应作为单独明细表或单独功能，不混入普通事项清单。

## 六、导入权限

1. `ADMIN` / `SUPERVISOR` 可按全局范围导入。
2. `DEPARTMENT_LEADER` / `DEPARTMENT_MANAGER` 只能导入主责部门包含本部门的事项；配合部门可以填写其他部门，但不能替代主责部门授权。
3. `VICE_PRESIDENT` / `PRESIDENT` 不默认获得全局导入权限；普通导入仅允许导入本人提出或本人审批的待办事项。
4. 责任领导、责任人、主责责任人、配合责任人只作为姓名文本导入，不校验系统用户，不产生权限。

## 七、导入字段兼容

1. 主责部门多选目标字段为 `responsibleDepartmentIds`。
2. 当前 schema 尚未提供 `responsibleDepartmentIds`，普通导入仍写入 `departmentIds`，并同步保留 `departmentId` 作为单主责 / 历史兼容字段。
3. 重点/主要工作通常只有一个主责部门，导入时写入 `departmentId`，并同步写入 `departmentIds: [departmentId]` 便于兼容 helper 读取。
4. 待办事项支持多个主责部门、多个配合部门、多个主责责任人、多个配合责任人。

## 八、已接受风险

1. 责任人姓名可能重名。
2. 人员调岗后历史姓名不会自动关联新部门。
3. 无法按人员 ID 精确统计个人责任事项。
4. 无法基于责任人姓名做系统通知和权限控制。
5. 如后续确有人员维度统计、通知、权限控制需求，再评估 `DepartmentMember` / `WorkParticipant`。
