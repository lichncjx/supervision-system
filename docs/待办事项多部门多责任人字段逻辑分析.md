## 一、新建待办事项逻辑

### 1.1 候选人员生成

`src\app\(app)\[type]\new\page.tsx:420-425`

```typescript
const responsiblePersonOptions = todoForm.departmentIds.flatMap(
  (departmentId: number) => departmentUsers[departmentId] || []
);
```

候选人员来源链路：

1. `useEffect` 监听 `todoForm.departmentIds` 和 `todoForm.cooperateDepartmentIds` 变化
2. 对每个未缓存的 deptId，调用 `getUsersByDepartment(deptId)` → `GET /api/users/by-department?departmentId=X`
3. 该 API 查询 `isActive: true` 的部门用户，返回 `{ id, name, role, departmentId, departmentName }`
4. 缓存到 `departmentUsers: Record<number, User[]>`
5. `responsiblePersonOptions` 是选中部门的所有用户 flatMap

**关键设计决策：选项的 value 使用 `person.name`（纯姓名），丢弃了 `person.id` 和 `person.departmentId`。**

```typescript
// new/page.tsx:430-434
const responsiblePersonSelectOptions = responsiblePersonOptions.map((person) => ({
  value: person.name,
  label: person.name,
  description: person.departmentName || '-',
}));
```

### 1.2/1.3 删除部门后的清理逻辑及依据

`src\app\(app)\[type]\new\page.tsx:721-735`

```typescript
onChange={(nextValues) => {
  const nextDepartmentIds = nextValues.map(Number);
  const nextDepartmentUserNames = new Set(
    nextDepartmentIds.flatMap((deptId) =>
      (departmentUsers[deptId] || []).map((person) => person.name),
    ),
  );
  setTodoForm((prev) => ({
    ...prev,
    departmentIds: nextDepartmentIds,
    responsiblePersons: prev.responsiblePersons.filter((personName) =>
      nextDepartmentUserNames.has(personName),
    ),
  }));
}}
```

**清理依据：纯按姓名匹配当前部门人员池（`Set<name>`）。**

不按 userId、不按 departmentId、不记录选择来源。

**同名问题示例：**

| 操作 | 结果 |
|------|------|
| 选部门 A（张三、李四）+ 部门 B（张三、王五） | 张三在选项中出现两次（一人两选项），选中后 responsiblePersons = ["张三", "李四", "王五"] |
| 取消勾选部门 A | 部门 B 仍有"张三" → "张三"保留（实际可能是部门 A 的张三，也可能是部门 B 的） |
| 取消勾选部门 B | 部门 A 仍有"张三" → "张三"保留（同上） |

无法区分同名人员属于哪个部门。这是 name-as-value 的根本性缺陷。

### 1.4 cooperateDepartmentIds / cooperatePersons

`src\app\(app)\[type]\new\page.tsx:774-788`

完全相同的逻辑：

```typescript
cooperatePersons: prev.cooperatePersons.filter((personName) =>
  nextDepartmentUserNames.has(personName),
),
```

### 1.5 小结

| 问题 | 是否存在 | 严重程度 |
|------|:---:|:---:|
| 删除部门后人员残留 | 是 | 中等 — 同名人员会残留，非同名可正确清除 |
| 清理依据是否准确 | 部分准确 | 按姓名 + 当前部门人员池过滤，无 userId |
| 同名人员问题 | 是 | 中 — 两个部门同名人无法区分 |

## 二、编辑待办事项逻辑

### 2.1 editForm 初始化

`src\app\(app)\[type]\[id]\page.tsx:117-144`

```typescript
setEditForm({
  departmentId: work.departmentId,
  responsiblePerson: work.responsiblePerson || '',    // 拼接字符串
  cooperateDepartment: work.cooperateDepartment || '',  // 拼接字符串
  cooperatePerson: work.cooperatePerson || '',          // 拼接字符串
  // 缺失: departmentIds, responsiblePersons, cooperateDepartmentIds, cooperatePersons
});
```

**关键发现：editForm 完全没有初始化数组字段。** 编辑表单使用的是**旧式字段**（拼接字符串），而非创建时使用的**数组字段**。

### 2.2 编辑 UI 组件

三个编辑入口使用的都是**纯文本 Input**（手动输入），而非 MultiSearchSelect：

| 编辑入口 | 组件 | 主责部门 | 主责责任人 | 配合部门 | 配合责任人 |
|---------|------|:---:|:---:|:---:|:---:|
| 退回重提 | WorkReturnedPanel | 单选 select | 文本 Input | 文本 Input | 文本 Input |
| 调整 | WorkActionDialogs | 单选 select | 文本 Input | 文本 Input | 文本 Input |
| 分解 | WorkDecomposePanel | — | — | — | — |

**这意味着编辑时：**
1. **无法修改多部门选择**（只有单一 departmentId 下拉）
2. **无法通过下拉选择责任人**（只能手打姓名文本）
3. 数组字段在编辑链路中完全不可修改

### 2.3 提交流程

`src\app\(app)\[type]\[id]\page.tsx:425-433`

```typescript
const patch: WorkEditablePatch = {
  ...editForm,
  title: editForm.workItem || editForm.title || work.title,
};
await resubmitRejectedWork(work, user, patch);
```

`updateWork` 序列化逻辑（`src\lib\work-store.ts:524-526`）：

```typescript
if (patch.responsiblePersons !== undefined) data.responsiblePersons = patch.responsiblePersons;
if (patch.cooperateDepartmentIds !== undefined) data.cooperateDepartmentIds = patch.cooperateDepartmentIds;
if (patch.cooperatePersons !== undefined) data.cooperatePersons = patch.cooperatePersons;
```

**因为 editForm 中没有这些数组字段，它们永远不会被包含在 patch 中，API 也不会更新它们。编辑时数组字段被彻底冻结。**

### 2.4 历史数据问题

对于已有数据：
- `responsiblePersons` 是 `String[]`（如 `["张三", "李四"]`）
- `responsiblePerson` 是拼接字符串（如 `"张三、李四"`），创建时由前端计算
- 编辑回显时只用了 `responsiblePerson`，无法还原 `responsiblePersons` 数组——但这不影响，因为数组在编辑时只读

**系统无法判断某个姓名属于哪个 departmentId**，因为选择时没有记录来源。

### 2.5 编辑流程总结

| 字段 | 创建时 | 编辑时 |
|------|--------|--------|
| `departmentIds` | MultiSearchSelect | **不可编辑**（editForm 无此字段） |
| `responsiblePersons` | MultiSearchSelect（按姓名） | **不可编辑**（editForm 用 responsiblePerson 替代） |
| `cooperateDepartmentIds` | MultiSearchSelect | **不可编辑**（editForm 无此字段） |
| `cooperatePersons` | MultiSearchSelect（按姓名） | **不可编辑**（editForm 用 cooperatePerson 替代） |

### 3.4 Prisma Schema

```prisma

cooperateDepartmentIds Int[]    @default([])
```

## 五、风险评估

### 5.1 重名人员风险：高

- 选项 value 是姓名，两个同名人无法区分选择来源
- 删除部门时按姓名过滤，同名者会残留（不准确但偏安全）
- 编辑回显时只有拼接字符串，无法还原部门归属
- API 端无重名校验

### 5.2 人员调岗风险：中

- 新建时 `departmentUsers` 取当前部门 `isActive: true` 用户，调岗人员出现在新部门
- 历史数据姓名仍指向旧部门，但系统无法知道（这实际上是 name snapshot 的设计意图）
- 不影响权限
- **如果不引入快照机制，调岗后新建的事项会关联错误的人员（用新部门的用户池，但姓名匹配到旧数据）**

### 5.3 人员停用/删除风险：中

- `getUsersByDepartment` 过滤 `isActive: true`，停用人员不出现在新建候选中
- API 不校验 `isActive`，如果绕过前端直接调 API，可写入停用人员姓名
- 编辑回显时无法判断历史姓名是否已停用

### 5.4 删除部门后人员残留风险：中

- 新建：按姓名过滤，基本可用但有同名缺陷
- 编辑：完全无清理（departmentIds 不可编辑）
- API：无校验兜底

### 5.5 历史数据显示风险：低-中

- 姓名数组 join 展示正常
- 无法追溯该人员当时属于哪个部门
- 人员姓名变更后显示旧姓名（这是设计意图）

### 5.6 导出数据准确性风险：高

- 导出缺少 `departmentIds` 和 `responsiblePersons` 列
- 导入 `cooperateDepartmentIds` 永远为空
- 导入导出无法闭环

### 5.7 是否影响权限判断：否

- 权限使用 `cooperateDepartmentIds`（ID 数组），是有效数据
- `responsiblePersons` / `cooperatePersons` 仅展示留底，不参与 `canAccessWorkItem`、附件权限等任何权限判断
- `src\lib\server-permissions.ts` 中的 `cooperateDepartmentIds` 权限匹配不依赖姓名

## 六、方案对比

### 方案 A：不改数据库，仅前端过滤

**做法**：继续使用姓名数组，不做任何后端改动，仅在前端根据当前部门人员池过滤。

| 优点 | 缺点 |
|------|------|
| 零 migration | 同名问题无解 |
| 改动最小 | 调岗/停用无感知 |
| 立即生效 | 编辑表单不支持多部门编辑 |
| | 导入导出不对齐 |

**能否解决重名/调岗问题**：否。

**适合短期吗**：仅适合作为"承认风险、暂不处理"的过渡策略，不应视为修复方案。

---

### 方案 B：前端内部 {departmentId, name}，提交时仍存姓名数组

**做法**：前端维护 `Map<name, departmentId>` 记录选择来源，删除部门时按 departmentId 精确过滤，提交时只序列化 name 数组给 API。

| 优点 | 缺点 |
|------|------|
| 不改数据库和 API | 对历史数据无效（无来源信息） |
| 删除部门时可精确清理 | 编辑回显无法还原 departmentId |
| 比纯姓名过滤更准确 | 仍不能解决同名 key 冲突（Map key 是 name） |
| | API 仍无校验 |

**对历史数据是否有效**：否。历史 `responsiblePersons` 只有姓名，无法解析出 `departmentId`。

**适合短期吗**：比方案 A 稍好，但投入产出比不高——核心重名问题仍未解决。

---

### 方案 C：新增结构化快照字段（推荐）

**做法**：新增两个 JSON 字段，记录 "姓名 + 来源部门" 的快照。

**字段定义**：

```prisma
responsiblePersonSnapshots  Json?
cooperatePersonSnapshots    Json?
```

**数据结构**：

```json
[
  { "departmentId": 1, "departmentName": "信息档案中心", "name": "张三" },
  { "departmentId": 2, "departmentName": "质量管理处", "name": "李四" }
]
```

**各维度评估**：

| 维度 | 评估 |
|------|------|
| 解决当前问题 | 是。删除部门按 departmentId 精确清理；同名不同部门可区分 |
| 比 responsiblePersonIds 更轻量 | 是。不需要 FK、不需要用户存在性校验、不需要处理用户删除 |
| 是否需要 migration | 是。需 `ALTER TABLE` 加两个 JSON 列 |
| 对导入的影响 | 导入时需同时写入快照字段 |
| 对导出的影响 | 导出时可从快照中取值 |
| 对前端展示的影响 | 现有的 `responsiblePersons.join('、')` 仍可工作；新增基于快照的展示（如按部门分组） |
| 编辑回显 | 可从快照还原部门-人员对应关系，支持精确编辑 |
| 兼容旧字段 | `responsiblePersons` / `cooperatePersons` 保留不删，作为兼容 |
| 同名处理 | 同名但不同 departmentId 的两个快照为不同条目，可精确区分 |
| 调岗/停用 | 快照在创建时冻结，后续人员变化不影响快照（符合设计意图） |

## 七、结论与建议

### 7.1 核心问题确认

1. **新建时**：有基本的姓名级过滤，但有同名缺陷（中等）
2. **编辑时**：`departmentIds`、`responsiblePersons`、`cooperateDepartmentIds`、`cooperatePersons` 四个数组字段在编辑链路中**完全不可编辑**，这是最大的功能缺失（高）
3. **API 端**：零校验，姓名数组直接透传入库（中）
4. **Excel 导入**：`cooperateDepartmentIds` 硬编码为空数组（高，纯 bug）
5. **Excel 导出**：缺少 `departmentIds` 和 `responsiblePersons` 列（中）

### 7.2 是否确实存在删除部门后人员残留

**是。** 场景如下：
- 新建时：同名人员在剩余部门中存在 → 残留（偏安全）
- 编辑时：departmentIds 完全不能修改 → 不存在"删除部门"的操作路径，但这是更大的问题

### 7.3 是否存在重名/调岗导致无法准确清理

**是。** 根本原因：选择阶段 value 使用 `person.name` 而非 `person.id`，用户身份信息在选择时即丢失。

### 7.4 是否需要立即修

**建议分级处理：**

| 优先级 | 问题 | 理由 |
|:---:|------|------|
| P0 | Excel 导入 `cooperateDepartmentIds = []` bug | 纯 bug，数据损坏，应立即修 |
| P1 | 编辑表单不支持数组字段编辑 | 功能缺失，编辑时人员信息被冻结 |
| P2 | 新建时同名清理缺陷 | 需要方案 C |
| P2 | 导出缺少列 | 需要方案 C |

### 7.5 推荐方案

**短期（立即，不改数据库）：**

1. 修复 Excel 导入 bug：将 `cooperateDepartmentNames` 按部门名查找 ID，写入 `cooperateDepartmentIds`
2. 在编辑表单的三个入口（WorkReturnedPanel、WorkActionDialogs、WorkDecomposePanel）补充 `departmentIds` / `cooperateDepartmentIds` / `responsiblePersons` / `cooperatePersons` 的 MultiSearchSelect，复用新建页的联动过滤逻辑

**中期（下个迭代，需 migration）：**

采用方案 C，新增 `responsiblePersonSnapshots` 和 `cooperatePersonSnapshots` JSON 字段：
- 新建时同时写入两个字段（JSON + 旧数组）
- 编辑时从快照还原选择状态
- 删除部门时按 departmentId 精确清理
- 修复导出缺失列，从快照字段取值
- 旧字段 `responsiblePersons` / `cooperatePersons` 保留兼容，不删除

| 方案 C | 是 | 新增 `responsiblePersonSnapshots Json?` 和 `cooperatePersonSnapshots Json?` |
| 旧数据回填 | 建议但不必须 | 可从现有 `responsiblePersons` + `departmentIds` + `cooperateDepartmentIds` 做一次近似回填 |

### 7.8 是否影响现有权限逻辑

**不影响。** `responsiblePersons` / `cooperatePersons` 仅用于展示留底，不参与：
- 事项可见性判断（`canAccessWorkItem`）
- 附件权限判断（`canViewAttachment` / `canUploadAttachment` / `canDeleteAttachment`）
- 审批路由
- 待处理统计
- 退回处理人判定

### 7.9 是否建议进入下一阶段开发

- **P0/P1 修复可以立即启动**，风险低、不需要 migration
- **方案 C 建议在短期修复验证通过后再启动**，涉及 migration 和全链路联动
- **不建议在当前阶段引入 responsiblePersonIds**（与之前 userId 不追踪的决策一致），方案 C 是"姓名快照"和"userId 追踪"之间的合理中间方案

## 八、Phase 4A 实施结论

### 8.1 已修复

| # | 问题 | 修复方式 |
|---|------|---------|
| P0 | Excel 导入 `cooperateDepartmentIds` 硬编码为 `[]` | `cooperateDepartmentNames` 按 `/` 分割后通过 `resolveDeptId` 转换 |

| P1 | `responsiblePerson` / `cooperatePerson` 字符串不同步 | PUT 和 updateWork 补上拼接字符串序列化分支 |

### 8.2 已知限制

1. `responsiblePersons` / `cooperatePersons` 仍为**姓名数组**，不关联 userId
2. 删除部门时按**剩余部门人员姓名集合**过滤人员，同名人员无法精确区分
3. 人员调岗、停用后历史数据中的姓名快照**不会自动关联新部门**
4. 编辑回显时无法从姓名还原「该人员属于哪个部门」的来源信息
5. 这些限制**不影响权限判断**，责任人字段仅用于显示留底

### 8.3 Phase 4B（后续备选）

当出现以下需求时可启动：

- 同名人员导致编辑时清理不准确
- 需要追溯「某责任人当时属于哪个部门」
- 导出需要按部门分组展示责任人

**方案 C 要点：**

| 项目 | 内容 |
|------|------|
| 新增字段 | `responsiblePersonSnapshots Json?` / `cooperatePersonSnapshots Json?` |
| 字段结构 | `[{ departmentId, departmentName, name }]` |
| 写入时机 | 创建/编辑时从 MultiSearchSelect 选择结果写入 |
| 编辑回显 | 从快照还原部门-人员对应关系 |
| 删除部门 | 按 departmentId 精确清理，不受同名影响 |
| 需要 migration | 是（ALTER TABLE 加两列） |
| 旧字段 | `responsiblePersons` / `cooperatePersons` 保留兼容，不删除 |
| 对权限影响 | 无 |
