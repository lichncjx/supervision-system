# 首页统计口径

本文档定义首页 Dashboard 的当前口径和目标口径。目标是让首页卡片数量、轻量列表和点击后的列表结果一致。

## 一、当前调用链

第四阶段后，当前首页调用链：

1. `GET /api/dashboard`：一次返回首页卡片统计、临超期轻量列表、待处理轻量列表。
2. `GET /api/excel/completion-rate`：仅用于管理员 / 督办导出完成率。

当前已完成：

1. 首页不再调用 `getVisibleWorks(user)` 或 `/api/works` 拉取完整事项后前端计算临超期列表。
2. 首页不再前端基于完整事项计算待处理列表，改用 `lists.myActionRequired`。
3. `GET /api/dashboard/summary` 仍保留兼容，但首页不再直接依赖它。

当前保留：

1. 完成率仍由独立 `GET /api/dashboard/completion-rate` / `GET /api/excel/completion-rate` 提供，第四阶段不强制并入 `GET /api/dashboard`。
2. `SUPERVISOR` 的“督办跟踪”与普通用户“待处理”仍需后续阶段拆分清楚。

## 二、Dashboard 调用链

```http
GET /api/dashboard
```

返回：

1. `summary`：卡片统计。
2. `lists.expiringAndOverdue`：临超期轻量列表。
3. `lists.myActionRequired`：待处理轻量列表。

首页不再调用 `/api/works` 拉全量事项后自行计算列表。

## 三、统计字段来源

| 字段 | 目标口径 |
| --- | --- |
| 总数 | 当前用户组织可见范围内按事项类型统计 |
| 待审批 | 当前用户作为当前审批节点的事项 |
| 待办理 | 当前用户按部门管理权限需要推进的事项 |
| 进行中 | 当前 15 状态中兼容 `APPROVED` / `IN_PROGRESS` |
| 完成中 | 当前 15 状态中兼容 `PENDING_COMPLETE` / `PENDING_EVIDENCE_DEPT` / `PENDING_EVIDENCE_COMPANY` |
| 已完成 | `COMPLETED` |
| 已取消 | `CANCELLED` |
| 逾期 | 可见范围内未完成、未取消且超过计划时间 |
| 即将到期 | 可见范围内未完成、未取消且接近计划时间 |
| 完成率 | 按主责部门和事项类型统一计算 |

审批态 `PROPOSING`、`ADJUSTING`、`CANCELLING`、`COMPLETING` 不单独计入 `IN_PROGRESS`，但可以按需要在明细或审批统计中展示。

## 四、`GET /api/dashboard/summary` 迁移期兼容口径

当前仍未做 9 状态 schema 迁移，`GET /api/dashboard/summary` 按目标口径读取当前 15 状态：

| 字段 | 当前兼容实现 |
| --- | --- |
| 统计范围 | 通过服务端统一权限范围 helper 计算，`ADMIN` / `SUPERVISOR` 全局，部门角色按主责部门或配合部门，公司领导不默认全局 |
| 待审批 | 复用服务端 `canApproveWorkItem`，只统计当前审批节点属于当前用户的事项 |
| 待办理 | 复用服务端 `canHandleWorkItem`，按主责部门管理权限统计待推进事项 |
| 待处理 | 当前接口返回 `approving` 和 `handling`，调用方按二者求和 |
| 进行中 | 兼容 `APPROVED` 和 `IN_PROGRESS` |
| 审批中 | 兼容 `PENDING_DEPT`、`PENDING_COMPANY`、`ADJUSTING`、`CANCELLING`、`PENDING_MAIN_LEADER_CANCEL`、`PENDING_COMPLETE`、`PENDING_EVIDENCE_DEPT`、`PENDING_EVIDENCE_COMPANY`，用于待审批判断，不单独返回卡片字段 |
| 完成中 | 兼容 `PENDING_COMPLETE`、`PENDING_EVIDENCE_DEPT`、`PENDING_EVIDENCE_COMPANY`，归入目标 `COMPLETING` |
| 已完成 | `COMPLETED` |
| 已取消 | `CANCELLED` |
| 草稿 / 待修改 | `DRAFT` 和 `REJECTED` 兼容目标 `DRAFT`；退回办理优先参考 `firstSubmitterId` |
| 逾期 | 未完成、未取消且计划时间早于当前时间 |
| 临期 | 未完成、未取消且计划时间在当前时间起 7 天内 |

说明：

1. 终态 `COMPLETED` / `CANCELLED` 不计入临期或逾期。
2. `DRAFT` / `REJECTED` 不因状态本身排除临期或逾期，是否显示由可见范围和计划时间决定。
3. 当前接口新增 `expiring` 字段；迁移期保留 `thisMonthDue` 作为兼容别名，但其值已不再表示“本月到期”。
4. `SUPERVISOR` 的“督办跟踪”列表口径不应与普通用户的 `待处理 = 待审批 + 待办理` 混用；第四阶段 Dashboard 待处理列表暂不混入该特殊口径，后续需单独拆分展示语义。

## 五、不同角色统计口径

| 角色 | 统计范围 |
| --- | --- |
| `ADMIN` | 全局 |
| `SUPERVISOR` | 全局 |
| `DEPT_MANAGER` | 本部门主责或配合相关事项 |
| `DEPT_LEADER` | 本部门主责或配合相关事项 |
| `VICE_PRESIDENT` | 本人提出、分管、当前审批或明确授权事项 |
| `PRESIDENT` | 本人提出、当前审批、主要领导审批事项；全局统计需明确授权 |

## 六、TODO 多主责部门和配合部门口径

### 可见

1. 主责部门可见。
2. 配合部门可见。
3. 公司授权范围内角色可见。

### 待办理

1. 主责部门的 `DEPT_MANAGER` / `DEPT_LEADER` 可按动作办理。
2. 配合部门默认不获得主责办理权。
3. 责任人姓名不产生办理权限。

### 待审批

1. 只看 `currentApproverId` / `currentApproverRole`。
2. 部门级审批必须匹配部门范围。
3. 责任人姓名不产生审批权限。

### 完成率

1. 主责部门计入完成率。
2. 多个主责部门时，同一事项可分别计入多个主责部门。
3. 配合部门不计入完成率分母和分子。
4. 如果本轮做数据库迁移，则将当前 `departmentIds` 迁移/重命名为 `responsibleDepartmentIds`。
5. 如果本轮暂不做数据库迁移，则通过统一 helper，例如 `getResponsibleDepartmentIds(work)`，按 `responsibleDepartmentIds` -> `departmentIds` -> `departmentId` 顺序读取。
6. 权限、统计、导入导出中不要各自手写部门字段兼容逻辑，应通过统一 helper 或统一权限模块读取。

第二阶段当前实现：

1. `GET /api/dashboard/completion-rate` 已按 `getResponsibleDepartmentIds(work)` 统计主责部门，配合部门不计入完成率分母和分子。
2. `GET /api/excel/completion-rate` 已使用相同主责部门 helper 聚合。
3. `GET /api/dashboard/summary` 已使用统一可见范围和待审批/待办理 helper；第三阶段新增 `expiring`，`thisMonthDue` 仅作为迁移期兼容别名保留。
4. 第四阶段新增 `GET /api/dashboard`，首页已改为使用该接口返回的 `summary` 和轻量列表。

## 七、轻量列表字段

首页轻量列表只返回：

1. `id`
2. `title`
3. `type`
4. `typeLabel`
5. `status`
6. `statusLabel`
7. `departmentName`
8. `departmentNames`
9. `responsibleDepartmentNames`
10. `cooperateDepartmentNames`
11. `completeTime`（重点工作、主要工作当前真实字段）
12. `planCompleteTime`（待办事项当前真实字段）
13. `dueTime`（服务端派生展示字段）
14. `isOverdue`
15. `isExpiring`
16. `actionType`
17. `currentApproverName`

说明：

1. `prisma/schema.prisma` 中当前不存在 `planEndDate` 字段。首页统一展示“计划完成时间”时，使用服务端由 `completeTime` / `planCompleteTime` 派生的 `dueTime`。
2. `responsibleDepartmentNames` 和 `cooperateDepartmentNames` 由当前部门 ID 字段派生；当前阶段不修改 Prisma schema。

不得返回完整 `WorkItem`、审批记录、附件列表、长文本描述等详情页字段。
