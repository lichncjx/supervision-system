# 业务规则

> 当前系统运行态为 9 状态，详见 [状态机设计](rules/状态机设计.md)。

## 一、目标业务规则

### 1. 角色与数据范围

角色枚举以 `prisma/schema.prisma` 的 `Role` 为准。

| 枚举 | 中文名称 | 目标职责 |
|------|----------|----------|
| `ADMIN` | 系统管理员 | 系统配置、用户和部门管理、全局查看 |
| `SUPERVISOR` | 督办管理员 | 督办跟踪、统计导出、全局查看 |
| `DEPARTMENT_MANAGER` | 部门事项管理岗 / 部门督办管理员 / 系统办理人 | 代表部门填报、办理、提交材料 |
| `DEPARTMENT_LEADER` | 部门领导 | 查看本部门相关事项、进行部门级审批或办理 |
| `VICE_PRESIDENT` | 公司主管领导 | 查看本人提出、本人分管、当前审批或明确授权范围内事项 |
| `PRESIDENT` | 公司主要领导 | 查看本人提出、当前审批、需主要领导决策事项；是否全局查看需明确授权 |

目标数据范围：

1. `ADMIN`、`SUPERVISOR`：全局查看。
2. `VICE_PRESIDENT`：默认查看本人提出、本人分管、当前审批、明确授权范围内事项，不默认全局。
3. `PRESIDENT`：至少可查看本人提出、当前审批、需主要领导决策事项；是否全局查看需明确授权。
4. `DEPARTMENT_LEADER`、`DEPARTMENT_MANAGER`：查看本部门主责或配合相关事项，不按 `creatorId` / `firstSubmitterId` 限制查看。

### 2. User 与业务责任人边界

`User` 是系统登录账号，用于登录、审批、办理、上传、权限判断和流程审计。

业务责任人用于事项展示、责任留痕、导入导出，不一定登录系统，也不参与系统审批、办理、数据可见权限判断。

关键系统用户字段：

| 字段 | 目标语义 |
|------|----------|
| `creatorId` | 系统创建人 / 系统填报人 / 事项在系统中的经办账号 |
| `firstSubmitterId` | 首次提交人，用于立项审批退回后的处理参考 |
| `currentApproverId` | 当前审批节点的精确用户 |
| `currentApproverRole` | 当前审批节点的角色 |

权限判断原则：

1. 查询统计按组织权限。
2. 办理操作按部门管理权限。
3. 审批操作按 `currentApproverId` / `currentApproverRole`。
4. 核心内容修改必须走 `ADJUSTING`。
5. 业务责任人姓名字段不参与权限判断。

### 3. 重点工作 / 主要工作责任字段

目标架构中，重点工作、主要工作的业务责任字段统一理解为：

| 字段 | 类型 | 说明 | 权限作用 |
|------|------|------|----------|
| `responsibleLeader` | 姓名文本 | 事项业务责任领导 | 无 |
| `responsiblePerson` | 姓名文本 | 事项实际责任人 | 无 |

规则：

1. `responsibleLeader` 是事项业务责任领导姓名文本。
2. `responsiblePerson` 是事项实际责任人姓名文本。
3. 二者不是 `User` 外键。
4. 二者不是 `Member` 外键。
5. 二者不代表系统登录用户。
6. 二者只用于展示、导入导出和历史留痕。
7. 二者不参与审批权限、办理权限、数据可见权限判断。
8. 审批权限仍依赖 `currentApproverId` / `currentApproverRole`。
9. 系统填报和退回处理仍依赖 `creatorId` / `firstSubmitterId`。
10. 部门办理权限仍依赖当前登录用户的角色和部门。

### 4. 待办事项责任字段

本轮不新增：

1. `Member` 表。
2. `DepartmentMember` 表。
3. `WorkParticipant` 表。

Phase 8B 已将责任方模型统一为：

| 字段 | 说明 | 权限作用 |
|------|------|----------|
| `departmentId` | 唯一主责部门 | 参与权限、可见范围、完成率统计 |
| `responsibleLeader` | 主责责任领导姓名文本 | 无（仅展示、导入导出、历史留痕） |
| `responsiblePerson` | 主责责任人姓名文本 | 无（仅展示、导入导出、历史留痕） |
| `cooperators` | 配合方列表 JSON | 参与可见范围（不参与完成率统计） |

已删除字段：`responsibleDepartmentIds`、`cooperateDepartmentIds`、`responsiblePersons`、`cooperatePersons`。

规则：

1. `responsibleLeader` / `responsiblePerson` / `cooperators[].leader` / `cooperators[].person` 不参与权限判断。
2. 主责部门唯一（`departmentId`）。
3. 配合方多个（`cooperators`），每个配合方包含 `departmentId` / `departmentName` / `leader` / `person`。
4. 配合部门仅参与可见范围，不计入完成率。

### 5. 部门字段目标口径

Phase 8B 已完成责任方模型统一：

| 字段 | 说明 |
|------|------|
| `departmentId` | 唯一主责部门 |
| `cooperators` | 配合方列表 JSON |

当前规则：

1. `getResponsibleDepartmentIds(work)`：返回 `[work.departmentId]`（唯一主责部门）。
2. `getCooperatorDepartmentIds(work)`：从 `cooperators` 解析配合部门 ID。
2. 权限、统计、导入导出中通过统一 helper 或统一权限模块读取，不各自手写兼容逻辑。
3. `departmentId` 作为单主责部门字段保留，`departmentIds` 已删除。

### 6. 目标状态机

目标 `WorkItemStatus` 收敛为 9 个状态：

| 分类 | 状态 | 说明 |
|------|------|------|
| 稳定业务态 | `DRAFT` | 草稿/待修改 |
| 稳定业务态 | `PENDING_DECOMPOSE` | 待分解，仅服务于公司领导发起的待办事项 |
| 稳定业务态 | `IN_PROGRESS` | 进行中 |
| 稳定业务态 | `COMPLETED` | 已完成 |
| 稳定业务态 | `CANCELLED` | 已取消 |
| 审批过程态 | `PROPOSING` | 立项/分解方案审批中 |
| 审批过程态 | `ADJUSTING` | 调整审批中 |
| 审批过程态 | `CANCELLING` | 取消审批中 |
| 审批过程态 | `COMPLETING` | 完成审批中 |

设计原则：

1. `APPROVED` 已取消，审批通过后统一进入 `IN_PROGRESS`。
2. `REJECTED` 已取消，立项审批退回后回到 `DRAFT`。
3. `PENDING_EVIDENCE_DEPT`、`PENDING_EVIDENCE_COMPANY`、`PENDING_COMPLETE` 已统一为 `COMPLETING`。
4. `PENDING_MAIN_LEADER_CANCEL` 不再单独作为状态，在 `CANCELLING` 状态下通过 `currentApproverId`、`currentApproverRole`、`approvalType` 判断当前审批节点。
5. `PENDING_DECOMPOSE` 保留，它不是审批态，而是公司领导发起待办后的业务准备 / 待分解状态。

完整状态设计见 `docs/rules/状态机设计.md`。

### 7. 审批辅助字段

PR 6.1 已在 `WorkItem` 上新增：

1. `beforeApprovalStatus WorkItemStatus?`
2. `approvalType ApprovalType?`

`ApprovalType` 建议包括：

1. `PROPOSE`
2. `ADJUST`
3. `CANCEL`
4. `COMPLETE`

进入审批态时：

1. `beforeApprovalStatus = 当前业务状态`。
2. `approvalType = 当前审批类型`。
3. `status = 对应审批态`。
4. `currentApproverId` 或 `currentApproverRole` 指向第一审批节点。

审批退回时：

1. `status = beforeApprovalStatus`。
2. `beforeApprovalStatus = null`。
3. `approvalType = null`。
4. `currentApproverId = null`。
5. `currentApproverRole = null`。

审批通过时：

1. 如果还有下一审批节点，则 `status` 保持当前审批态，仅切换 `currentApproverId` / `currentApproverRole`。
2. 如果审批结束，则 `status` 进入目标业务状态，并清空审批辅助字段和当前审批人字段。

特别说明：

1. 普通事项 `DRAFT -> PROPOSING`，退回后回 `DRAFT`。
2. 公司领导发起待办 `PENDING_DECOMPOSE -> PROPOSING`，退回后回 `PENDING_DECOMPOSE`。
3. 不使用 `needDecompose` 之类布尔字段解决退回路径，退回路径以 `beforeApprovalStatus` 为准。

### 8. 目标审批与办理口径

待我处理 = 待审批 + 待办理。

待审批：

1. 只统计当前用户作为当前审批节点的事项。
2. 判断依据为 `currentApproverId` / `currentApproverRole`。
3. 部门级审批必须匹配部门范围。

待办理：

1. 指不需要当前用户审批，但需要当前用户立即处理的非审批事项。
2. 当前阶段具体包括：
   - `DRAFT`：仅 `creatorId` 对应用户（非退回草稿）或 `firstSubmitterId ?? creatorId` 对应用户（退回草稿）；部门领导、部门主管不能因同部门而处理他人草稿。
   - `PENDING_DECOMPOSE`：仅主责部门相关部门级角色（`DEPARTMENT_MANAGER`、`DEPARTMENT_LEADER`）；配合部门不默认计入。
   - `IN_PROGRESS`（审批退回后）：仅当 `rejectReason` / `rejectedFromStatus` 非空时，由 `firstSubmitterId ?? creatorId` 对应用户办理；普通 `IN_PROGRESS` 不进入待办理。
3. 配合部门可见不等于待办理；公司领导、督办管理员的全局查看权限不产生待办理。
4. 责任领导、责任人、主责责任人、配合责任人姓名不产生办理权限。

### 9. 目标统计口径

1. 首页统计卡片必须与点击后的列表结果一致。
2. `待审批`、`待办理`、`待我处理` 的统计条件必须与对应列表查询条件一致。
3. 超期、进行中、已完成等状态统计必须使用统一状态集合。
4. TODO 多主责部门完成率按主责部门计入；配合部门不计入完成率分母和分子。
5. 统计涉及主责部门时必须通过统一 helper 或统一权限模块读取主责部门集合。

### 10. Dashboard API

首页已统一调用：

```http
GET /api/dashboard
```

首页不再调用 `/api/works` 拉全量事项后在前端计算列表。目标接口一次返回：

1. `summary`：卡片统计。
2. `lists.expiringAndOverdue`：临超期轻量列表。
3. `lists.myActionRequired`：待处理轻量列表。

轻量列表只返回首页需要字段，不返回完整 `WorkItem`。

### 11. API 和服务端约束

1. 创建、更新、删除、审批、提交材料、导入导出等关键接口必须校验登录态。
2. 涉及部门数据时必须校验数据范围。
3. 涉及审批时必须校验当前审批人。
4. 涉及状态变化时必须校验当前状态是否允许该动作。
5. 不允许通过普通更新接口绕过 workflow 修改状态、当前审批人或审批辅助字段。
6. 核心内容修改必须走 `ADJUSTING`。

### 12. 附件与证明材料目标口径

1. 附件查看应与事项查看权限保持一致。
2. 附件上传应与事项办理权限或当前动作权限保持一致。
3. 上传人可以删除自己上传的附件。
4. `ADMIN` / `SUPERVISOR` 可按管理口径查看和删除附件。
5. 业务责任人姓名字段不产生附件查看、上传或删除权限。

### 2. Phase 7 已删除的历史字段

以下字段原存于当前实现，Phase 7 已从 schema 中删除：

| 已删除字段 | 原用途 | 替换为 |
|------------|--------|--------|
| `deptLeaderId` | 部门领导用户关联 | 删除，不引入替代 ID 字段 |
| `deptManagerId` | 部门主管/管理岗用户关联 | 删除，不引入替代 ID 字段 |
| `deptLeaderName` | 部门领导姓名快照 | `responsibleLeader` |
| `deptManagerName` | 部门主管姓名快照 | `responsiblePerson` |
| `supervisor` | 历史文本字段 | `responsiblePerson` |
| `departmentIds` | 多部门 ID 数组 | `responsibleDepartmentIds` |

当前目标字段统一为：`departmentId`（唯一主责部门）、`responsibleLeader`（责任领导）、`responsiblePerson`（责任人）、`cooperators`（配合方）。`responsibleDepartmentIds` / `cooperateDepartmentIds` / `responsiblePersons` / `cooperatePersons` 已删除（Phase 8B）。

### 3. 历史公司领导查看范围差异

历史部分代码和文档曾把 `VICE_PRESIDENT` / `PRESIDENT` 作为公司级角色处理，可能获得较宽甚至全局查看范围。当前目标口径不是这样：

1. `ADMIN`、`SUPERVISOR` 才是默认全局查看。
2. `VICE_PRESIDENT` 默认只查看本人提出、本人分管、当前审批、明确授权范围内事项。
3. `PRESIDENT` 至少查看本人提出、当前审批、需主要领导决策事项；是否全局查看需明确授权。

### 4. 历史退回实现差异

历史实现存在 `REJECTED` 状态，创建审批退回后进入 `REJECTED`，重新提交再进入 `PENDING_DEPT` 或 `PENDING_COMPANY`。

当前目标架构已取消 `REJECTED`，立项审批退回恢复为 `DRAFT`。退回路径以 `beforeApprovalStatus` 为准，退回处理仍参考 `firstSubmitterId`。

### 5. 历史完成审批实现差异

历史实现中：

1. 重点/主要工作使用 `PENDING_EVIDENCE_DEPT` / `PENDING_EVIDENCE_COMPANY`。
2. 待办事项使用 `PENDING_COMPLETE`。

当前目标架构统一为 `COMPLETING`。

### 6. 当前附件权限实现差异

当前附件权限主要由 `src/lib/attachment-permissions.ts` 管理，但部分判断仍依赖标量 `departmentId`、创建人或当前历史责任字段。目标上附件权限应与统一事项查看和办理权限模块对齐。

## 三、兼容说明

### 1. 状态兼容

当前运行态只维护 9 状态。旧状态映射仅用于历史数据迁移说明，不再作为当前业务口径。

### 2. 字段兼容

Phase 7 已完成字段命名规范化，旧字段 `deptLeaderId`、`deptManagerId`、`deptLeaderName`、`deptManagerName`、`supervisor`、`departmentIds` 已从 schema 删除。

重点/主要工作当前字段为：

1. `responsibleLeader`：责任领导姓名文本；
2. `responsiblePerson`：责任人/实际负责人姓名文本。

待办事项当前字段为：

1. `responsiblePersons`：主责责任人姓名数组；
2. `cooperatePersons`：配合责任人姓名数组。

上述字段不是 `User` 外键，不是 `Member` 外键，不参与权限判断。

### 3. 部门读取

`getResponsibleDepartmentIds(work)` 返回 `[work.departmentId]`（唯一主责部门）。

`getCooperatorDepartmentIds(work)` 从 `cooperators` JSON 解析配合部门 ID。

`responsibleDepartmentIds` / `cooperateDepartmentIds` / `departmentIds` 已删除（Phase 7 重命名 → Phase 8B 删除）。

权限、统计、导入导出、Dashboard 不得各自手写兼容逻辑。

### 4. 人员名录兼容

本轮保持以下结论：

1. 不新增 `Member` / `DepartmentMember` 表。
2. 不新增 `WorkParticipant` 表。
3. `responsiblePersons` / `cooperatePersons` 继续保留姓名文本或 JSON 数组逻辑。
4. `responsibleLeader` / `responsiblePerson` 也是姓名文本字段。
5. 所有业务责任人姓名字段只用于展示、导入导出、历史留痕，不参与权限判断。
