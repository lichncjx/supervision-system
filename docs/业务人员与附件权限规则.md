# 业务人员与附件权限规则

> 本文档是业务人员字段命名体系、人员体系分类、附件机制和附件权限规则的集中参考。
> 权限、审批流、状态机、统计口径的权威文档仍是 `docs/业务规则.md`。

## 一、业务人员字段命名规则

所有涉及人员信息的字段遵循 **xxxId / xxxName 两层命名约定**：

### xxxId（真实关联字段）

- 类型：`Int` 或 `Int?`，FK 指向 `users.id`
- 用途：权限判断、流程路由、审批指派、上传权限、退回处理人判定、待处理统计
- 特征：有值时必须指向有效的用户记录；服务端校验用户存在性、所属部门、角色和启用状态

### xxxName（历史快照字段）

- 类型：`String?`，纯文本
- 用途：页面展示、Excel 导出、历史留痕
- 特征：创建/更新时从 User 记录同步写入，后续即使 User 姓名变更也不回溯更新
- 没有 FK 约束，不参与权限判断

### 旧字段兼容策略

| 旧字段 | 新字段（Name） | 新字段（ID） | 状态 |
|--------|---------------|-------------|------|
| `responsibleLeader` | `deptLeaderName` | `deptLeaderId` | legacy，保留不删除 |
| `supervisor` | `deptManagerName` | `deptManagerId` | legacy，保留不删除 |
| `responsiblePersons` | — | — | legacy，暂不做 userId 化 |
| `cooperatePersons` | — | — | legacy，暂不做 userId 化 |

**权限判断原则：只认 ID，不认姓名文本。** 服务端权限校验一律使用 ID 字段匹配。

## 二、重点工作 / 主要工作人员体系

重点工作（`PRIORITY`）和主要工作（`MAIN`）采用**单部门、单线条办理体系**。

### 字段一览

| 字段 | 类型 | 说明 |
|------|------|------|
| `departmentId` | `Int?` | 单一主办部门 ID（FK → departments） |
| `deptLeaderId` | `Int?` | 部门领导用户 ID（FK → users），关联 `DEPARTMENT_LEADER` 角色用户 |
| `deptLeaderName` | `String?` | 部门领导姓名快照，展示/导出优先读此字段 |
| `deptManagerId` | `Int?` | 主管人员/业务主管人员用户 ID（FK → users），关联 `DEPARTMENT_MANAGER` 角色用户 |
| `deptManagerName` | `String?` | 主管人员姓名快照，展示/导出优先读此字段 |
| `responsibleLeader` | `String?` | **[legacy]** 部门领导姓名，兼容 `deptLeaderName` |
| `supervisor` | `String?` | **[legacy]** 主管人员姓名，兼容 `deptManagerName` |

### 关键规则

1. `supervisor` 不是 `User.role = SUPERVISOR`。
   - `User.role = SUPERVISOR` 表示**督办管理员 / 全局监督管理角色**。
   - 事项字段 `supervisor` 是旧文本字段，表示该事项的业务主管人员。
2. 创建/更新时双写新旧字段（如同时写入 `deptLeaderId` + `responsibleLeader` + `deptLeaderName`），确保旧字段兼容。
3. 服务端校验所选用户所属部门、角色（`DEPARTMENT_LEADER` / `DEPARTMENT_MANAGER`）和启用状态（`isActive`）。
4. 部门切换（`departmentId` 变化）时清空 `deptLeaderId` / `deptManagerId`，需要重新选择。
5. 退回编辑表单支持部门领导和主管人员的下拉选择。
6. 详情页、列表页、Excel 导出优先展示新字段，旧字段 fallback。
7. 搜索功能包含 `deptLeaderName` / `deptManagerName`。

## 三、待办事项人员体系

待办事项（`TODO`）采用**多部门、多责任人体系**。

### 字段一览

| 字段 | 类型 | 说明 |
|------|------|------|
| `departmentIds` | `Int[]` | 主责部门 ID 数组（当前命名，未来可迁移为 `responsibleDepartmentIds`） |
| `responsiblePersons` | `String[]` | **[legacy]** 主责责任人姓名数组，仅用于展示留底 |
| `cooperateDepartmentIds` | `Int[]` | 配合部门 ID 数组 |
| `cooperatePersons` | `String[]` | **[legacy]** 配合责任人姓名数组，仅用于展示留底 |

### 关键规则

1. 待办事项责任人暂不做 userId 化，仍使用姓名字符串数组。
2. 配合部门（`cooperateDepartmentIds`）中的用户可通过 `canAccessWorkItem` 查看事项（服务端已做权限匹配）。
3. 配合部门用户**不因配合部门身份**获得上传或删除附件权限。
4. `departmentIds` 当前既表示主责部门也承担部分旧逻辑，后续可考虑重命名为 `responsibleDepartmentIds` 并与 `cooperateDepartmentIds` 明确分离。

## 四、公司领导字段

| 字段 | 类型 | 说明 |
|------|------|------|
| `proposedLeaderId` | `Int?` | 提出领导 ID（`VICE_PRESIDENT` 或 `PRESIDENT`），FK → users |
| `approvalLeaderId` | `Int?` | 审批领导 ID，负责调整、取消、完成审批 |
| `proposedLeader` | relation | 前端展示字段，由 Prisma relation 从 `proposedLeaderId` 解析 |
| `approvalLeader` | relation | 前端展示字段，由 Prisma relation 从 `approvalLeaderId` 解析 |

### 关键规则

1. `approvalLeaderId` 默认等于 `proposedLeaderId`（创建/导入/表单三处统一）。
2. 特殊情况下二者可以不同（如提出领导变更或指定其他审批人）。
3. `proposedLeader` / `approvalLeader` 如果在前端存在，仅作为展示字段或由 relation 解析，不作为数据库主字段。
4. 后续如有迁移，可考虑 `proposedLeaderName` / `approvalLeaderName` 命名方式。

## 五、证明材料 / 附件机制

### 存储模型

- 通用附件和证明材料**统一存储在 `attachments` 表**。
- `attachments.category` 字段区分类型：
  - `"general"` — 普通附件（默认值，兼容历史数据）
  - `"evidence"` — 完成证明材料 / 见证材料

### 上传流程

```
WorkOperationPanel 选择文件
  → POST /api/upload (category=evidence)
  → 文件写入磁盘 + attachments 表记录
  → 返回 attachment 对象
  → 前端收集 evidence attachment 列表
  → 点击"提交完成材料"
  → POST /api/works/[id]/workflow { action: 'evidence', proof }
  → submitEvidence 仅处理 proof 文本和状态流转
```

### 前端展示规则

| 区域 | 显示内容 | 数据来源 |
|------|---------|---------|
| 普通附件面板（WorkAttachmentPanel） | `category != "evidence"` 的附件 | `work.attachments` 过滤 |
| 证明材料区 | `category == "evidence"` 的附件 | `work.attachments` 过滤 |
| 证明材料文字 | `proof` 文本 | `work_items.proof` |

### 与旧系统的区别

| 维度 | 旧方案 | 新方案 |
|------|--------|--------|
| 证明文件存储 | 前端 React state（base64），页面刷新丢失 | `attachments` 表持久化 |
| 证明文件传输 | `submitComplete` 丢弃 `_proofFiles` 参数 | 通过 `/api/upload` 上传后 workflow 引用 |
| 附件分类 | 无分类 | `category` 字段区分 `general` / `evidence` |
| 详情页证明展示 | 死代码，永远不执行 | 从 `work.attachments` 按 category 筛选展示 |

### 不变的部分

- `proof` 文本字段仍只保存文字说明。
- `submitEvidence`（`src/lib/workflow.ts`）不改，继续只处理 proof 文本和状态流转。
- workflow `action=evidence` 路由不改，不从 body 解构文件数据。

## 六、附件权限规则

附件权限统一由 `src/lib/attachment-permissions.ts` 模块集中管理。三个 API 端点（`/api/upload`、`/api/attachments/[id]`、`/api/attachments/[id]/download`）统一调用此模块的三个导出函数。

### 6.1 查看 / 下载权限 (`canViewAttachment`)

```
ADMIN / SUPERVISOR                     → 全权限
事项创建人 (creatorId)                  → 可查看下载
同部门用户 (departmentId 匹配)          → 可查看下载
重点/主要工作的 deptManagerId 对应用户   → 可查看下载
VICE_PRESIDENT / PRESIDENT             → 公司级可见（isCompanyLevel）
```

### 6.2 上传权限 (`canUploadAttachment`)

```
ADMIN / SUPERVISOR                                          → 可上传
deptManagerId 对应用户（重点/主要工作）                       → 可上传（含 rejected 状态）
同部门 DEPARTMENT_MANAGER / DEPARTMENT_LEADER (非 rejected)  → 可上传
同部门 DEPARTMENT_MANAGER / DEPARTMENT_LEADER (rejected)     → 不可上传
VICE_PRESIDENT / PRESIDENT                                  → 不可上传
配合部门用户                                                  → 不因配合身份获得上传权限
终止态 (completed / cancelled)                               → 普通用户和 deptManagerId 均不可上传
```

**deptManagerId 特殊规则**：
- `deptManagerId` 在 `rejected` 状态下也可上传，用于退回后补充材料。
- 这是 `deptManagerId` 与同部门普通 `DEPARTMENT_MANAGER` 的上传权限区别。

### 6.3 删除权限 (`canDeleteAttachment`)

删除权限已收紧（Phase 3C），当前规则：

```
ADMIN              → 可删除所有附件
SUPERVISOR         → 可删除所有附件
上传者本人           → 可删除自己上传的附件
deptManagerId      → 只能删除自己上传的附件，不可删除他人附件
同部门 DEPARTMENT_MANAGER / DEPARTMENT_LEADER → 不再默认删除他人附件
配合部门用户         → 不能删除他人附件
```

**历史变更**：Phase 3C 之前，同部门 `DEPARTMENT_MANAGER` / `DEPARTMENT_LEADER` 可以删除部门下所有附件。该宽权限已被移除，统一收紧为上传者本人才能删除。

### 6.4 权限模块接口

```typescript
// src/lib/attachment-permissions.ts

interface AttPermUser {
  id: number;
  role: string;
  departmentId: number;
}

interface AttPermWorkItem {
  departmentId: number | null;
  status: string;
  creatorId: number;
  type: string; // 'PRIORITY' | 'MAIN' | 'TODO'
  deptManagerId?: number | null;
}

interface AttPermAttachment {
  userId: number;
}

canViewAttachment(user, workItem): boolean;
canUploadAttachment(user, workItem): boolean;
canDeleteAttachment(user, workItem, attachment): boolean;
```

## 七、数据库字段变更记录

### Phase 2 — WorkItem 新增字段

```prisma
deptLeaderId    Int?     // FK → users, onDelete: SetNull
deptManagerId   Int?     // FK → users, onDelete: SetNull
deptLeaderName  String?  @db.VarChar(50)
deptManagerName String?  @db.VarChar(50)
```

- 有 Prisma relation（`deptLeader` / `deptManager`），`onDelete: SetNull`
- 历史数据需要执行回填 SQL

### Phase 3A — Attachment 新增字段

```prisma
category  String  @default("general")  @db.VarChar(30)
```

- 默认 `"general"`，存量附件自动兼容
- 无需回填

## 八、部署 / migration 注意事项

1. Vercel Preview 环境**不会自动执行数据库 migration**，需手动执行 `pnpm prisma:deploy` + `pnpm prisma:generate`。
2. Supabase / 本地 Docker / 群晖内网数据库都需要执行 `prisma migrate deploy`。
3. 执行 migration 后需要 `prisma generate`，确保 Prisma Client 类型一致。
4. 涉及 Phase 2 migration 时，除 DDL 外还需执行回填 SQL。
5. 旧字段 `responsibleLeader` / `supervisor` 未删除，不需要清理历史数据。
6. `attachments.category` 默认 `general`，可兼容存量附件，无需额外操作。
7. Phase 1 无数据库结构变化。
