# 回归测试清单

> 后续测试策略以目标业务规则为准，优先运行自动化验证脚本。
> 不再为了保护当前 Dashboard 统计、状态口径、权限范围中的潜在错误而编写大量人工基线测试。

本文档分为两部分：

1. **目标口径自动化测试**：使用固定测试数据和目标业务规则计算期望结果，优先用于后续重构验收。
2. **当前实现迁移前烟测**：只用于确认当前系统基本可操作，不作为统计、权限、状态口径的保护基线。

## 自动化验证脚本

脚本位置：

| 脚本 | 作用 |
|------|------|
| `tests/target-contract/seed-target-data.cjs` | 清空测试数据并创建固定角色、部门、事项数据 |
| `tests/target-contract/verify-target-contract.cjs` | 调用当前接口并按目标口径比对结果 |
| `tests/target-contract/run-target-contract.cjs` | 一键执行 seed + verify |
| `tests/target-contract/target-contract-data.cjs` | 固定测试数据和目标口径计算规则 |

推荐执行：

```bash
TARGET_CONTRACT_ENV=local TARGET_CONTRACT_RESET=1 node tests/target-contract/run-target-contract.cjs --base-url http://localhost:5000
```

说明：

1. 执行前需要测试服务已启动，并连接一次性本地 / 测试数据库。
2. seed 脚本会清空 `work_items`、`workflow_records`、`attachments`、`operation_logs`、`users`、`departments` 后重建固定数据。
3. seed 脚本默认拒绝清库，必须显式设置 `TARGET_CONTRACT_ENV=local|test` 且 `TARGET_CONTRACT_RESET=1`。
4. seed 和 verify 均拒绝 Vercel 环境、Supabase/Vercel/production/preview 字样的 `DATABASE_URL`、以及非本地主机数据库。
5. verify 脚本只允许调用本地 `http://localhost` / `http://127.0.0.1` / `http://[::1]` 服务。
6. 脚本会输出脱敏后的 `DATABASE_URL`，用于确认当前连接的是一次性本地 / 测试数据库。
7. verify 脚本不会以当前接口结果为基线，而是按 docs 中的目标业务规则计算期望值。
8. 当前不通过的目标口径会输出 `EXPECTED_FAILURE`，并同步记录到 `docs/待修复问题.md`。

第四阶段 Dashboard 统一接口和首页调用链收口后的目标结果：

```text
PASS 66
```

说明：

1. 第二阶段已通过的 `/api/works` 可见范围、`VICE_PRESIDENT` 默认可见范围、部门多主责 / 配合部门可见、完成率部门 B 多主责统计，不再作为业务待修复问题记录。
2. 第三阶段收口后，`GET /api/dashboard/summary` 相关角色断言已全部按正常目标断言显示为 `PASS`。
3. 第四阶段新增 `GET /api/dashboard` summary 一致性、轻量列表字段、临超期列表、待处理列表口径断言，已全部显示为 `PASS`。
4. 当前已无 `EXPECTED_FAILURE`、`UNEXPECTED_PASS` 或普通 `FAIL`。
4. 后续如果出现普通 `FAIL`，才代表脚本错误、环境错误或新增非预期回归。

固定测试账号默认密码均为 `123456`。

## 一、目标口径自动化测试

本部分作为后续重构验收主线，优先通过脚本验证，不要求人工逐项点击。

### 1. 固定角色用户

seed 脚本创建以下用户：

2. `supervisor`

5. `vp_b`
6. `dept_leader_a`
7. `dept_manager_a1`
8. `dept_manager_a2`
9. `dept_leader_b`
10. `dept_manager_b1`

### 2. 固定事项覆盖

seed 脚本创建覆盖以下场景的事项：

1. 普通重点工作。
2. 普通主要工作。
3. 公司领导发起的待办。
4. 多主责部门待办。
5. 多配合部门待办。
6. 待分解事项。
7. 立项审批中事项。
8. 进行中事项。
9. 调整中事项。
10. 取消中事项。
11. 完成中事项。
12. 已完成事项。
13. 已取消事项。
14. 临期事项。
15. 超期事项。

### 3. 自动化断言范围

验证脚本至少覆盖：

1. 不同角色调用 Dashboard 后的 summary 数量。
2. 待审批数量。
3. 待办理数量。
4. 待处理数量。
5. 进行中数量。
6. 临期和超期数量。
7. 公司领导不能看到未到公司层面的部门内部事项。
8. `VICE_PRESIDENT` 不能默认看到另一个副总负责的事项。
9. `DEPARTMENT_MANAGER` / `DEPARTMENT_LEADER` 查看本部门相关事项时不按 `creatorId` / `firstSubmitterId` 限制。
10. `responsiblePersons` / `cooperatePersons` 不参与权限判断。
11. `responsibleDepartmentIds` / `cooperateDepartmentIds` 参与组织权限判断。
12. 完成率统计中主责部门和配合部门的口径。

脚本输出必须显示：

1. 角色。
2. 接口。
3. 实际结果。
4. 期望结果。
5. 是否通过。

### 4. 目标 9 状态路径

目标状态：

`DRAFT`、`PENDING_DECOMPOSE`、`PROPOSING`、`IN_PROGRESS`、`ADJUSTING`、`CANCELLING`、`COMPLETING`、`COMPLETED`、`CANCELLED`。

目标验证路径：

1. `DRAFT -> PROPOSING -> IN_PROGRESS`
2. `PROPOSING` 退回 -> `beforeApprovalStatus`
3. `PENDING_DECOMPOSE -> PROPOSING -> IN_PROGRESS`
4. `PENDING_DECOMPOSE -> PROPOSING` 退回 -> `PENDING_DECOMPOSE`
5. `IN_PROGRESS -> ADJUSTING -> IN_PROGRESS`
6. `IN_PROGRESS -> CANCELLING -> CANCELLED`
7. `IN_PROGRESS -> COMPLETING -> COMPLETED`
8. `COMPLETING` 退回 -> `IN_PROGRESS`
9. `DRAFT -> CANCELLED`

目标测试中，重点工作、主要工作字段使用：

1. `responsibleLeader`：事项业务责任领导姓名文本。
2. `responsiblePerson`：事项实际责任人姓名文本。

目标测试不再把“部门领导 / 主管人员”作为重点/主要事项的目标业务责任字段。`DEPARTMENT_MANAGER` 表示部门事项管理岗 / 部门督办管理员 / 系统办理人，不等于事项实际责任人。

## 三、目标架构人工补充检查

本部分只在自动化脚本无法覆盖 UI 展示、按钮可见性、表单回显等体验问题时补充执行。

目标状态仍使用：

`DRAFT`、`PENDING_DECOMPOSE`、`PROPOSING`、`IN_PROGRESS`、`ADJUSTING`、`CANCELLING`、`COMPLETING`、`COMPLETED`、`CANCELLED`。

目标测试中，重点工作、主要工作字段使用：

1. `responsibleLeader`：事项业务责任领导姓名文本。
2. `responsiblePerson`：事项实际责任人姓名文本。

目标测试不再把“部门领导 / 主管人员”作为重点/主要事项的目标业务责任字段。`DEPARTMENT_MANAGER` 表示部门事项管理岗 / 部门督办管理员 / 系统办理人，不等于事项实际责任人。

### 1. 立项审批

- [ ] 重点/主要工作从 `DRAFT` 提交后进入 `PROPOSING`
- [ ] 待办事项从 `DRAFT` 提交后进入 `PROPOSING`
- [ ] `PROPOSING` 审批通过后进入 `IN_PROGRESS`
- [ ] `PROPOSING` 退回后恢复 `beforeApprovalStatus`
- [ ] 普通事项 `DRAFT -> PROPOSING` 退回后回到 `DRAFT`

### 2. 公司领导发起待办分解

- [ ] 公司领导发起待办后进入 `PENDING_DECOMPOSE`
- [ ] 部门完成分解并提交后进入 `PROPOSING`
- [ ] `PENDING_DECOMPOSE -> PROPOSING -> IN_PROGRESS` 路径通过
- [ ] `PENDING_DECOMPOSE -> PROPOSING` 退回后恢复 `PENDING_DECOMPOSE`
- [ ] 不使用 `needDecompose` 之类布尔字段判断退回路径

### 3. 调整审批

- [ ] `IN_PROGRESS -> ADJUSTING`
- [ ] `ADJUSTING` 审批通过后回到 `IN_PROGRESS`
- [ ] `ADJUSTING` 退回后恢复 `beforeApprovalStatus`
- [ ] 多节点调整审批中，状态保持 `ADJUSTING`，只切换 `currentApproverId` / `currentApproverRole`

### 4. 取消审批

- [ ] `IN_PROGRESS -> CANCELLING`
- [ ] `CANCELLING` 审批通过后进入 `CANCELLED`
- [ ] 重点工作需要主要领导审批时，状态仍保持 `CANCELLING`
- [ ] 主要领导审批节点通过 `currentApproverId` / `currentApproverRole` / `approvalType` 表达
- [ ] 不再使用 `PENDING_MAIN_LEADER_CANCEL` 作为目标状态

### 5. 完成审批

- [ ] `IN_PROGRESS -> COMPLETING`
- [ ] `COMPLETING` 审批通过后进入 `COMPLETED`
- [ ] `COMPLETING` 退回后恢复 `IN_PROGRESS`
- [ ] 重点/主要工作和待办事项完成审批统一使用 `COMPLETING`

### 6. 草稿取消

- [ ] `DRAFT -> CANCELLED`
- [ ] 草稿取消不产生审批过程态
- [ ] 取消后不可继续编辑或提交

### 7. 审批辅助字段

- [ ] 进入 `PROPOSING`、`ADJUSTING`、`CANCELLING`、`COMPLETING` 时写入 `beforeApprovalStatus`
- [ ] 进入审批态时写入 `approvalType`
- [ ] 审批退回时清空 `beforeApprovalStatus`、`approvalType`、`currentApproverId`、`currentApproverRole`
- [ ] 审批结束时清空 `beforeApprovalStatus`、`approvalType`、`currentApproverId`、`currentApproverRole`
- [ ] 审批仍有下一节点时，状态保持当前审批态，仅切换当前审批节点

### 8. 目标字段与权限

- [ ] 重点/主要工作使用 `responsibleLeader` 和 `responsiblePerson` 展示业务责任信息
- [ ] `responsibleLeader` / `responsiblePerson` 不是 `User` 外键
- [ ] `responsibleLeader` / `responsiblePerson` 不是 `Member` 外键
