# 待修复问题

本文档记录目标口径自动化验证脚本当前预期会暴露的问题。这里的问题不表示脚本错误，而是当前接口与目标业务规则之间的差异。

验证脚本：

```bash
TARGET_CONTRACT_ENV=local TARGET_CONTRACT_RESET=1 node tests/target-contract/run-target-contract.cjs --base-url http://localhost:5000
```

安全要求：

1. `TARGET_CONTRACT_ENV` 必须显式设置为 `local` 或 `test`。
2. `TARGET_CONTRACT_RESET=1` 是清库 seed 的强制条件。
3. `DATABASE_URL` 必须指向本地主机数据库，允许 host：`localhost`、`127.0.0.1`、`::1`、`db`、`postgres`。
4. 脚本拒绝包含 `supabase.co`、`pooler.supabase.com`、`vercel`、`production`、`prod`、`preview` 的数据库 URL。
5. 脚本拒绝在 `VERCEL` / `VERCEL_ENV` 环境中运行。
6. verify 脚本只允许调用本地 `http://localhost` / `http://127.0.0.1` / `http://[::1]` 服务。
7. 脚本会输出脱敏后的 `DATABASE_URL`，用于人工确认没有误连生产库、Supabase Preview 库或 Vercel 环境库。

## 一、Dashboard 目标接口缺失

目标接口为：

```http
GET /api/dashboard
```

当前仍主要使用：

```http
GET /api/dashboard/summary
```

当前 summary 接口存在的目标口径差异：

1. 仍使用当前 15 状态分组，不支持目标 9 状态口径。
2. `inProgress` 只统计 `IN_PROGRESS`，目标口径下 `APPROVED` 迁移为 `IN_PROGRESS`。
3. 部门用户部分统计仍使用标量 `departmentId`，未统一使用 `responsibleDepartmentIds` / `departmentIds` / `departmentId` helper。
4. 公司领导角色仍可能按公司级角色获取过宽统计，不符合 `VICE_PRESIDENT` 不默认全局的目标规则。
5. 当前返回 `thisMonthDue`，目标脚本按临期口径计算 `expiring`。

自动化脚本中相关断言标记为 `EXPECTED_FAILURE`。

## 二、事项列表可见范围不符合目标组织权限

当前：

```http
GET /api/works
```

主要差异：

1. `VICE_PRESIDENT` / `PRESIDENT` 当前仍可能按公司级角色查看过宽数据。
2. 部门用户当前主要按标量 `departmentId` 过滤，未完整覆盖多主责部门 `departmentIds` 和配合部门 `cooperateDepartmentIds`。
3. 目标口径要求 `responsiblePersons` / `cooperatePersons` 不参与权限判断；脚本会验证姓名相同不能授权。
4. 目标口径要求 `responsibleDepartmentIds` / `departmentIds` / `departmentId` 和 `cooperateDepartmentIds` 参与组织权限判断。

自动化脚本中副总互相可见、多主责部门可见、配合部门可见等断言可能为 `EXPECTED_FAILURE`。

## 三、完成率统计仍需补齐多主责部门口径

当前：

```http
GET /api/dashboard/completion-rate
```

当前验证结果：

1. 部门 A fixture 当前已通过目标口径断言，说明该单一主责部门样例与当前实现刚好一致。
2. 这不代表完成率接口整体已经符合目标口径，部门 B / 多主责部门场景仍为 `EXPECTED_FAILURE`。

主要差异：

1. 当前完成率统计使用标量 `departmentId`。
2. 目标口径要求按主责部门集合统计，多主责部门事项可分别计入多个主责部门。
3. 配合部门只参与可见，不计入完成率分母和分子。
4. 当前接口尚未统一通过 `getResponsibleDepartmentIds(work)` 或统一权限模块读取主责部门。

自动化脚本中部门 A 完成率用例按正常目标断言处理；部门 B / 多主责部门完成率用例继续标记为 `EXPECTED_FAILURE`。

## 四、目标 9 状态尚未落地到 schema

当前 `WorkItemStatus` 仍为 15 状态。自动化测试数据用当前状态承载目标语义：

| 当前状态 | 目标语义 |
|----------|----------|
| `PENDING_DEPT` / `PENDING_COMPANY` | `PROPOSING` |
| `APPROVED` / `IN_PROGRESS` | `IN_PROGRESS` |
| `PENDING_COMPLETE` / `PENDING_EVIDENCE_DEPT` / `PENDING_EVIDENCE_COMPANY` | `COMPLETING` |
| `PENDING_MAIN_LEADER_CANCEL` / `CANCELLING` | `CANCELLING` |
| `REJECTED` / `DRAFT` | `DRAFT` |

后续迁移到目标 9 状态后，需要同步更新 seed 数据和验证脚本中的兼容映射。

## 五、目标责任字段尚未完整落地

当前 schema 尚未提供重点/主要事项的独立 `responsiblePerson` 字段。目标口径为：

1. `responsibleLeader`：事项业务责任领导姓名文本。
2. `responsiblePerson`：事项实际责任人姓名文本。
3. 二者不是 `User` 外键，不是 `Member` 外键，不参与权限判断。

当前测试脚本不会把业务责任人姓名作为权限依据；后续 schema 落地后需要补充字段级断言。
