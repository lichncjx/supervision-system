# 业务规则

本文档是公司督办管理系统的业务规则权威入口。涉及权限、审批流、状态机、统计口径、数据范围的修改，必须先阅读本文档，并在 PR 中说明影响范围。

## 角色定义

角色枚举以 `prisma/schema.prisma` 的 `Role` 为准。

| 枚举 | 中文名称 | 核心职责 |
|------|----------|----------|
| `ADMIN` | 系统管理员 | 系统配置、用户和部门管理 |
| `SUPERVISOR` | 督办管理员 | 督办跟踪、数据查看、统计导出 |
| `DEPARTMENT_MANAGER` | 部门主管 | 发起事项、维护本部门事项、提交材料 |
| `DEPARTMENT_LEADER` | 部门领导 | 查看本部门事项、进行部门级审批 |
| `VICE_PRESIDENT` | 公司主管领导 | 查看全部事项、进行公司级审批 |
| `PRESIDENT` | 公司主要领导 | 查看全部事项、审批重点工作取消等高层事项 |

默认测试账号以 `prisma/seed.ts` 为准，默认密码为 `123456`。

> **重要区分**：`SUPERVISOR` 角色（督办管理员）与事项字段 `supervisor`（主管人员/业务主管人员）**不是同一个概念**。
> - `SUPERVISOR` 是系统级角色，拥有全局查看、导入导出权限。
> - `supervisor` 是 WorkItem 上的姓名文本字段（legacy），表示该事项的业务主管人员，现已迁移为 `deptManagerName`。
> - 权限判断只认角色枚举 `SUPERVISOR`，不认 `supervisor` 字段。

## 业务人员字段命名规则

本系统采用 **xxxId / xxxName 两层命名约定**，所有业务人员字段遵循此规则：

1. **xxxId**（如 `creatorId`、`deptLeaderId`、`deptManagerId`、`proposedLeaderId`、`approvalLeaderId`）：真实关联字段，FK 指向 `users.id`，用于权限判断、流程路由、审批指派、上传权限、退回处理人判定。
2. **xxxName**（如 `deptLeaderName`、`deptManagerName`）：历史快照字段，仅用于页面展示、Excel 导出、历史留痕。创建/更新时从 User 记录同步写入，后续即使 User 姓名变更也不回溯更新。
3. 文本旧字段（`responsibleLeader`、`supervisor`、`responsiblePersons`、`cooperatePersons` 等）不立即删除，作为兼容和历史快照保留。
4. **权限判断原则上只认 ID，不认姓名文本**。服务端权限校验一律使用 ID 字段匹配。

## 重点工作 / 主要工作人员体系

重点工作（`PRIORITY`）和主要工作（`MAIN`）是**单部门、单线条办理体系**：

| 字段 | 说明 |
|------|------|
| `departmentId` | 单一主办部门 ID（FK → departments） |
| `deptLeaderId` | 部门领导用户 ID（FK → users，关联 `DEPARTMENT_LEADER` 角色用户） |
| `deptLeaderName` | 部门领导姓名快照（优先用于展示/导出，fallback 到 `responsibleLeader`） |
| `deptManagerId` | 主管人员/业务主管人员用户 ID（FK → users，关联 `DEPARTMENT_MANAGER` 角色用户） |
| `deptManagerName` | 主管人员姓名快照（优先用于展示/导出，fallback 到 `supervisor`） |
| `responsibleLeader` | **旧字段（legacy）**，兼容 `deptLeaderName`。不参与权限判断 |
| `supervisor` | **旧字段（legacy）**，兼容 `deptManagerName`。注意：与系统角色 `SUPERVISOR`（督办管理员）无关 |

**创建/更新规则**：
- 新建时从部门人员下拉选择，服务端校验所选用户所属部门、角色（`DEPARTMENT_LEADER` / `DEPARTMENT_MANAGER`）和启用状态（`isActive`）。
- 更新时同时写入新旧字段（双写），确保旧字段兼容。
- 部门切换（`departmentId` 变化）时清空 `deptLeaderId` / `deptManagerId`，需要重新选择。
- 退回编辑表单支持部门领导和主管人员的下拉选择。

**展示/搜索规则**：
- 详情页、列表页优先展示 `deptLeaderName` / `deptManagerName`，旧字段兜底。
- Excel 导出同理，优先新字段，fallback 到旧字段。
- 搜索功能包含 `deptLeaderName` / `deptManagerName`。

## 待办事项人员体系

待办事项（`TODO`）是**多部门、多责任人体系**：

| 字段 | 说明 |
|------|------|
| `departmentIds` | 主责部门 ID 数组（当前使用此名称，未来可迁移为 `responsibleDepartmentIds`） |
| `responsiblePersons` | 主责责任人姓名数组（legacy，当前仅用于展示留底，暂不做 userId 化） |
| `cooperateDepartmentIds` | 配合部门 ID 数组 |
| `cooperatePersons` | 配合责任人姓名数组（legacy，当前仅用于展示留底，暂不做 userId 化） |

**当前规则**：
1. 待办事项责任人暂不做 userId 化，仍使用姓名字符串数组。
2. 配合部门（`cooperateDepartmentIds`）中的用户可查看该待办事项（服务端已做 `cooperateDepartmentIds` 权限匹配）。
3. 配合部门用户**不因配合部门身份**获得上传或删除附件权限。

## 公司领导字段

| 字段 | 说明 |
|------|------|
| `proposedLeaderId` | 提出领导 ID（公司领导：`VICE_PRESIDENT` 或 `PRESIDENT`），FK 指向 users |
| `approvalLeaderId` | 审批领导 ID，负责后续调整、取消、完成审批 |
| `proposedLeader` | 前端展示字段，由 Prisma relation 从 `proposedLeaderId` 解析，不作为数据库主字段 |
| `approvalLeader` | 前端展示字段，由 Prisma relation 从 `approvalLeaderId` 解析，不作为数据库主字段 |

**规则**：
1. `approvalLeaderId` 默认等于 `proposedLeaderId`（创建/导入/表单三处统一）。
2. 特殊情况下二者可以不同（如提出领导变更或指定其他审批领导）。
3. 后续如有迁移，可考虑 `proposedLeaderName` / `approvalLeaderName` 命名方式。

## 数据范围

1. 公司级角色可查看全部数据：`ADMIN`、`SUPERVISOR`、`VICE_PRESIDENT`、`PRESIDENT`。
2. 部门角色默认只能查看和操作本部门数据：`DEPARTMENT_MANAGER`、`DEPARTMENT_LEADER`。
3. 服务端权限校验优先于前端按钮展示。
4. 页面隐藏按钮不是权限控制，关键操作必须在 API 或服务端 workflow 层再次校验。

## 事项类型

| 类型 | 枚举值 | 说明 |
|------|--------|------|
| 重点工作 | `PRIORITY` / `priority` | 公司级重要事项 |
| 主要工作 | `MAIN` / `main` | 部门主要工作 |
| 待办事项 | `TODO` / `todo` | 日常任务或领导交办事项 |

## 状态定义

状态枚举以 `prisma/schema.prisma` 的 `WorkItemStatus` 为准。

| 状态 | 说明 |
|------|------|
| `draft` | 草稿 |
| `pending_dept` | 待部门审批 |
| `pending_company` | 待公司审批 |
| `approved` | 已立项 |
| `in_progress` | 进行中 |
| `pending_decompose` | 待分解 |
| `pending_complete` | 待完成确认 |
| `pending_evidence_dept` | 待部门见证审批 |
| `pending_evidence_company` | 待公司见证审批 |
| `pending_main_leader_cancel` | 待公司主要领导审批取消 |
| `adjusting` | 调整审批中 |
| `cancelling` | 取消审批中 |
| `completed` | 已完成 |
| `cancelled` | 已取消 |
| `rejected` | 已退回 |

## 状态流转原则

1. 页面不得直接修改事项状态。
2. 状态流转必须通过 `src/lib/workflow.ts` 或统一 workflow 服务处理。
3. 每次状态流转必须写入审批或流程记录。
4. 审批记录不得删除。
5. 被退回的事项应保留退回原因和退回前状态。
6. 事项在审批环节被退回后，退回给首次提交审批人（`firstSubmitterId`）处理，而非创建人或部门角色。
7. `firstSubmitterId` = 首次将事项送入正式审批链路（`pending_dept` 或 `pending_company`）的操作人，不含 `pending_decompose`。
8. `firstSubmitterId` 在首次进入正式审批链路时写入，后续退回再提交不覆盖。

## 退回处理规则

### 退回处理人判定

1. 退回后仅 `firstSubmitterId` 对应用户可以查看和处理退回事项。
2. 不再按 `creatorId`、关联部门角色（`DEPARTMENT_MANAGER` / `DEPARTMENT_LEADER`）泛化判断。
3. 历史数据中 `firstSubmitterId` 为空的，fallback 使用 `creatorId`（仅用于兼容引入该字段前的旧数据）。

### 公司领导创建待办的特殊规则

1. 公司领导创建待办事项后提交到 `pending_decompose` 时，**不设置** `firstSubmitterId`。
2. 后续部门主管/领导完成分解并提交进入 `pending_dept` 或 `pending_company` 时，才设置 `firstSubmitterId`。
3. 退回时，事项退给该部门主管/领导，不出现在公司领导的待我处理列表中。

### 重新提交流程

1. 退回事项重新提交时，**不允许**直接通过 PUT 修改 status。
2. 正确流程：先通过 PUT 保存内容（保持 status 为 `rejected`），再调用 workflow submit action 重新进入审批链路。
3. workflow 层负责重新计算下一审批状态、`currentApproverRole`、`currentApproverId`。
4. 重新提交会生成新的 workflow record。
5. 重新提交**不覆盖** `firstSubmitterId`。

## 审批规则

### 重点工作和主要工作

1. 部门主管提交后进入 `pending_dept`，由本部门部门领导审批。
2. 部门领导审批通过后进入 `pending_company`，由公司主管领导审批。
3. 公司主管领导审批通过后进入 `approved`。
4. 提交见证材料后进入部门或公司见证审批。
5. 见证审批通过后进入 `completed`。
6. 重点工作取消需要公司主要领导最终审批。

### 待办事项

1. 公司领导发起的待办事项可进入 `pending_decompose`，等待责任部门分解。
2. 部门主管发起的待办事项先进入 `pending_dept`。
3. 部门领导发起或审批通过后，进入 `pending_company`，由提出领导审批。
4. 提出领导审批通过后进入 `in_progress`。
5. 提交完成材料后进入 `pending_complete`。
6. 当前完成审批人审批通过后进入 `completed`。

### 调整和取消

1. `approved` 或 `in_progress` 状态可申请调整或取消。
2. 调整进入 `adjusting`。
3. 取消进入 `cancelling`。
4. 主要工作取消通常由部门领导和公司主管领导逐级审批。
5. 重点工作取消在公司主管领导审批后，还需要公司主要领导审批。
6. 待办事项调整或取消应优先按提出领导和当前审批人规则处理。

## 待我处理口径

待我处理 = 待审批 + 待办理。

### 待审批

只统计当前用户作为当前审批人的事项。判断依据包括：

1. `currentApproverId` 等于当前用户 ID。
2. `currentApproverRole` 等于当前用户角色，并且部门级审批必须匹配本部门。
3. 公司主要领导只处理明确需要公司主要领导审批的事项。

### 待办理

待办理指不需要当前用户审批，但需要当前用户处理的事项，例如：

1. 待分解的待办事项。
2. 被退回后需要首次提交审批人（`firstSubmitterId`）修改并重新提交的事项。
3. 待提交见证材料或完成材料的事项。
4. 待完成确认但当前用户是处理责任人的事项。

> **注意**：退回事项的待办理判断使用 `firstSubmitterId`，不是 `creatorId`。
> 历史数据中 `firstSubmitterId` 为空的，fallback 使用 `creatorId`。

## 统计口径

1. 首页统计卡片必须与点击后的列表结果一致。
2. `待审批`、`待办理`、`待我处理` 的统计条件必须与对应列表查询条件一致。
3. 涉及统计修改时，必须同时检查 Dashboard API、列表查询 API 和前端筛选逻辑。
4. 超期、进行中、已完成等状态统计必须使用统一状态集合，不得页面各自定义。

## API 和服务端约束

1. 创建、更新、删除、审批、提交材料、导入导出等关键接口必须校验登录态。
2. 涉及部门数据时必须校验数据范围。
3. 涉及审批时必须校验当前审批人。
4. 涉及状态变化时必须校验当前状态是否允许该动作。
5. 不允许通过普通更新接口绕过 workflow 修改状态。

## 附件与证明材料机制

### 存储模型

1. 通用附件和证明材料**统一存储在 `attachments` 表**。
2. `attachments.category` 字段用于区分附件类型：
   - `"general"` — 普通附件（默认值，兼容历史数据）
   - `"evidence"` — 完成证明材料 / 见证材料
3. 历史附件默认 `category = "general"`，无需数据回填。

### 上传流程

1. 所有附件通过 `POST /api/upload` 上传，传入 `category` 参数区分类型。
2. 证明材料文件在调用 workflow 之前上传（与通用附件上传流程一致）。
3. workflow `action=evidence` 只处理 `proof` 文本和状态流转，**不处理文件上传**。
4. `submitEvidence` 不改，继续只写入 `work_items.proof` 文本字段。

### 前端展示

1. 普通附件面板（`WorkAttachmentPanel`）只显示 `category != "evidence"` 的附件。
2. 证明材料区域只显示 `category == "evidence"` 的附件。
3. `proof` 字段仍只保存文字说明，与证明文件分离展示。
4. `proofFiles` 前端 base64 内存方案已废弃，相关死代码已移除。

### 与旧系统的区别

| 维度 | 旧方案 | 新方案 |
|------|--------|--------|
| 证明文件存储 | 前端 React state（base64），页面刷新丢失 | `attachments` 表持久化 |
| 证明文件传输 | `submitComplete` 丢弃 `_proofFiles` 参数 | 通过 `/api/upload` 上传后 workflow 引用 |
| 附件分类 | 无分类，全部混在一起 | `category` 字段区分 general/evidence |
| 详情页证明文件展示 | 死代码，永远不执行 | 从 `work.attachments` 按 category 筛选展示 |

## 附件权限规则

附件权限统一由 `src/lib/attachment-permissions.ts` 模块集中管理，消除 upload / delete / download 三处重复逻辑。

### 查看 / 下载权限 (`canViewAttachment`)

| 用户 | 权限 |
|------|------|
| `ADMIN` / `SUPERVISOR` | 可查看下载所有附件 |
| 事项创建人 (`creatorId`) | 可查看下载 |
| 同部门用户 (`departmentId` 匹配) | 可查看下载 |
| 重点/主要工作的 `deptManagerId` 对应用户 | 可查看下载 |
| `VICE_PRESIDENT` / `PRESIDENT` | 公司级可见，可查看下载 |

### 上传权限 (`canUploadAttachment`)

| 用户 | 权限 |
|------|------|
| `ADMIN` / `SUPERVISOR` | 可上传所有附件 |
| 同部门 `DEPARTMENT_MANAGER` / `DEPARTMENT_LEADER` | 可上传（`rejected` 状态除外） |
| 重点/主要工作的 `deptManagerId` 对应用户 | 可上传普通附件和证明材料（**含 `rejected` 状态**，用于退回后补充材料） |
| `VICE_PRESIDENT` / `PRESIDENT` | 不可上传 |
| 配合部门用户 | **不因配合部门身份获得上传权限** |
| 终止态（`completed` / `cancelled`） | 普通用户和 `deptManagerId` 均不可上传 |

### 删除权限 (`canDeleteAttachment`)

删除权限已收紧（Phase 3C），规则如下：

| 用户 | 权限 |
|------|------|
| `ADMIN` | 可删除所有附件 |
| `SUPERVISOR` | 可删除所有附件 |
| 上传者本人 (`attachment.userId`) | 可删除自己上传的附件 |
| `deptManagerId`（非上传者） | **不可删除他人上传的附件** |
| 同部门 `DEPARTMENT_MANAGER` / `DEPARTMENT_LEADER`（非上传者） | **不再默认删除他人附件** |
| 配合部门用户 | **不能删除他人附件** |

### 附件权限模块说明

- 模块位置：`src/lib/attachment-permissions.ts`
- 三个导出函数：`canViewAttachment`、`canUploadAttachment`、`canDeleteAttachment`
- 入参接口：`AttPermUser`（`id, role, departmentId`）、`AttPermWorkItem`（`departmentId, status, creatorId, type, deptManagerId?`）、`AttPermAttachment`（`userId`）
- `/api/upload`、`/api/attachments/[id]`、`/api/attachments/[id]/download` 三处统一调用此模块
