# 后端模块化架构约定

## 一、架构定位

本项目为基于 **Next.js 全栈单体应用** 的公司督办管理系统，当前不采用传统意义上的前后端物理分离结构，也不机械套用教科书式 Clean Architecture。

本项目采用：

> **Modular Monolith + Feature-based Layered Architecture 简化方案**

即：

```txt
第一层：按业务功能模块划分；
第二层：每个业务模块内部按职责分层；
第三层：保留 Next.js 现有 app/api 与 app/(app) 入口结构。
```

本次架构治理的目标不是重写系统，而是在不改变现有接口路径、业务行为和数据库结构的前提下，逐步解决当前代码中存在的：

```txt
1. API Route 过厚；
2. src/lib 职责过杂；
3. 权限、状态、审批流规则分散；
4. Prisma 查询散落；
5. 类型定义、DTO、接口返回结构难找；
6. 前端 API 调用、UI 展示、后端业务逻辑混杂。
```

## 三、为什么不采用 src/backend + src/frontend

本项目当前是 Next.js 全栈单体，不是传统的：

```txt
后端：.NET Web API / Spring Boot
前端：Vue / React
```

因此当前阶段不采用：

```txt
src/backend/
src/frontend/
```

这种顶层前后端物理分离方式。

原因：

```txt
1. Next.js 的 app/api 和 app/(app) 本身就是框架约定；
2. 当前主要问题不是前后端没分离，而是业务模块边界不清；
3. 许多业务类型、状态枚举、DTO、展示名称存在前后端共用需求；
4. 过早前后端物理拆分会增加路径复杂度；
5. 当前更适合先在单体内部建立清晰模块边界。
```

后续如果确有需要拆成独立前后端工程，可在当前模块化结构稳定后，再按：

```txt
features/*/application + infrastructure + presentation → 后端项目
features/*/client + ui                             → 前端项目
features/*/domain                                  → 共享类型包或分别复制
```

进行演进。

## 四、feature 模块划分原则

业务代码优先按功能模块划分，而不是按技术类型划分。

推荐模块包括：

```txt
works           事项模块，包含重点工作、主要工作、待办事项的基础业务；
workflow        审批流模块，包含提交、审批、驳回、完成、调整、取消、分解；
attachments     附件模块，包含证明材料上传、下载、删除、权限；
excel           Excel 模块，包含导入、导出、模板、完成率导出；
dashboard       看板与统计模块；
users           用户、角色、密码、账号管理；
departments     部门组织模块；
operation-logs  操作日志模块。
```

不要新建笼统的大模块，例如：

```txt
services/
repositories/
types/
utils/
```

也不要继续把业务代码堆到：

```txt
src/lib/
```

---

## 五、每个 feature 内部的六类目录

每个业务模块统一采用并排六个目录：

```txt
domain/
application/
infrastructure/
presentation/
client/
ui/
```

不再额外增加过深的：

```txt
server/domain/
server/application/
server/infrastructure/
```

目录层级保持简洁，方便查找和维护。

# 1. domain

## 职责

`domain` 放当前模块的核心业务语言和业务规则。

包括：

```txt
业务类型
枚举
状态定义
状态判断
业务规则
权限规则
状态流转规则
模块内常量
```

示例：

```txt
features/works/domain/
  work.types.ts
  work-status.ts
  work-status.rules.ts
  work.rules.ts
  work.permissions.ts
  work.visibility-policy.ts
```

## 约定

`domain` 应尽量保持纯净：

```txt
尽量不直接访问 Prisma；
尽量不读取 cookies；
尽量不依赖 NextResponse；
尽量不直接操作 fs；
尽量不写 React 组件；
尽量不写 UI 样式。
```

允许前端、后端、UI 共用其中的纯类型、枚举、状态名称、状态判断函数。

例如：

```txt
WorkStatus
WorkType
WORK_STATUS_LABELS
isCompletedStatus
isApprovalStatus
getWorkTypeLabel
```

## server-only 规则

由于本项目采用简化目录结构，不再单独拆 `server/domain`。

如果某些 domain 文件只适合服务端使用，应通过文件命名明确标识：

```txt
*.server-permissions.ts
*.server-policy.ts
*.prisma-policy.ts
*.visibility-policy.ts
```

例如：

```txt
work.server-permissions.ts
work.visibility-policy.ts
```

这类文件不得被 `client` 和 `ui` 引用。

# 2. application

## 职责

`application` 放具体业务用例。

它负责组织一次完整业务操作。

包括：

```txt
查询事项
新建事项
更新事项
删除事项
提交审批
审批通过
审批驳回
提交完成申请
提交调整申请
提交取消申请
上传附件
导出完成率
```

示例：

```txt
features/works/application/
  query-works.usecase.ts
  get-work-detail.usecase.ts
  create-work.usecase.ts
  update-work.usecase.ts
  delete-work.usecase.ts
```

```txt
features/workflow/application/
  submit-for-approval.usecase.ts
  approve-work.usecase.ts
  reject-work.usecase.ts
  submit-completion.usecase.ts
  submit-adjustment.usecase.ts
  submit-cancel.usecase.ts
```

## application 做什么

```txt
1. 接收 route 传入的参数；
2. 调用 domain 规则判断业务是否合法；
3. 调用 infrastructure 查询或保存数据；
4. 调用 operation log 记录操作；
5. 必要时组织数据库事务；
6. 返回 presentation 需要的结果。
```

## application 不做什么

```txt
不直接写 HTTP Response；
不直接读取 cookies；
不直接写 React 组件；
不直接拼复杂 UI 文案；
不直接把 Prisma 原始对象暴露给前端。
```

包括：



```txt

5. infrastructure 不应依赖 ui。
```

## DTO 约定

本项目不要求所有小接口都建立复杂 DTO，但以下接口建议必须有 DTO：

```txt
事项列表
事项详情
新建事项
更新事项
Workflow 操作
审批记录
Excel 导入结果
完成率统计
Dashboard Summary
附件上传/下载/删除
用户列表/新增/修改
```

## Prisma select 与 DTO 的关系

```txt
Prisma select/include 用于控制数据库查询字段；
DTO 用于控制接口对外返回结构；
两者不是二选一。
```

推荐模式：

```txt
repository 查询数据；
presenter 转换为 response DTO；
route 返回 DTO。
```

包括：

```txt

前端 ViewModel
前端筛选
前端排序
前端按钮显示判断
```

示例：

```txt

  work-actions-api.ts
  work-view-model.ts
  work-client-filters.ts
  work-client-permissions.ts
```

```txt
features/users/client/
  user-api.ts
  auth-api.ts
```

## 约定


前端的权限判断只用于界面展示，例如是否显示按钮；真正权限必须由后端 application/domain 再判断一次。

## 七、app/api 约定

`app/api` 继续作为 Next.js API Route 入口。

但 API Route 应逐步变薄。

目标形态：

```txt
1. 获取当前用户；
2. 解析参数；
3. 调用 application use case；
4. 返回响应。
```

不应继续在 route.ts 中堆积：

```txt
复杂 Prisma 查询
复杂权限判断
状态流转规则
Excel 解析
文件存储细节
大量 DTO 拼装
重复 try/catch
重复登录校验
```

示例：

```ts
export const GET = withApiHandler(async (request) => {
  const currentUser = await requireCurrentUser(request);
  const query = parseWorkQuery(request);

  const result = await queryWorksUseCase({
    currentUser,
    query,
  });

  return ok(result);
});
```

## 八、app/(app) 页面约定

`app/(app)` 继续作为页面路由入口。

页面组件应尽量变薄，优先引用 feature 中的：

```txt
client
ui
domain
```

页面不应直接包含大量：

```txt
复杂 fetch 封装
业务筛选规则
状态名称映射
按钮权限判断
表格字段转换
```

这些逻辑应进入：

```txt
features/{module}/client
features/{module}/ui
features/{module}/domain
```

## 九、src/lib 处理原则

`src/lib` 不立即删除，但要逐步瘦身。

后续不再向 `src/lib` 新增核心业务逻辑。

当前 `src/lib` 中的内容应逐步迁移：

```txt
prisma.ts                  → shared/db/prisma.ts
server-auth.ts             → shared/auth/*
server-permissions.ts      → features/works/domain/*、features/attachments/domain/*
workflow.ts                → features/workflow/*
work-status.ts             → features/works/domain/*
dashboard-data.ts          → features/dashboard/*
excel-utils.ts             → features/excel/*
work-store.ts              → features/works/client/*
attachment-permissions.ts  → features/attachments/domain/*
status-colors.ts           → features/works/ui/*
utils.ts                   → shared/utils、shared/ui、features/works/presentation
```

迁移时允许保留兼容导出。

例如：

```ts
// src/lib/prisma.ts
export { prisma as default } from '@/shared/db/prisma';
```

或者：

```ts
// src/lib/work-status.ts
export * from '@/features/works/domain/work-status';
export * from '@/features/works/domain/work-status.rules';
```

这样可以避免一次性修改全项目 import。

## 十、依赖方向约定

推荐依赖方向：

```txt
app/api → application
application → domain + infrastructure + presentation
infrastructure → shared/db + shared/utils
presentation → domain
client → domain + presentation 类型
ui → domain + client
```

尽量避免：

```txt
domain → application
domain → infrastructure
domain → ui
infrastructure → ui
application → ui
client → infrastructure
ui → infrastructure
```

底线原则：

```txt
后端不要依赖 UI；
前端不要依赖 Prisma；
业务规则不要依赖 React；
UI 展示不要承担后端权威权限判断。
```

## 十三、风险控制原则

## 1. 不一次性全量重构

每一阶段只解决一类问题。

不要同时做：

```txt
结构迁移
业务 bug 修复
权限口径调整
数据库 schema 修改
前端页面重构
```

---

## 2. 保持接口行为不变

结构治理阶段必须保证：

```txt
API 路径不变；
入参格式不变；
返回字段不变；
状态流转不变；
权限口径不变；
数据库 schema 不变。
```

---

## 3. 保留兼容导出

迁移初期允许旧路径继续可用，降低改造风险。

---

## 4. 每阶段单独提交

建议分支：

```txt
refactor/modular-structure-phase-0
refactor/modular-structure-phase-1-shared
refactor/modular-structure-phase-2-work-domain
refactor/modular-structure-phase-3-works-query
```

---

## 5. 每阶段完成后必须验证

至少执行：

```txt
pnpm lint
pnpm build
```

如有测试脚本，应执行对应 API 回归测试。

---

## 十四、禁止事项

后续新增代码应避免：

```txt
1. 在 route.ts 中继续写大段业务逻辑；
2. 在 route.ts 中直接写复杂 Prisma 查询；
3. 在 src/lib 中继续堆业务规则；
4. 新增巨型 types.ts；
5. 新增巨型 permissions.ts；
6. 前端直接引用 Prisma 类型；
7. client/ui 引用 infrastructure；
8. UI 组件承担后端权威权限判断；
9. 为了追求架构纯粹而引入过度抽象；
10. 在当前阶段引入 repository interface 或复杂依赖倒置。
```

---

## 十五、最终目标

本次模块化改造的最终目标是：

```txt
1. 事项规则集中到 works；
2. 审批流规则集中到 workflow；
3. 附件规则集中到 attachments；
4. Excel 导入导出集中到 excel；
5. Dashboard 和完成率口径集中到 dashboard；
6. 用户角色规则集中到 users；
7. API Route 变薄；
8. src/lib 逐步瘦身；
9. Prisma 查询集中；
10. DTO、validator、presenter 有固定位置；
11. 前端 API、ViewModel、UI 组件按模块归位。
```

一句话总结：

> 本项目后续采用 **模块优先、分层适度、路径清晰、渐进迁移** 的架构治理原则。
> 不追求形式上的 Clean Architecture，而追求适合当前 Next.js 全栈单体项目的可维护结构。
